## 1. Objective

Two coupled deliverables, both due Oct 15.

### 1.1 Integration spec (the contract)

A normative, versioned **integration contract** for the `litany` wire stack as
consumed by the rest of the Vaked fleet, so WP4 daemons and the `eventd` audit
spine can depend on a frozen surface. It must pin, with RFC citations:

1. **The `litany` ↔ `eventd` chain-admission contract** — `litany` feeds `eventd`
   **only** strictly-canonical `frame-bytes` (header + body) per RFC 0002 §6.8,
   plus connection-lifecycle events; a header-clean-but-non-canonical body is a
   frame-level `error` that is **never** chained (RFC 0003 §11 Conformance item 9;
   §9.2; §10). The contract names the exact bytes handed across and the rejection
   point, cross-referencing [`docs/superpowers/specs/2026-06-12-eventd-design.md`](2026-06-12-eventd-design.md)
   and the `eventd` reference impl ([`eventd/`](../../../eventd/), tests in
   [`tests/spec/test_eventd.py`](../../../tests/spec/test_eventd.py)).
2. **The handshake / capability surface** WP4 peers must satisfy: magic `HCP\0`
   preamble, `wire_version`, `max_frame` ≥ 64 KiB, schema-digest intersection,
   orderly `REFUSE` + graceful close on incompatibility (RFC 0003 §5, §9.1a; §11
   Conformance items 3–4).
3. **The transport-binding profiles** a daemon may bind: `stdio` (RFC 0003 §13.1),
   Unix domain socket (§13.2), `vsock` for the MirageOS unikernel surface (§13.3) —
   marked normatively non-normative per §13's own framing.
4. **The decision-rule boundary** (RFC 0003 §9.3): which failures a consumer sees
   as transport faults (connection torn down) vs frame-level `error`s (connection
   survives), so daemon error-handling code is written against a stable taxonomy
   (RFC 0003 §9.1, §9.1a, §9.2, §9.2.1).
5. **The codegen contract** for non-Rust consumers: the Zig and Erlang/OTP shapes
   the wire types lower to (RFC 0002 §7 mapping table line 907; §7.1; Appendix
   A.1/A.2), so the WP4 Zig daemons and any OTP peer share one schema source.

### 1.2 Paper eval (#113)

The **evaluation section** of the #113 protocol paper, structured exactly like the
#103 language paper's eval (benchmarks + interop/case studies + threat model;
[`docs/papers/vaked-language-v0.1.md`](../../../docs/papers/vaked-language-v0.1.md)
§5, §1.4 "Contributions at a Glance"), and governed by the **same honesty
contract** as [`examples/evaluation/METHODOLOGY.md`](../../../examples/evaluation/METHODOLOGY.md)
(every number labelled **Measured / Projected / Retracted**, with environment
provenance; "if a number appears in a paper table but not in the ledger, treat it
as unverified").

The headline claims the eval must **measure** (not assert from estimates):

1. **Wire-size / message-overhead reduction** vs JSON-RPC 2.0 / CBOR / protobuf,
   computed by byte-counting the *same* semantic messages — the strongest claim
   fully verifiable on M1 (§4.1).
2. **Encode/decode latency & throughput**, including the kickoff's **≤10µs/frame**
   gate (kickoff line 54) — arch-specific, `dev-cx53`-only as an absolute number;
   relative-regression-only on M1 (§4.2, §5).
3. **Determinism / replay reproducibility** — session capture → `litanyreplay` →
   byte-identical, the protocol analogue of the language paper's `baseline.json`
   determinism oracle (RFC 0002 §9; RFC 0003 §10) (§4.3).
4. **Cross-implementation interop** — Rust `litany` ↔ the Python reference impl on
   the shared golden + rejection corpus (§4.4).
5. **The protocol-layer threat model** — DoS resistance, tamper-evidence via
   canonical encoding, control-plane deadlock-freedom — sourced from S7's
   empirical results and RFC 0003 §14.1–§14.5 + RFC 0002 §11 (§4.5).

> **Honesty guard, explicit.** [`ROADMAP_2026-2027.md`](../../../ROADMAP_2026-2027.md)
> line 160 states "2–3 KB message overhead reduction" as if it were a result. Per
> this project's culture (#103 retracted a ">120s" figure; see METHODOLOGY.md),
> the eval **measures** overhead from golden vectors and reports the measured
> value with provenance — it does **not** cite the a-priori roadmap estimate as a
> finding. If the measurement disagrees with "2–3 KB", the measurement wins and
> the estimate is dropped.

## 4. Algorithm / design

### 4.1 Wire-size / message-overhead comparison (the M1 centerpiece)

The strongest claim that is **fully verifiable on M1** because it is deterministic
byte-counting, not timing.

1. **Fix the semantic message set.** Take the worked-example messages the RFCs
   already define canonically: the `hcp.core.ToolCallRequest` family from RFC 0002
   §10 (line 1009) / §10.1, the primitive/record/map/union worked examples (§6.1.1,
   §6.3.1, §6.6.1, §6.7.1), and the framed examples from RFC 0003 §4.6 / §5.8.
   These are the only messages whose canonical `hcpbin` bytes are RFC-given; do not
   add messages whose canonical form must be invented.
2. **Encode the *same* semantics** in JSON-RPC 2.0, CBOR (RFC 8949 canonical
   form), and protobuf proto3 (`comparand_encoders.py`). For protobuf, the proto
   schema is derived from the `.hcplang` record (a deterministic, documented
   mapping — record fields → numbered proto fields by tag).
3. **Tabulate** per-message byte counts into `overhead.json` and render the table
   in `OVERHEAD.md` / the paper section. Report absolute bytes and the
   reduction-vs-each-comparand, **with the message set named** so the claim is
   reproducible. This replaces the roadmap's a-priori "2–3 KB" estimate with a
   measured number (§1.2 honesty guard).
4. **Caveat in the ledger:** the comparison is over the worked-example corpus, not
   a production trace; label it **Measured (corpus)**, and state the corpus, the
   comparand library versions, and CBOR/protobuf canonicalization settings — the
   #103 paper's "small examples dominated by startup floor" honesty move
   (METHODOLOGY.md §1), applied here as "results are over the RFC worked-example
   corpus, not field traffic."

### 4.2 Latency / throughput (arch-sensitive — see §5)

The kickoff success criterion is **≤10µs per frame encode/decode on `dev-cx53`
(8-core, no-alloc hot path)** (kickoff line 54). As established in S7 §4.6, an
**absolute µs threshold is not portable to M1 (aarch64-darwin)**. S8 therefore:

- on **M1**, runs the S6 criterion benches for **relative regression** vs the S6
  baseline only, and labels every M1 latency number **Measured (M1, relative
  only — not the gate)** in the ledger;
- pins the **absolute ≤10µs gate** to `dev-cx53` and labels it **Projected** in
  any draft written before a `dev-cx53` run, flipping to **Measured (dev-cx53)**
  only after the canonical host run (which is off-limits this window — §5.2);
- reuses the S6 latency benches; it does **not** author new latency
  micro-benchmarks (that is S6's deliverable). The only new bench S8 adds is
  `overhead.rs` (byte counts, §4.1), which is size not time.

### 4.3 Replay determinism (the protocol `baseline.json`)

`litanydump` captures a live or synthetic frame stream to a portable form;
`litanyreplay` replays it and asserts the re-emitted `frame-bytes` are
**byte-identical** to the capture (RFC 0002 §9 source-mapping/evidence; RFC 0003
§10 determinism/evidence/replay). The committed `replay_baseline.json` records N
iterations per scenario, exactly mirroring the language paper's determinism oracle
(`examples/evaluation/baseline.json`, 20 iterations/example). The eval reports the
iteration count actually committed — never a larger figure than the data shows
(the #103 paper's "earlier 100-iteration figures are not what the committed data
shows" lesson, METHODOLOGY.md / paper §5.4 item 4).

### 4.4 Cross-implementation interop (the M1 integration spine)

`interop_python.rs` + `tests/spec/test_wire_interop.py` run the round-trip across
two independent implementations over the shared corpus:

- **Positive:** Rust `litany` encodes each golden message → Python reference
  decodes → re-encodes → bytes equal the golden (and vice versa). This is the RFC
  0002 §6.8 "two conforming implementations MUST agree byte-for-byte" claim,
  demonstrated empirically — the protocol analogue of the language paper's
  differential oracle.
- **Negative:** both implementations reject the S7 rejection corpus, and land each
  malformed input in the **same** RFC 0003 §9.3 category (transport fault vs
  frame-level `error`). This reuses S7's negative table; it does not author a new
  one.

> **Precondition, not in-repo today.** The Python reference `hcpbin`/wire impl is
> a future precondition: [`ROADMAP_2026-2027.md`](../../../ROADMAP_2026-2027.md)
> line 108 says it is "available by Jun 26" (today is Jun 14), and a repo search
> finds **no** Python `hcpbin` module (only the Rust crate under
> `protocol/hcp/hcpbin/`). Until it lands, the interop leg is **blocked** (§7.1);
> the single-impl round-trip fixed point (§4.3) still runs on M1.

### 4.5 Protocol threat-model eval

A prose section sourced **entirely from existing artifacts** — it states no new
empirical claim of its own:

- **Resource-exhaustion DoS resistance** — `max_frame` bounds per-frame alloc,
  per-stream credit bounds per-stream buffering, liveness bounds idle cost (RFC
  0003 §14.4; RFC 0002 §11.1), backed by S7's `flood`/`soak`/`credit_stall`
  results (S7 §4.5, cited, not re-run here).
- **Tamper-evidence via canonical encoding** — non-canonical bytes cannot enter
  the `eventd` chain (RFC 0003 §14.3, §11 Conformance item 9), backed by the
  `eventd_admission` integration test (§4.4) and S7's decision-rule table.
- **Control-plane deadlock-freedom** — RFC 0003 §14.5, §11 Conformance items 5–7,
  backed by S7's `deadlock_freedom` harness.
- **Scope honesty** — mirror the language paper's "What Vaked does NOT guarantee"
  (paper §5.3): the wire gives canonical framing + flow-control liveness, **not**
  substrate confidentiality (delegated to the transport, RFC 0003 §14.1) and
  **not** authority/policy decisions (delegated to `preceptord`, RFC 0003 §14.2).

### 4.6 The honesty ledger (`examples/evaluation/protocol/METHODOLOGY.md`)

A direct adaptation of [`examples/evaluation/METHODOLOGY.md`](../../../examples/evaluation/METHODOLOGY.md):
every number in the paper's eval tables appears here labelled **Measured /
Projected / Retracted**, with environment provenance (host = M1 or `dev-cx53`,
toolchain versions, comparand library versions, iteration counts). "If a number
appears in a paper table but not in this ledger, treat it as unverified." Each

### 5.3 CI wiring

## 6. Acceptance criteria

S8 is done when **all** hold:

1. **Integration contract published** ([`docs/protocol/integration-contract.md`](../../../docs/protocol/))
   pinning the five surfaces of §1.1, each clause RFC-cited, versioned, and
   reviewed by a WP4 consumer — M1-reviewable (prose).
2. **Wire-size overhead table is Measured on M1**: `overhead.json` committed,
   `bench_wire.py` + `cargo bench --bench overhead` reproduce it, the table names
   its message corpus and comparand versions, and the paper cites the **measured**
   reduction — **not** the roadmap's "2–3 KB" estimate (§1.2 honesty guard).
3. **Replay determinism oracle committed** (`replay_baseline.json`) with
   `litanyreplay` proving byte-identity for the committed iteration count; the
   paper cites no figure larger than the committed data (§4.3).
4. **Cross-impl interop passes on M1** (positive: byte-for-byte agreement per RFC
   0002 §6.8; negative: same-category rejection per RFC 0003 §9.3) — **or**, if the
   Python reference impl has not landed, the leg is explicitly marked blocked in
   the ledger with the §7.1 dependency named (no silent skip).
5. **eventd chain-admission test passes on M1** (canonical chained, non-canonical
   never chained — RFC 0003 §11 Conformance item 9, §9.2, §10).
6. **Threat-model section** written from S7 results + RFC 0003 §14.1–§14.5 / RFC
   0002 §11, including the explicit "what the wire does NOT guarantee" scope
   (substrate confidentiality, authority decisions) — no new empirical claim
   originates here.
7. **Honesty ledger complete** (`examples/evaluation/protocol/METHODOLOGY.md`):
   every paper-eval number labelled Measured / Projected / Retracted with host
   provenance; the ≤10µs gate is **Projected** until a `dev-cx53` run flips it to
   **Measured (dev-cx53)**; no unledgered number appears in any table.
8. **No invented `hcp.wire` digest or conformance vectors** anywhere in the
   contract or paper (RFC 0003 OQ1, §5.6 / §11). Any clause needing the pinned
   digest is marked blocked on OQ1 (§7.1).
9. Every new test/eval file cites the RFC § it enforces, matching the
   adversarial-independence convention of
   ``protocol/hcp/hcpbin/tests/golden.rs`` (`../../../protocol/hcp/hcpbin/tests/golden.rs`).
