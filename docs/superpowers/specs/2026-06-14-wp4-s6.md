## 0. Scope correction (read first)

Two upstream wording issues this spec resolves explicitly rather than silently
picking a side:

1. **"3 daemons" is exactly `sandboxd` + `agent-supervisord` + `eventd`.** The WP4
   kickoff §Scope (`docs/superpowers/plans/2026-06-14-wp4-kickoff.md` lines 7–14)
   names these three as the WP4 MVP roster. **`agent-guardd` is NOT one of the
   three** — it is the separately-shipped `network`/`ebpf` vertical slice
   (`docs/runtime/agent-guardd.md`) with its own Python reference
   (`agent_guardd/`). It *coexists* on vakedos and its eBPF cgroup hooks sit
   alongside sandboxd's cgroup slice (kickoff lines 43, 66), so it appears in this
   spec only as **environmental context**, never as a test target.

2. **"on vakedos" means the vakedos NixOS *config* exercised as a VM test, not the
   bare-metal box.** WP4-S6 (Sep 4–17) **precedes** WP4-S7 ("vakedos deploy test
   (#114)", Sep 18–30) in the kickoff sprint table. S7 is the bare-metal
   `nixos-anywhere` deploy (`hosts/vakedos/README.md` §Install). S6 therefore
   cannot mean the physical EPYC — it is the integration test of the *same
   `nixosConfigurations.vakedos` config* booted in a NixOS VM test, where the three
   daemons are started under systemd and driven by a test script. S7 promotes that
   to bare metal.

This spec does **not** reproduce WP3-S3's RFC 0002 `hcpbin` citation-correction
block: WP4-S6 specifies no wire byte-encoding (see §0.3), so that correction is not
relevant here. It is cited only as the precedent for "cite only what you use."

### 0.1 The `vakedz lower` vs `vakedc lower` discrepancy (OPEN — do not silently pick)

The kickoff success criteria (line 73) says the daemons start "from systemd units
generated by **`vakedz lower`**" (the Zig compiler port). But the working,
shipping compiler is **`vakedc`** (`python3 -m vakedc lower`, used throughout
`docs/runtime/agent-guardd.md` §"Run it" and `hosts/vakedos/README.md`
§"Forward pointer"). `vakedz/` is the in-progress Zig port (`vakedz/src/lower.zig`).
**This spec treats `vakedc` as the reference unit-emitter and notes `vakedz` as the
eventual byte-parity port** (the #15 pattern). The discrepancy is flagged as an
open item in §7; S6 must not assert `vakedz` is the producer when it is not yet.

### 0.2 The systemd-unit emitter does not exist yet (load-bearing)

`docs/language/0012-lowering.md` §6 (the emitter table, line 333) lists
`systemd.units` for `fiber`/`parallel`/`surface` as **"(no-op today)"**, and §4.3
(lines 465–468) states the eBPF policy manifest, OTel config, **systemd unit
details**, and surface launcher are **deferred (§7)** — "the module slot that
references them exists; the artifacts themselves are no-ops today." So the kickoff
criterion "daemons start from generated systemd units" depends on an emitter that
is itself unbuilt. This spec makes that dependency explicit (§6) and specifies the
**minimal hand-written wiring module** S6 needs as a fallback so the integration
test is not blocked on the full §7 emitter (§3.3).

### 0.3 Transport scope — Litany Wire is out

In the WP4 timeframe the daemons talk over **reference transports**, not the wire:

- `agent-supervisord` ↔ control: the **`control.json`** file polled per tick plus
  the **oracle-port bridge** (`python3 -m eventd {verify,append,state,coldstart}`,
  frozen exit codes 0/2/3/4/5) — `docs/superpowers/specs/2026-06-12-agent-supervisord-design.md`
  §Shape and §"Reference phase bridge".
- `sandboxd` ↔ control: the **HTTP control plane** (`POST /spawn`, `POST /kill`,
  `GET /status/<pid>`, `GET /health`) — `agent_sandboxd/daemon.py` §Endpoints,
  default `127.0.0.1:7240`.
- The **eventd hash-chained log** (`/var/lib/vaked/...`) is the shared bus and the
  audit oracle for all three.

RFC 0005-over-Litany-Wire control frames are an explicit forward dependency
(supervisord design §Plan step 3: "RFC 0005 frame intake over Litany Wire;
retire the control file"). **S6 does not exercise the wire** (that is WP3 + a later
WP4 sprint). Normative inputs are RFC 0004 (lifecycle / state-dependency) and the
frozen eventd entry format (`eventd/core.py`); RFC 0003 is **not** cited because S6
specifies no framing.

## 1. Objective

Prove, as an automated and repeatable test, that the three WP4 daemons —
`sandboxd`, `agent-supervisord`, `eventd` — **boot together under systemd on the
`vakedos` NixOS configuration, wire up, and execute one end-to-end runtime
scenario whose every security-relevant decision lands on the eventd hash chain and
verifies**. Concretely, S6 delivers a NixOS VM integration test
(`tests/integration/vakedos/`) that:

1. Boots `nixosConfigurations.vakedos` + the runtime wiring module (§3.3) in a VM.
2. Starts the three daemons as systemd services and waits for them healthy
   (`systemctl is-active`, sandboxd `GET /health`, eventd log present + `verify`).
3. Drives the **canonical scenario** (§4): supervisord boots its lowered tree and
   takes the eventd writer; an agent workload is `spawn`ed through sandboxd into a
   real namespace boundary with real cgroup-v2 limits; a policy violation is
   refused; a producer is rewound and the dependent consumer is paused
   `stale_dependency` (RFC 0004 §6 lifecycle).
4. Asserts the **oracle agreement**: the live eventd log produced by the Zig daemon
   is byte-identical (per the frozen canonical-JSON contract, `eventd/core.py`) to
   what the Python references (`agent_sandboxd`, `eventd`) produce for the same
   scenario, and `eventd verify` (chain intact) + the RFC 0004 cold-start verdict
   match the oracle.

The verification spine is **differential against the existing Python oracles**
(the #15 pattern: Python fixes the bytes + the decision; the Zig daemons reproduce
them under live supervision).

### 2.2 Differential oracles (executable; the verification spine)

These run **on M1 today** (pure Python) and are the byte/decision authority the
live Zig daemons are diffed against:

- **`eventd/core.py`** — the FROZEN canonical-JSON hash-chain contract:
  `canonical_json` (sort_keys, `(",",":")`, `ensure_ascii=False`, `allow_nan=False`),
  `chain_hash` (`sha256(prev_hex || canonical(payload))`), `make_entry`,
  `verify_chain`. Any byte the Zig eventd emits MUST match this.
- **`eventd/__main__.py`** — the CLI oracle and **frozen exit-code contract**
  (`verify`=0/`tampered`=4/`locked`=5; `coldstart`=0 RUNNING / 3 PAUSED). S6
  asserts the live log passes `python3 -m eventd verify <log>` and that
  `python3 -m eventd coldstart <log> <consumer>` returns 3 at the paused step.
- **`eventd/statedep.py`** — `DependencyIndex.from_entries` / `verify_cold_start` /
  `gc_floor`: the RFC 0004 fold the scenario's pause verdict (§4 t3) is checked
  against.
- **`eventd/runtime.py`** — the run/step operational fold (`run_started`,
  `step_started`, `step_finished`, `run_finished`); the supervisor's
  `workflow_engine` emits these as the agent workload runs.
- **`agent_sandboxd/`** — the sandboxd Python reference and **event-payload
  contract** the Zig sandboxd must reproduce:
  - `agent_sandboxd/eventd.py`: payload builders `spawn_event`/`kill_event`/
    `access_event` — `kind ∈ {"sandbox_spawn","sandbox_kill","sandbox_fs_access"}`,
    `v=1`, write-ahead discipline, **no wall-clock in the hashed body**.
  - `agent_sandboxd/policy.py`: `decide_path` (deny-by-default fs verdict),
    `ProcessGrant.none ⇒ deny` (the violation path §4 t2).
  - `agent_sandboxd/namespace.py`: the `unshare --user --map-root-user --mount
    --pid --fork --net` plan (`build_namespace_plan`) and `probe_userns` — the
    exact isolation the Zig native-exec backend reproduces; **`enter_namespace(...,
    dry_run=True)` and `probe_userns()` run on M1**, real entry needs Linux.
  - `agent_sandboxd/cgroup.py`: `apply_cgroup_limits` writing `memory.max`/
    `pids.max`/`cpu.max`/`io.max` under `vaked/<agent-id>/`; **mock `cgroup_root`
    runs on M1**, real `/sys/fs/cgroup` writes need Linux.
  - `agent_sandboxd/daemon.py`: the HTTP control surface (`/spawn`, `/kill`,
    `/status/<pid>`, `/health`) the Zig daemon mirrors as a TCP listener (its own
    docstring: "the Zig port replaces this with a TCP listener on the same HTTP
    API").

### 2.3 Code dependencies (must exist before S6 *executes*)

Flake wiring (edit `flake.nix`): add `checks.<system>.vakedos-integration =

flake today** (`flake.nix` has only `nixosConfigurations.vakedos`, line 89) — S6
adds the first one.

### 3.2 The VM test harness (`vm-test.nix`)

Use the standard nixpkgs NixOS test framework. Verify the exact entry point at
implementation time (`pkgs.testers.runNixOSTest` is the current idiom;
`nixosTest` / `pkgs.nixosTest` are the older spellings — pick whatever the pinned
nixpkgs in `flake.lock` exposes; do **not** assert one here without checking the
lock). Shape:

```nix
# tests/integration/vakedos/vm-test.nix
{ pkgs, self, ... }:
pkgs.testers.runNixOSTest {
  name = "vakedos-3daemon-integration";
  nodes.machine = { ... }: {
    imports = [
      self.nixosModules.vakedosBase     # the vakedos config, minus disko/ZFS (§5.4)
      ./../../../hosts/vakedos/runtime-test.nix   # §3.3 wiring module
    ];
    virtualisation = { memorySize = 4096; cores = 2; };  # userns + cgroup need a real kernel
  };
  testScript = builtins.readFile ./driver.py;
}
```

Why a VM test and not the bare box: the daemons are **intrinsically Linux** —
namespace syscalls, cgroup-v2 writes, BEAM/OTP supervision, systemd. A VM test
boots the *same config* with a real Linux kernel and real systemd, drives it from
the host-side Python `testScript`, and is reproducible in CI on an `x86_64-linux`
builder. It is exactly the vehicle the vakedos README already gestures at with
`nixos-anywhere ... --vm-test` (README line 121) and
`nix build .#nixosConfigurations.vakedos...toplevel` (README line 116), promoted
to an asserting integration test.

### 3.3 The wiring module (`runtime-test.nix`) — the §0.2 fallback

Because the `0012 §7` `systemd.units` emitter is a no-op today, S6 ships a
**minimal hand-written `nixosModule`** that declares the three services, following
the §4.3 contract (it *installs and points at* artifacts; it does not re-declare
policy). This is the explicit stand-in until the emitter lands (§7 open item).

```nix
# hosts/vakedos/runtime-test.nix  (sketch — exact unit options at impl time)
{ config, pkgs, ... }:
{
  systemd.services.eventd = {
    wantedBy = [ "multi-user.target" ];
    serviceConfig = {
      ExecStart = "${pkgs.vaked-eventd}/bin/eventd serve /var/lib/vaked/log.jsonl";
      StateDirectory = "vaked";          # /var/lib/vaked, the disko dataset home
      Type = "notify";                   # or simple + a health gate
    };
  };
  systemd.services.agent-supervisord = {
    wantedBy = [ "multi-user.target" ];
    after = [ "eventd.service" ];        # supervisord owns the eventd writer (design §2)
    requires = [ "eventd.service" ];
    serviceConfig.ExecStart =
      "${pkgs.vaked-supervisord}/bin/agent_supervisord --gen /etc/vaked/otp "
      + "--eventd /var/lib/vaked/log.jsonl --control /run/vaked/control.json";
  };
  systemd.services.sandboxd = {
    wantedBy = [ "multi-user.target" ];
    after = [ "agent-supervisord.service" ];
    serviceConfig = {
      ExecStart = "${pkgs.vaked-sandboxd}/bin/sandboxd --listen 127.0.0.1:7240 "
        + "--eventd /var/lib/vaked/log.jsonl --cgroup-root /sys/fs/cgroup";
      Delegate = "yes";                  # cgroup-v2 subtree delegation (cgroup.py §Hierarchy)
      AmbientCapabilities = [ "CAP_SYS_ADMIN" ];  # userns/mount/cgroup; tighten in S5
    };
  };
}
```

Boot order encodes the dependency contract: **eventd first** (the audit spine and
single-writer target), **supervisord second** (it `verify_on_boot`s the log as a
hard precondition — supervisord design §Purpose item 2, and owns the writer),
**sandboxd third** (writes spawn/kill/access events through the same log). The
`vaked-{eventd,supervisord,sandboxd}` packages are the Zig/BEAM build outputs from
S1–S4; until those land, the module references the **Python references** as a
bring-up bridge (§6 "bring-up ladder").

### 3.4 Service-readiness gating

Each daemon exposes a health signal the driver waits on (no fixed `sleep`):

| Daemon | Readiness probe |
|---|---|
| eventd | log file exists at `/var/lib/vaked/log.jsonl` and `eventd verify` exits 0 (genesis ok) |
| agent-supervisord | `systemctl is-active agent-supervisord` = active **and** a `boot`/`tree_started` event observable via `eventd state` (design §Verification item 1, `task otp-smoke`) |
| sandboxd | `GET /health` → `{"status":"ok"}` (daemon.py do_GET) |

## 4. The canonical scenario (`scenario.py`)

One scenario, five steps, mirroring the supervisord design §Verification item 2
("Rewind demo … the epic's acceptance 3, end-to-end") and the RFC 0004 §5 rewind
example. The **same `scenario.py` drives both** the oracle generator (Python refs,
M1) and the VM driver (live daemons, Linux) — that shared definition is what makes
the differential assertion meaningful.

```
Setup: eventd log empty (genesis). supervisord booted its lowered tree
       (two workers: producer P, consumer C). current epoch = N.

t0  supervisord starts a workflow run R (run_started + step_started events).
      assert: eventd log has run_started{run=R}; chain verifies.

t1  spawn an agent workload through sandboxd:
      POST /spawn  (fixtures/spawn-allow.json: process_policy.grant="spawn_sandboxed",
                    fs allowed_write=[/tmp/work], agent_id="C")
      → real unshare(--user --mount --pid --fork --net) + cgroup vaked/C/ limits.
      assert (Linux): status="ok", pid set; child runs in its own pid ns.
      assert (oracle): eventd has sandbox_spawn{agent_id=C, action="allow", v=1}
                       BYTE-IDENTICAL to agent_sandboxd.eventd.spawn_event(...).

t2  policy violation — spawn with process grant=none:
      POST /spawn  (fixtures/spawn-deny.json: process_policy.grant="none")
      assert: HTTP 403 status="denied"; eventd has sandbox_spawn{action="deny",
              reason="process grant=none …"}; daemon exit/refusal matches
              agent_sandboxd.daemon.spawn deny path. (kickoff success criterion:
              "policy violations → eventd + exit 1".)

t3  rewind P below C's anchor (control.json: rewind P to step 50; C anchored @95):
      supervisord appends the RFC 0004 rewind event, pauses C, runs cold-start
      verify.
      assert: eventd coldstart <log> C  exits 3  (PAUSED(stale_dependency));
              statedep.verify_cold_start(C, entries) returns a StaleDependency
              with topology_epoch; C's lifecycle = PAUSED. P's worker never
              restarts (one_for_one isolation — supervisord design §"NOT here").

t4  re-anchor C at step 50; supervisord re-verifies:
      assert: eventd coldstart <log> C  exits 0 (RUNNING); C resumes; queued
              run R step delivered → step_finished; run_finished{status=completed}.

Teardown: eventd verify <log> exits 0 (whole chain intact); the full live log
          equals oracle/expected/scenario.events.jsonl byte-for-byte.
```

The deliver/pause/deny verdicts at t1/t2/t3/t4 are exactly what the Python oracles
(`spawn_event` action, `verify_cold_start`, exit codes) produce — that equality is
the acceptance assertion (§5.2, §8).

## 5. Test plan

### 5.1 The honest M1-vs-Linux split

WP4-S6 is **the opposite of WP3-S3**: WP3-S3 was pure transport-agnostic Rust, so
its acceptance ran fully on M1. **S6's live gate is intrinsically Linux** —
namespace syscalls, real cgroup-v2 writes, BEAM/OTP under systemd, a NixOS VM with
a real kernel. macOS M1 cannot run the live 3-daemon gate. What M1 *can* do is the
**oracle / compile / eval slice**, which is enough to develop and validate every
contract S6 asserts without a Linux box.

#### Runs on M1 today (assertable, no Linux)

| Item | How | Gate |
|---|---|---|
| Oracle log generation | `python3 tests/integration/vakedos/oracle/gen_oracle.py` → `expected/scenario.events.jsonl` + `verdicts.json` | M1 ✓ |
| eventd chain + verdict oracle | `python3 -m eventd verify/coldstart/state` over the generated log | M1 ✓ |
| sandboxd plan + policy + payload contract | `agent_sandboxd` with `enter_namespace(dry_run=True)`, `probe_userns()`, `apply_cgroup_limits(cgroup_root=<tmp mock>)`, `decide_path`, `spawn_event` byte check vs `eventd.canonical_json` | M1 ✓ |
| Cross-daemon event-payload contract | assert each `kind`/`v=1` payload the scenario emits canonicalizes identically across `agent_sandboxd` + `eventd` refs | M1 ✓ |
| Zig daemon compile-only | `zig build -Dtarget=x86_64-linux` in `daemons/sandboxd/` (and `daemons/eventd/` once it lands) — compile, **do not run** (namespace syscalls run only on Linux) | M1 ✓ |
| NixOS config **eval** | `nix flake check` restricted to eval (the vakedos config + the new check *evaluate*); `nix eval .#nixosConfigurations.vakedos.config.system.build.toplevel.drvPath` | M1 ✓ (eval only) |

Caveat made explicit: **building** the toplevel or **running** the VM test needs an
`x86_64-linux` builder. On M1 `nix flake check` will *evaluate* the VM test
derivation but cannot *build/run* it (it is a Linux VM). That is expected and is
not an M1 failure.

#### Linux-only — the actual S6 gate

| Item | Why Linux | Where it runs |
|---|---|---|
| The VM test itself (boot config, start 3 services, drive scenario) | needs a real Linux kernel + systemd + KVM | any `x86_64-linux` builder (CI runner, dev-cx53 when free, or `--build-on-remote` to vakedos) |
| Real `unshare`/`exec` into the namespace boundary (t1) | namespace syscalls are Linux | VM |
| Real cgroup-v2 enforcement under `vaked/<agent>/` | cgroup-v2 is Linux | VM |
| eventd Zig daemon byte-identical to Python oracle **under live concurrent writes** | the daemon binary is Linux; static byte-match is M1-checkable, live single-writer behavior is not | VM |
| supervisord OTP boot + lifecycle under systemd | BEAM + systemd | VM |

**dev-cx53 is not the constraint.** Its 6h off-limits window is a transient
autoresearcher hold (today, 2026-06-14); S6 runs Sep 4–17. Any `x86_64-linux`
builder satisfies the gate — CI's Linux runner is the primary target, dev-cx53 a
convenience, the EPYC itself only at S7. The gate is *Linux-required*, not
*dev-cx53-required*, and is **not blocked by the inability to run it on M1 today**.

### 5.2 The differential assertion (verification spine)

1. **Generate** `oracle/expected/scenario.events.jsonl` + `verdicts.json` from the
   Python refs (`gen_oracle.py`, runs `scenario.py` against `agent_sandboxd` +
   `eventd`). Commit both. Register `gen_oracle`'s self-check in
   `tests/spec/run_all.py` (a new `vakedos_oracle` module in `ALL_MODULES`,
   `tests/spec/run_all.py` line 36) so the oracle stays green on M1 in the
   `full`/`standard` tier.
2. **On the VM (Linux):** `driver.py` runs the *same* `scenario.py` against the
   live daemons, captures `/var/lib/vaked/log.jsonl`, and asserts:
   - the live log **byte-equals** `expected/scenario.events.jsonl` (hashed-payload
     bytes via `eventd.core.canonical_json`; the wall-time `_ts_ns` sidecar is
     excluded from the hash by construction — `agent_sandboxd/eventd.py` §discipline);
   - each step's verdict equals `verdicts.json` (t1 allow / t2 deny / t3 exit-3 /
     t4 exit-0 / health ok);
   - `eventd verify` exits 0 (chain intact end to end).
3. **Regression guard:** the t2 deny path and the t3 rewind→stale-dependency path
   are the two security-critical assertions; they must each FAIL if the respective
   daemon decision regresses (deny silently allowing, or a stale consumer staying
   RUNNING). This satisfies CLAUDE.md §5 (regression test that fails without the
   behavior).

### 5.4 ZFS / disko in the VM

`hosts/vakedos/disko.nix` declares a ZFS mirror (`/var/lib/vaked` is a ZFS dataset,
lines 106–110). The VM test does **not** need ZFS to validate daemon wiring: import
the vakedos config as a `nixosModules.vakedosBase` that **excludes `disko.nix` and
the ZFS/boot stanzas**, and let `StateDirectory=vaked` provide `/var/lib/vaked` on
the VM's plain rootfs. Refactor `configuration.nix` to split the
substrate-membrane facilities (userns, cgroup, sysctls, nftables, journald,
packages) into a reusable module that both the real host and the VM test import;
the ZFS/disko/GRUB/EPYC-tuning stays host-only. (This refactor is itself a small S6
deliverable — flagged in §7.) ZFS-backed tamper-evidence is validated at S7 on the
real box, not here.

## 7. Risks / open questions

- **`vakedz lower` vs `vakedc lower` (§0.1).** OPEN. Kickoff line 73 says `vakedz`;
  the shipping tool is `vakedc`. S6 uses `vakedc`-emitted (or, until the §7 emitter
  exists, hand-wired) units and treats `vakedz` as the eventual byte-parity port.
  Resolve with the lowering owner before S6's gate asserts a specific producer.
- **The systemd-unit emitter is a no-op today (§0.2).** S6's `runtime-test.nix` is
  an explicit hand-wired stand-in. Risk: it drifts from what the real emitter will
  produce. Mitigation: keep it minimal and §4.3-conformant (install + point, no
  policy), and add a follow-up to diff hand-wired vs emitted units once §7 lands.
- **vakedos config not VM-importable as-is (§5.4).** ZFS/disko/GRUB/EPYC tuning
  can't boot in a plain VM. Requires the base-module split (small refactor,
  S6-owned). Risk: the split diverges the VM config from the real host. Mitigation:
  both import the *same* membrane-substrate module; only host-physical stanzas
  differ.
- **NixOS test API spelling.** `runNixOSTest` vs `nixosTest` vs `pkgs.nixosTest`
  depends on the pinned nixpkgs in `flake.lock`. Do not hard-code; check the lock
  at impl time. The flake has **no `checks` today** — S6 adds the first.
- **`CAP_SYS_ADMIN` / userns in the VM.** The vakedos config sets
  `allowUserNamespaces` + a high `max_user_namespaces`; the VM inherits this. If
  the CI runner forbids nested userns (the agent-guardd doc notes nested-container
  attach refusal, `docs/runtime/agent-guardd.md` §"What is real"), the VM's own
  kernel (not nested) should still permit it — but verify on the chosen runner; a
  KVM-accelerated VM (`kvm-amd`, `configuration.nix` line 76) is the reliable path.
- **Supervisord reference-phase shape.** During bring-up supervisord shells out to
  the eventd oracle via ports (design §"Reference phase bridge"). S6 asserts
  *behavior* (lifecycle verdicts, log bytes), not the BEAM-native fold — the
  BEAM-native fold's parity is a later gate (#15 tooling), not S6's.
- **eventd transport ("append-only socket", kickoff line 60).** The kickoff's S1
  first-task mentions logging "via append-only socket"; the shipping eventd ref is
  file + flock single-writer (`eventd/log.py`, `EventLog(writer=True)`). S6 uses
  the file path as the shared bus (§0.3); if S4 introduces a socket front-end, the
  driver's readiness probe (§3.4) adapts, but the byte contract is unchanged.

## 8. Acceptance criteria

S6 is done when ALL hold. Criteria are **split by where they run** (the §5.1 honest
split) — do not claim the live gate runs on M1.

**M1-green (oracle / compile / eval — assertable on the dev laptop):**

1. `tests/integration/vakedos/oracle/gen_oracle.py` produces committed
   `expected/scenario.events.jsonl` + `verdicts.json`, and the oracle self-check is
   registered in `tests/spec/run_all.py` and green: `python3 tests/spec/run_all.py
   --tier standard`.
2. The cross-daemon event-payload contract holds: every scenario payload
   (`sandbox_spawn`/`sandbox_kill`/`sandbox_fs_access` `v=1`, `run_*`/`step_*`)
   canonicalizes identically via `eventd.core.canonical_json`, asserted in the
   oracle check. No daemon computes its own hash (the eventd oracle owns bytes).
3. `daemons/sandboxd/` compiles cross-target: `zig build
   -Dtarget=x86_64-linux` clean on M1 (compile-only; not run). Same for
   `daemons/eventd/` once S4 lands.
4. The vakedos config + the new VM-test check **evaluate**: `nix eval
   .#nixosConfigurations.vakedos.config.system.build.toplevel.drvPath` succeeds and
   `flake.nix` exposes `checks.x86_64-linux.vakedos-integration`.

**Linux-gated (the real S6 integration gate — runs on any `x86_64-linux`
builder, NOT M1, NOT dependent on dev-cx53):**

5. `nix build .#checks.x86_64-linux.vakedos-integration` builds and the VM test
   **passes**: the vakedos config boots; `eventd`, `agent-supervisord`, `sandboxd`
   reach active+healthy in that order (§3.4); the driver runs the §4 scenario.
6. **Real isolation:** t1 spawns through sandboxd into a real
   `unshare(user,mount,pid,net)` boundary with real cgroup-v2 limits under
   `vaked/C/`; the child is pid 1 of its own pid ns.
7. **Violation refused (kickoff criterion):** t2 (process grant=none) is denied,
   logged `action="deny"`, and surfaces as a refusal — matching the
   `agent_sandboxd.daemon` deny path.
8. **RFC 0004 lifecycle (kickoff/epic acceptance):** t3 rewind pauses C
   `stale_dependency` (`eventd coldstart` exit 3, `verify_cold_start` returns a
   `StaleDependency`) while P's worker does not restart; t4 re-anchor returns C to
   RUNNING (exit 0) and the run completes.
9. **Differential + integrity:** the live `/var/lib/vaked/log.jsonl` is
   byte-identical to `expected/scenario.events.jsonl` (hashed payload bytes) and
   `eventd verify <log>` exits 0 (whole chain intact).
10. The gate is reproducible on a CI Linux runner; **nothing in 5–9 requires
    dev-cx53 specifically** (it is one of several `x86_64-linux` builders), and
    the bare-metal EPYC deploy is explicitly deferred to WP4-S7 (#114).
