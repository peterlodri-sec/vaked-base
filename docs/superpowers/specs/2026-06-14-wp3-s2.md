# WP3-S2 вЂ” Litany frame layer (RFC 0003 В§4) + round-trip suite (design)

## Status

Design (2026-06-14). WP3 sprint S2 (Jul 9вЂ“Jul 23) of the HCP wire epic
(`docs/superpowers/plans/2026-06-14-wp3-kickoff.md`). Depends on WP3-S1
(`protocol/hcp/hcpbin`, codec types already landed). Creates the new
`protocol/hcp/litany/` Rust crate. Convention here: design в†’ plan в†’ impl; this
is the design.

This sprint delivers the **synchronous frame codec**: the canonical
delimitation of Votive Frames on a byte-stream (RFC 0003 В§4) and the frame
header encode/decode (RFC 0003 В§4.4 / В§4.4.1), plus a round-trip + golden +
violation test suite. It is **pure `bytes в†’ (header, opaque body) в†’ bytes`** with
**no I/O and no connection state machine**, so the whole deliverable is
`cargo test`-verifiable on the M1 dev machine.

## Scope decision (load-bearing вЂ” read first)

RFC 0003 (`protocol/rfcs/0003-litany-wire.md`) is the entire wire stack:
transport contract (В§3), frame delimitation (В§4), handshake/negotiation (В§5),
chapter/stream lifecycle (В§6), liveness (В§7), flow control (В§8), errors (В§9),
evidence (В§10), conformance (В§11), transport bindings (В§12). That is many
sprints of work and most of it requires async I/O, a connection state machine,
and (for vsock/namespaces) Linux. A 2-week sprint whose stated dependency is
"WP3-S1 codec types" and whose stated deliverable is a "round-trip suite" can
only own the part that is a **pure function over bytes**.

**Discriminator:** anything that is `bytes в†” frame-record` with no socket and no
state is IN; anything that needs a socket, a timer, or a multi-frame state
machine is OUT.

### IN (this sprint)

- **В§4.1 frame record** вЂ” `length := varint(byte-len of frame-bytes)` followed by
  `frame-bytes := header-bytes ++ body-bytes`; back-to-back, no padding (В§4.1).
- **В§4.2 length prefix** вЂ” unsigned LEB128 varint, reusing hcpbin's varint
  verbatim; strict overlong/non-minimal rejection; the `max_frame`-width overlong
  bound (В§4.2, В§4.3).
- **В§4.3 max-frame guard** вЂ” `length > max_frame` is a framing violation rejected
  *without reading the body*; `length == 0` is illegal (В§4.1, В§4.3).
- **В§4.4 / В§4.4.1 frame header** вЂ” encode/decode of `kind`/`corr`/`stream`/`seq`/
  `end` (the reserved `@0..@4` space, RFC 0002 В§4.2), byte-exact per the resolved
  layout in В§"Frame-header byte layout" below.
- **В§4.4.1 decoder invariant** вЂ” header/body boundary, optional-field omission,
  out-of-order / duplicate-tag / out-of-range rejection.
- **Frame-stream codec** вЂ” encode a `Vec<Frame>` to one byte string and decode a
  byte string to `Vec<Frame>` (the canonical-framing fixed point of В§4, В§10).
- **В§4.5 violation taxonomy** вЂ” every framing violation classified and surfaced
  as a typed `FrameError`; **no mid-stream resync** (В§4.5).
- **Round-trip + golden + violation test suite** (the named deliverable).

### OUT (deferred вЂ” later sprints, with reason)

| Deferred | Reason | Owner |
|----------|--------|-------|
| Handshake / version / schema-digest negotiation (В§5) | live exchange + `hcp.wire` digest is **TBD** (RFC 0003 Open question 1) | WP3-S2.5 / S3 |
| Chapter & stream lifecycle, multiplexing runtime (В§6) | connection state machine | WP3-S3 (RFC 0004 routing) |
| Liveness timers (В§7), flow-control runtime / credit (В§8) | async + timers | WP3-S4 (RFC 0005 control) |
| Transport-fault vs orderly close at the *connection* level (В§9.1/В§9.1a) | needs a live connection; S2 produces the typed *framing-violation* values only | WP3-S3+ |
| `eventd` admission / hash chain (В§10) | integration | WP3-S5 |
| Transport bindings stdio / unix / vsock (В§12) | I/O + Linux (vsock) | WP3-S6+ |
| `hcp.wire` control-frame *bodies* (Appendix A schema) | digest TBD; needs RFC 0002 В§6.2/В§6.3 record codec | WP3-S2.5 |
| в‰¤10Вµs/frame perf gate (kickoff success criterion) | threshold is `dev-cx53` hardware | WP3-S6 |

The frame *header* codec is in scope; frame *body* parsing is **not** вЂ” the wire
treats the body as an opaque self-delimiting `hcpbin` blob (RFC 0003 В§4.1, В§4.4:
"the wire treats `frame-bytes` as opaque except for parsing the header it itself
wrote"). S2 carries the body bytes verbatim and never decodes them.

## Objective

A `litany` crate exposing a synchronous, allocation-conscious frame codec such
that:

1. `decode_frame(&[u8], max_frame) -> Result<(Frame, consumed), FrameError>` reads
   exactly one frame record (В§4.1) and reports how many octets it consumed.
2. `encode_frame(&Frame) -> Vec<u8>` produces byte-identical output to the
   delimitation of В§4.1/В§4.2 (canonical framing вЂ” В§4, В§10).
3. The pair is a **fixed point**: `encode(decode(bytes)) == bytes` for every
   canonical input, and `decode(encode(frame)) == frame` for every frame (the
   В§6.8 / В§7.2 conformance shape, applied to the framing layer).
4. Every framing violation in the В§4.5 set is rejected with a typed error and
   *no* attempt at resynchronization.

## Frame-header byte layout (normative for this crate вЂ” resolves RFC 0003 gaps)

RFC 0003 В§4.4.1 is internally inconsistent in three places. The spec resolves
each by ranking **normative prose + RFC 0002 В§6.2** above example annotations,
and pins the result with a byte-exact golden vector. These are flagged as RFC
0003 errata to fold back when the RFC leaves Draft.

The header is a record encoded under hcpbin canonicality (RFC 0002 В§6.2):
**`field := varint(tag) value-bytes`, fields in strictly ascending tag order.**

| Tag | Name | Type | Wire form | Present when |
|-----|------|------|-----------|--------------|
| @0 | `kind` | enum u8 (0вЂ“4) | `varint(0)` `varint(kind)` | **always** (errata R2) |
| @1 | `corr` | uuid | `varint(1)` + **16 raw bytes** (big-endian, NOT length-prefixed) | **always** |
| @2 | `stream` | u64? | `varint(2)` `varint(stream)` | only if stream present |
| @3 | `seq` | u64? | `varint(3)` `varint(seq)` | only if stream present (R3) |
| @4 | `end` | bool | `varint(4)` `0x01` | only if `end == true` |

### Errata R1 вЂ” `kind` is fully tagged (`00 ++ varint(kind)`)

В§4.4.1's own annotated examples show `kind` *bare* (`00` = kind 0; `02` = kind 2,
no tag byte), but В§4.4 prose says the header is "encoded with the same `hcpbin`
rules as the body," and RFC 0002 В§6.2 fixes `field := varint(tag) value-bytes`.
The В§5.8 handshake transcript agrees: it shows `01 01` for "tag @0: kind=1" (two
bytes: tag `0x00`? вЂ” see note). **Resolution:** every header field is fully
tagged. `kind` is `0x00` (tag @0) followed by `varint(kind)`. The bare-byte
annotations in В§4.4.1 are a documentation shorthand, not the wire form.

Note on В§5.8: that transcript writes `01 01` and labels it "tag @0: kind=1"; the
leading `01` there is mislabeled (tag @0 is byte `0x00`). The correct encoding of
`kind=1` is `00 01` (tag @0, value 1). This spec uses `00 ++ varint(kind)`. (R1
erratum covers both the bare-byte and the В§5.8 mislabel.)

### Errata R2 вЂ” `kind` and `corr` are exempt from default-omission

RFC 0002 В§6.3 would *omit* an enum field whose value equals case 0, and would
omit any field at its default. The frame **class must always be explicit** on the
wire (the wire routes by it, В§6.3 of RFC 0003), and В§4.4.1's "Canonical
encoding" list emits `kind` unconditionally and `corr` unconditionally.
**Resolution:** `kind` (@0) and `corr` (@1) are **always emitted**, overriding
the generic В§6.3 default-omission for these two header fields. `stream`/`seq`/
`end` *do* obey omission (absent / false в‡’ omitted), per В§4.4.1 "Optional fields
are omitted." (`corr` is a `uuid`, which RFC 0002 В§6.3 already lists as
"no implicit default, always emit," so its always-emit is consistent; the
exception is only really needed for `kind`.)

### Errata R3 вЂ” header/body boundary; `seq` requires `stream`; `end` requires `stream`

В§4.4.1 step 6 says "any tag > @4 begins the frame body," but В§4.6 and RFC 0002
В§6.2 restart **body** tags at @1. A monotonic single tag space across header and
body is impossible (body @1 < header @4). **Resolution:**

- The **header** is the leading run of fields with tags in `{@0, @1, @2, @3, @4}`
  in ascending order; the **body** is the remainder, an opaque `hcpbin` blob whose
  own tags restart at @1 (RFC 0002 В§6.2, В§4.6). The decoder stops reading header
  fields after it has consumed the `@0..@4` prefix; everything after is body bytes
  it does not parse.
- The decoder distinguishes "header @1 (`corr`)" from "body @1" by **position**:
  `corr` is the first field after `kind` and is mandatory, so exactly one @1 is a
  header field (the one immediately after @0). After the optional `@2/@3/@4`
  prefix, the next field (whatever its tag, including a second @1) is body.
- `seq` (@3) MUST NOT appear unless `stream` (@2) is present (a per-stream
  sequence number with no stream is meaningless). `end` (@4) MUST NOT appear
  unless `stream` (@2) is present (В§4.4.1 step 5 normative rule: "@4 is INVALID if
  @2 is absent"). Violations в‡’ `FrameError::HeaderViolation`.
- Out-of-order header tags (e.g. @4 before @2) or duplicate header tags are
  framing violations (В§4.4.1 step 5, RFC 0002 В§6.2 rule 1).

### Worked header golden vectors (byte-exact, normative for S2)

Let `CORR = 11 22 33 44 55 66 77 88 99 aa bb cc dd ee ff 00` (16 bytes).

**G-H1 вЂ” single-shot request** (kind=0, corr, no stream/seq/end):

```
00 00                                  # @0 kind = varint(0)
01 11 22 33 44 55 66 77 88 99 aa bb cc dd ee ff 00   # @1 corr (16 raw bytes)
```
`header-bytes` length = 2 + 17 = 19 octets.

**G-H2 вЂ” multi-frame event, final chunk** (kind=2 event, stream=5, seq=9, end):

```
00 02                                  # @0 kind = varint(2)
01 <CORR 16 bytes>                     # @1 corr
02 05                                  # @2 stream = varint(5)
03 09                                  # @3 seq = varint(9)
04 01                                  # @4 end = true (0x01)
```

**G-F1 вЂ” full frame record** (G-H1 header + RFC 0002 В§10 body), reproducing
RFC 0003 В§4.6:

```
<len>                                  # varint(19 + 13) = varint(32) = 0x20
00 00                                  # header: @0 kind=0
01 <CORR 16 bytes>                     # header: @1 corr
01 07 66 73 2e 72 65 61 64             # body: @1 string len 7 "fs.read"
02 03 6b 65 79                         # body: @2 bytes len 3 0x6b 0x65 0x79
```
`frame-bytes` = 19 (header) + 13 (body) = 32 octets в‡’ `length = 0x20`.
Total record = 1 + 32 = 33 octets. (Body bytes are the RFC 0002 В§10 vector,
carried opaque.)

These three vectors are committed as the canonical fixtures; the test suite
encodes to them and decodes from them.

## Algorithm / design

### In-memory model (`frame.rs`)

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum FrameKind { Request = 0, Response = 1, Event = 2, Control = 3, Error = 4 }

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct Frame {
    pub kind: FrameKind,         // @0, always present (errata R2)
    pub corr: [u8; 16],          // @1, always present (uuid, 16 raw bytes)
    pub stream: Option<u64>,     // @2
    pub seq: Option<u64>,        // @3 (only valid when stream is Some)
    pub end: bool,               // @4 (only true when stream is Some)
    pub body: Vec<u8>,           // opaque hcpbin body bytes (NOT parsed by S2)
}
```

`body` is the bytes after the header in `frame-bytes`; the codec never inspects
or validates it (RFC 0003 В§4.1, В§4.4). Whether the body is canonical hcpbin is a
*frame-level* concern (RFC 0003 В§9.2) owned by a later sprint; S2 round-trips it
verbatim.

### Header codec (`header.rs`)

`corr` is the gap WP3-S1 leaves: hcpbin has **no `uuid` type** (confirmed вЂ”
`hcpbin/src/lib.rs` implements only varint/bool/bytes/hash; uuid is a 16-byte raw
field per RFC 0003 В§4.4.1 "16 raw bytes вЂ¦ not length-prefixed"). RFC 0002 В§6.2/
В§6.3 tagged-record + default-omission are also explicit TODOs in `lib.rs` (lines
494вЂ“495). **Decision: hand-roll a bespoke 5-field header codec in `litany`** using
hcpbin's `Reader`/`Writer` primitives, rather than extend hcpbin. Rationale
(Simplicity First): the header is a fixed 5-field shape with non-generic rules
(R1вЂ“R3), the only consumer is the wire, and leaving the general record/uuid codec
to whichever sprint owns `hcp.wire` bodies keeps S2 surgical and keeps hcpbin's
scope (RFC 0002 В§6) clean.

`encode_header(&Frame, &mut hcpbin::Writer)`:
1. `w.put_uvarint(0); w.put_uvarint(kind as u64);`  // @0, always
2. `w.put_uvarint(1); w.put_raw(&corr);`            // @1, always, 16 raw bytes
3. if `stream` Some: `w.put_uvarint(2); w.put_uvarint(stream);`
4. if `stream` Some && `seq` Some: `w.put_uvarint(3); w.put_uvarint(seq);`
5. if `end`: `w.put_uvarint(4); w.put_bool(true);`  // bool true = 0x01

Encoder invariants asserted before emit (debug + release): if `seq.is_some()` or
`end` then `stream.is_some()` (R3); `kind` numeric in 0..=4. These are the
"wire layer guarantees" of В§4.4.1; a `Frame` that violates them is a programmer
error, not decodable input вЂ” return `FrameError::HeaderViolation` from a checked
`encode` entry point rather than panicking.

`decode_header(&mut hcpbin::Reader) -> Result<Frame-without-body, FrameError>`:
1. Read tag (uvarint); MUST be 0 в‡’ else `HeaderViolation`. Read `kind` (uvarint),
   MUST be в‰¤ 4 в‡’ else `KindOutOfRange` (a framing violation, В§4.4.1 step 1).
2. Read tag; MUST be 1 в‡’ else `HeaderViolation`. Read 16 raw bytes into `corr`
   (EOF в‡’ `UnexpectedEof`).
3. Peek next tag. If 2: read `stream` (uvarint). If absent, stream = None and
   steps 4вЂ“5 must not see @3/@4.
4. If next tag 3: require stream present (R3) else `HeaderViolation`; read `seq`.
5. If next tag 4: require stream present (R3) else `HeaderViolation`; read `end`
   (hcpbin `get_bool`, which already rejects non-`0x00/0x01`, В§6.4).
6. Enforce ascending order: a tag в‰¤ the last header tag read в‡’ `HeaderViolation`
   (duplicate / out-of-order, В§4.4.1 step 5). Once a tag > 4 (or a second @1, or
   anything not continuing the @2в†’@3в†’@4 run) is seen, the header is finished and
   that field begins the body.
7. Remaining `Reader` bytes are the opaque body.

Header decode is driven inside the `frame-bytes` slice the record layer already
delimited, so "remaining bytes = body" is well-defined.

### Record codec (`record.rs`)

`encode_frame(&Frame, max_frame) -> Result<Vec<u8>, FrameError>`:
1. Encode header + append body into a scratch `Writer` в‡’ `frame-bytes`.
2. `if frame_bytes.len() == 0 || frame_bytes.len() as u64 > max_frame { Err(FrameTooLarge) }`
   (length 0 illegal В§4.1; > max_frame В§4.3).
3. Prepend `varint(frame_bytes.len())` в‡’ record.

`decode_frame(&[u8], max_frame) -> Result<(Frame, usize), FrameError>`:
1. `len = get_uvarint(input)` вЂ” strict/minimal (hcpbin already enforces non-minimal
   and overlong rejection, В§6.1; reused verbatim per В§4.2). On `NonMinimal`/
   `Overlong` в‡’ map to `FramingViolation`.
2. **Overlong-by-construction bound (В§4.2):** reject any `length` varint encoded
   in more bytes than the minimal width of `max_frame`, even if numerically valid
   вЂ” the canonical-width guard. Compute `minimal_width(max_frame)` once.
3. `if len == 0 в‡’ FramingViolation` (В§4.1).
4. `if len > max_frame в‡’ FrameTooLarge` вЂ” **without reading the body** (В§4.3).
5. Slice exactly `len` octets as `frame-bytes`; short input в‡’ `UnexpectedEof`
   (mid-frame EOF is a framing violation, В§4.5).
6. `decode_header` over the slice; the rest is `body`.
7. Return `(Frame, 1?+len)` = `length-prefix-bytes + len`.

**Frame-stream helpers** (the round-trip target of В§4, В§10):

```rust
pub fn encode_frames(frames: &[Frame], max_frame: u64) -> Result<Vec<u8>, FrameError>
pub fn decode_frames(bytes: &[u8], max_frame: u64) -> Result<Vec<Frame>, FrameError>
```

`decode_frames` loops `decode_frame`, advancing by `consumed`, until the input is
exhausted; back-to-back with no padding (В§4.1). A trailing partial record в‡’
`UnexpectedEof`. `encode_frames` concatenates `encode_frame` outputs. The pair is
the canonical-framing fixed point of В§4 / В§10.

### Error model (`lib.rs`)

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum FrameError {
    UnexpectedEof,                  // mid-frame / mid-header EOF (В§4.5)
    FramingViolation,               // non-minimal/overlong/zero length (В§4.1, В§4.2, В§4.5)
    FrameTooLarge { len: u64, max: u64 }, // length > max_frame (В§4.3)
    KindOutOfRange(u64),            // kind not in 0..=4 (В§4.4.1 step 1)
    HeaderViolation,                // out-of-order/dup tag, seq/end without stream (R3, В§4.4.1)
    Codec(hcpbin::DecodeError),     // wrapped primitive-decode failure
}
```

Every variant maps to a В§4.5 "framing violation": the record envelope or header
could not be trusted, so (in the live wire, later sprints) the connection would
abort with **no resync** (В§4.5). S2 only returns the typed value; it performs no
I/O and tears nothing down. This is the clean boundary the spec keeps: S2 owns
*classification*, the connection runtime (S3+) owns *reaction* (abort vs orderly
close, В§9.1/В§9.1a).

**Explicitly NOT in S2's error model:** the *frame-level* `error` Votive Frame
(RFC 0003 В§9.2) and non-canonical-*body* rejection (В§9.2, RFC 0002 В§6.8). The body
is opaque here; body canonicality is a later sprint that decodes bodies.

## Test plan

All tests are `cargo test` in `protocol/hcp/litany/`, **fully runnable on the M1
dev machine** (pure functions, zero I/O, no Linux dependency). Follow the
adversarial-independence convention of `hcpbin/tests/golden.rs`: expectations are
derived only from the RFCs (and the errata resolutions above), written against
the public API, citing the exact RFC clause.

### Local (M1, `cargo test`) вЂ” the entire deliverable

1. **Golden header (`golden_header.rs`)** вЂ” encode produces G-H1, G-H2 byte-exact;
   decode of G-H1/G-H2 yields the expected `Frame` fields. Cite В§4.4.1.
2. **Golden record (`golden_record.rs`)** вЂ” G-F1 byte-exact, reproducing RFC 0003
   В§4.6 (`length = 0x20`, header + opaque В§10 body). Decode recovers the frame and
   the verbatim body bytes. Cite В§4.1, В§4.6.
3. **Round-trip fixed point (`roundtrip.rs`)** вЂ”
   - `decode_frame(encode_frame(f)) == f` for a matrix of frames: every
     `FrameKind`; corr all-zero / all-0xff / mixed; stream None vs Some; seq None
     vs Some(0)/Some(large); end true/false; body empty / small / spanning a
     multi-byte length prefix (в‰Ґ128 octets, forcing a 2-byte `length`).
   - `encode_frame(decode_frame(bytes)) == bytes` for the golden byte fixtures
     (the В§6.8 re-encode fixed point applied to framing).
   - `decode_frames(encode_frames(v)) == v` for multi-frame streams (back-to-back,
     В§4.1) including interleaved kinds and a stream id retired then a fresh frame.
4. **Violations (`violations.rs`)** вЂ” each rejected with the right `FrameError`:
   - `length == 0` в‡’ `FramingViolation` (В§4.1).
   - non-minimal `length` (`0x80 0x00`, i.e. 0 overlong) в‡’ `FramingViolation`
     (В§4.2; mirrors hcpbin `uvarint_strict_reject_overlong`).
   - `length` overlong-by-construction (more bytes than minimal width of
     `max_frame`) в‡’ `FramingViolation` (В§4.2 width bound).
   - `length > max_frame` в‡’ `FrameTooLarge`, asserting the body slice is **not**
     read (e.g. truncated input after the length still errors as too-large, not
     EOF) (В§4.3).
   - mid-frame EOF (input shorter than declared `length`) в‡’ `UnexpectedEof` (В§4.5).
   - `kind` tag missing / not @0 в‡’ `HeaderViolation`; `kind` value 5 в‡’
     `KindOutOfRange` (В§4.4.1 step 1).
   - missing `corr` / short corr (<16 bytes) в‡’ `UnexpectedEof`/`HeaderViolation`.
   - out-of-order header tags (@2 then @1-as-header, @4 before @2) в‡’
     `HeaderViolation` (В§4.4.1 step 5).
   - duplicate header tag (two @2) в‡’ `HeaderViolation`.
   - `seq` (@3) without `stream` (@2) в‡’ `HeaderViolation` (R3).
   - `end` (@4) without `stream` (@2) в‡’ `HeaderViolation` (В§4.4.1 step 5 / R3).
   - `end` byte `0x02` в‡’ `Codec(InvalidBool(0x02))` (hcpbin В§6.4, reused).
5. **No-resync assertion** вЂ” a frame-stream with a corrupt record in the middle:
   `decode_frames` errors at that record and does **not** attempt to scan forward

## Acceptance criteria

1. `protocol/hcp/litany/` exists with the layout above; `cargo build` and
   `cargo test` pass on the M1 dev machine with **zero** I/O and zero Linux
   dependency.
2. `encode_frame` reproduces G-H1, G-H2, and G-F1 (the В§4.6 record)
   **byte-for-byte**; `decode_frame` recovers the exact `Frame` and the verbatim
   opaque body.
3. Round-trip fixed point holds: `decode(encode(f)) == f` over the test matrix,
   and `encode(decode(bytes)) == bytes` over the golden fixtures (В§6.8 / В§4
   canonicality). `decode_frames(encode_frames(v)) == v` for multi-frame streams.
4. Every В§4.5 + В§4.4.1 violation in the test plan is rejected with the specified
   typed `FrameError`; **no mid-stream resync** is attempted (assertion #5 in the
   test plan).
5. `length` reuses hcpbin's strict varint (no duplicated varint code); the
   `max_frame` guard rejects oversize lengths **without reading the body**; the
   overlong-by-construction width bound is enforced.
6. The three RFC 0003 errata (R1 fully-tagged `kind`, R2 `kind`/`corr` always
   emitted, R3 header/body boundary + `seq`/`end` require `stream`) are encoded in
   the codec and pinned by golden vectors, and recorded in the spec for fold-back
   into RFC 0003 before it leaves Draft.
7. A perf benchmark for `encode_frame`/`decode_frame` exists and reports an M1
   baseline; the в‰¤10Вµs/frame **threshold** is explicitly deferred to WP3-S6 on
   `dev-cx53` (no S2 criterion depends on dev-cx53).
8. Test files are adversarially independent (expectations from RFCs + errata,
   each citing its clause), matching the `hcpbin/tests/golden.rs` convention.

## Risks / open items

- **RFC 0003 В§4.4.1 inconsistencies (R1вЂ“R3)** are resolved here by spec fiat
  ranking prose over example annotations. If the RFC authors resolve them
  differently when Draft closes, the golden vectors and codec must follow the RFC;
  the resolutions are isolated in `header.rs` + `golden_header.rs` to make that a
  contained change.
- **`hcp.wire` digest / В§11 conformance vectors are TBD.** S2 deliberately scopes
  *below* them; handshake conformance cannot be tested until Open question 1
  closes. This is a known, accepted gap, not a defect.
- **Body opacity.** S2 never validates body canonicality (В§9.2 / RFC 0002 В§6.8);
  a non-canonical body round-trips verbatim through S2. That is correct for the
  wire framing layer (the body is opaque, В§4.1) and is the later sprint's concern.
