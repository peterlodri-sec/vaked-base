## 0. Reading-note correction (load-bearing; do this first)

The WP3 kickoff plan (`docs/superpowers/plans/2026-06-14-wp3-kickoff.md` line 43) and issue #167 say golden vectors derive from **"RFC 0002 ¬ß4 / Appendix A."** This wording is **stale and wrong** and S6 must correct it as a docs deliverable (¬ß6 below):

- The byte encoding ("`hcpbin` encoding rules") lives in **RFC 0002 ¬ß6** (`0002-hcplang.md:439`), not ¬ß4. RFC 0002 ¬ß4 ("The Votive Frame model", line 194) is the *frame model*, and ¬ß4.2 (line 226) explicitly states the frame header (`kind`/`corr`/`stream`/`seq`/`end`) is **supplied/validated by the wire layer and never in `hcpbin`** ("The wire-layer representation of header fields is normative in RFC 0003 ¬ß4", `0002-hcplang.md:241`).
- There is **no Appendix A of golden vectors**. RFC 0002 *does* have an "Appendix A: Generated code examples" (`0002-hcplang.md:1287`) ‚Äî Zig/Erlang codegen, not byte vectors. The actual golden byte vectors are in the **worked examples**: ¬ß6.1.1 (primitives, `0002-hcplang.md:473`), ¬ß6.3.1 (default omission, line 622), ¬ß6.6.1 (lists/maps, line 825), ¬ß6.7.1 (unions/enums, line 774), and ¬ß10 / ¬ß10.1 (full frame bodies, lines 1009 / 1079). The frame-header-on-wire vectors are in **RFC 0003 ¬ß4.4.1** (`0003-litany-wire.md:293`) and ¬ß4.6 (worked framing example, line 386).

The hcpbin crate already documents the correct scope (`protocol/hcp/hcpbin/src/lib.rs:22-23`, `:497-498`): "the frame header (kind/corr/stream/seq/end) lives in the WIRE layer (RFC 0003), explicitly NOT in hcpbin (RFC 0002 ¬ß4.2, ¬ß6 scope note)." S6 propagates that correction into the kickoff plan and any docs that still cite "¬ß4 / Appendix A."

## 1. Objective

Two deliverables, equal weight (the sprint is literally "Docs + perf baseline"):

1. **Docs.** Ship publishable documentation for the `litany` frame layer and the `hcpbin` codec: crate-level rustdoc, a `protocol/hcp/litany/README.md`, and a perf-results writeup that becomes the **WP3 results section of the paper eval (#113)** required "ready by Oct 15" (kickoff line 55). 2. **Perf baseline.** A reproducible, spec-anchored microbenchmark that measures **encode and decode latency per frame** for the Litany wire frame (header + body, RFC 0003 ¬ß4.4) and compares it, **at the codec boundary only**, against protobuf and CBOR encodings of the *same logical message*. Headline acceptance: **‚â§10¬µs per direction per frame** on `dev-cx53` for the designated profile (¬ß4.2). The number is produced on `dev-cx53`; M1 certifies everything *except* the absolute latency (¬ß5).

### 1.1 The 10¬µs target ‚Äî disambiguated (normative for this sprint)

The kickoff says "‚â§10¬µs per frame encode/decode" (line 54), which is ambiguous. **S6 adopts the per-direction reading** (stricter, more useful): for the designated profile, `encode(frame) ‚â§ 10¬µs` **and** `decode(frame_bytes) ‚â§ 10¬µs` measured independently, not a combined round-trip budget. Reported as median and p99 from criterion; the p99 is the gate, the median is for the paper.

This is a `dev-cx53` figure (8-core, "no alloc hot path", kickoff line 54). M1 (Apple ARM, different microarchitecture, different memory subsystem) **cannot certify a Linux x86 latency number** and S6 must not claim it does (¬ß5).

## 4. Algorithm / design

### 4.1 Frame corpus (with a size distribution ‚Äî the 10¬µs gate is meaningless on tiny frames only)

The ¬ß10 `ToolCallRequest` body is ~14 bytes and encodes in tens of nanoseconds; a corpus of only small frames passes 10¬µs trivially and proves nothing. The corpus therefore spans sizes, and the gate (¬ß4.2) is pinned to the profile where 10¬µs actually bites.

| Profile | Frame shape (RFC anchor) | Approx `frame-bytes` size | Purpose |
|---|---|---|---|
| `P1_small_request` | Single-shot `ToolCallRequest`, header `{kind=0, corr}` only (RFC 0003 ¬ß4.6, line 386; body RFC 0002 ¬ß10) | ~31 B (16 B uuid + body) | Lower bound; sanity; the spec's own worked example. |
| `P2_response_union` | `ToolCallResponse` with `ToolResult` union arm + omitted default `severity` (RFC 0002 ¬ß10 lines 1060‚Äì1077, ¬ß6.7.1) | ~50 B | Exercises union arm-tag + byte-len framing and default omission. |
| `P3_event_stream` | `event` with full header `{kind=2, corr, stream, seq, end=true}` (RFC 0003 ¬ß4.4.1 lines 343‚Äì351) | ~40 B | Exercises every optional header field ‚Äî header encode path that P1 skips. |
| `P4_chunked_large` | `ToolCallResponse` with a `binary` arm payload sized to **`max_frame` floor (64 KiB, RFC 0003 ¬ß11 item 3)** minus header | ~64 KiB | **The 10¬µs gate profile.** Large `bytes` copy is where per-frame latency is real. |

Each corpus message is built as a real `litany` frame (header + hcpbin body) and `corpus_golden.rs` asserts:
1. The body bytes equal the RFC ¬ß6/¬ß10 golden vector (for P1/P2 these are byte-exact from the RFC; for P3 the header bytes match RFC 0003 ¬ß4.4.1; for P4 only the framing structure is asserted, payload is fixed-seed pseudorandom).
2. The ¬ß6.8 fixed point: `encode(decode(encode(frame))) == encode(frame)`.

Timing runs **only after** these pass ‚Äî a fast-but-wrong codec is not a baseline.

### 4.2 The 10¬µs gate (`benches/frame_latency.rs`, hcpbin/litany only)

Criterion benchmark groups, one per profile, two functions each:
- `encode/<profile>`: build the typed frame once outside the timed loop; time `litany::encode_frame(&frame) -> Vec<u8>` (or the no-alloc `encode_into(&frame, &mut buf)` variant once it exists, ¬ß4.4).
- `decode/<profile>`: pre-encode `frame_bytes` once; time `litany::decode_frame(&frame_bytes) -> Frame`.

**Gate (normative):** for **`P4_chunked_large`**, both `encode` and `decode` p99 ‚â§ 10¬µs on `dev-cx53` (¬ß1.1, per-direction). P1‚ÄìP3 are reported for the paper but are not the gate (they are expected to be sub-microsecond and so the 10¬µs ceiling is not informative for them). Criterion config: `sample_size = 200`, `measurement_time = 10s`, `warm_up_time = 3s`.

### 4.3 Codec comparison (`benches/codec_compare.rs`)

Apples-to-apples at the **codec boundary**, never the transport. The comparison measures encode+decode of the **same logical message** (RFC 0002 ¬ß10 `ToolCallRequest`) in three encodings:

| Arm | Crate | What is timed |
|---|---|---|
| `hcpbin` | `hcpbin` + `litany` (this repo) | `encode_frame` / `decode_frame` of the ¬ß10 frame (header + body). |
| `protobuf` | `prost` (codec only; **do not** pull `tonic`/HTTP-2) | `Message::encode` / `Message::decode` of the `hcp_core.proto` mirror. |
| `cbor` | `ciborium` (or `minicbor` ‚Äî decide at impl time) | serialize / deserialize the equivalent CBOR map. |

**Honesty constraint (normative):** this is a **codec** comparison. gRPC's value is its transport (HTTP/2, multiplexing) ‚Äî that is RFC 0003's concern, not the codec's. Folding tonic's HTTP/2 stack in would measure the transport and make "vs grpc" dishonest. We therefore compare against **prost (protobuf codec)** and state in the writeup that the gRPC framing layer is out of scope for the codec baseline, with a forward-pointer to a future transport-level comparison (Litany wire vs gRPC HTTP/2) if #113 needs it. The protobuf arm is labelled "protobuf (prost codec)", not "gRPC", in all outputs.

**Versions:** crate versions are **pinned at implementation time**, not guessed here (per CLAUDE.md: do not guess versions). The `Cargo.toml` stub lists the crate *names* only; `cargo add` resolves exact versions in the dev shell on `dev-cx53` / locally.

### 4.4 No-alloc hot path (the cross-arch correctness proxy ‚Äî see ¬ß5.2)

The kickoff requires "no alloc hot path" (line 54). The bench drives the **allocation-free** encode/decode surface:
- `encode_into(&frame, &mut Vec<u8>)` reusing a caller buffer across iterations (the `Writer` in `hcpbin` already appends into an owned `Vec`; litany exposes a buffer-reuse variant). Timed loop must not allocate beyond a one-time buffer grow.
- `decode_frame` over `litany`'s borrowed `Reader<'a>` (`hcpbin/src/lib.rs:193`, zero-copy slice reader) ‚Äî `get_bytes` returns `&'a [u8]` (line 343), so the body large-payload path borrows rather than copies; only `to_vec()`-using convenience wrappers allocate and the bench avoids them.

Allocation count on the hot path is **deterministic across architectures** ‚Äî this is what M1 can certify (¬ß5.2).

## 5. Test plan ‚Äî what M1 certifies vs what needs dev-cx53/Linux

The dev-cx53 build target is **OFF-LIMITS for autoresearcher windows** and is in any case a remote Linux x86 host; M1 is the local dev machine (confirmed: `cargo 1.95.0`, `zig 0.16.0`). The split is the spine of this sprint.

### 5.1 Locally on M1 ‚Äî CERTIFIABLE

| Check | Command | What it proves |
|---|---|---|
| Bench harness compiles | `cargo bench --manifest-path protocol/hcp/perf/Cargo.toml --no-run` | criterion + prost (build.rs) + cbor arms build; `[[bench]]` wiring correct. |
| Corpus correctness | `cargo test --manifest-path protocol/hcp/perf/Cargo.toml --test corpus_golden` | Every corpus message encodes to its RFC ¬ß6/¬ß10 golden bytes and satisfies ¬ß6.8 fixed point. **Arch-independent.** |
| Codec arms agree on a message | (in `corpus_golden.rs`) | All three arms round-trip the ¬ß10 message; hcpbin arm matches the RFC byte vector. |
| **Zero-alloc hot path** | `cargo test --manifest-path protocol/hcp/perf/Cargo.toml --test alloc` (dhat-rs or a counting `#[global_allocator]`) | Hot-path encode/decode performs **0 allocations** after warm-up. **Deterministic across arch ‚Üí this is the strongest M1 acceptance proxy for the perf design.** |
| Bench *runs* and emits a number | `cargo bench --manifest-path protocol/hcp/perf/Cargo.toml` | criterion produces medians/p99 ‚Äî but the **absolute M1 numbers are advisory only** (recorded as "M1 reference", never compared to 10¬µs). |
| Relative shape on M1 | criterion output | The *ranking* hcpbin vs protobuf vs cbor and the *scaling* across P1‚ÜíP4 are informative on M1 even though absolute ¬µs differ; recorded as a sanity cross-check, not a gate. |

Zig is not exercised by S6 (the litany crate is Rust); `zig build -Dtarget=x86_64-linux` is irrelevant here and is noted only to confirm S6 has no Zig surface.

### 5.2 On dev-cx53 (Linux x86, 8-core) ‚Äî REQUIRED for the headline number

The ‚â§10¬µs gate (¬ß4.2) is **only valid on dev-cx53** and can be produced **only when dev-cx53 is in-bounds** (not during an autoresearcher reservation). Reproducibility controls (Linux-specific, listed so the run is repeatable):

- Pin to one isolated core: `taskset -c 3 cargo bench ...`, criterion single-threaded.
- `cpufreq` governor = `performance` (disable turbo variance): `sudo cpupower frequency-set -g performance`.
- Disable ASLR jitter for the run if needed; record kernel + CPU model + rustc version in the writeup.
- Run the gate three times; the gate passes iff **P4 encode p99 ‚â§ 10¬µs AND P4 decode p99 ‚â§ 10¬µs** in all three.

The dev-cx53 number, the comparison table (hcpbin/protobuf/cbor across P1‚ÄìP4), and the methodology go into `docs/superpowers/research/2026-09-17-wp3-perf-results.md` ‚Üí the #113 paper results section.

### 5.3 Regression guard

After the first green dev-cx53 run, commit a criterion baseline (`--save-baseline wp3-s6`) so later sprints (S7 hardening) detect regressions with `--baseline wp3-s6`. On M1, the regression guard is the zero-alloc test (a new allocation on the hot path fails locally before it ever reaches dev-cx53).
