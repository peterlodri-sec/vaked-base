## 1. Objective

Make the Litany wire (RFC 0003) feed the **eventd** append-only, hash-chained audit log (the immutable leg of the 1.0 epic, [`docs/superpowers/specs/2026-06-12-eventd-design.md`](2026-06-12-eventd-design.md)) so that:

1. Every admitted Litany frame, plus every connection-lifecycle moment, becomes a durable, tamper-evident eventd entry (RFC 0003 ¬ß10, ¬ß11 item 9).
2. The state-dependency control frames of RFC 0004 ¬ß2 (`DependencyRegistration`, `ConsumerCheckpoint`, `RewindEvent`) ‚Äî which **MUST** be durably logged before consumption (RFC 0004 ¬ß3.1) ‚Äî are written through this same path and fold into the existing `DependencyIndex` / GC-floor / cold-start machinery ([`eventd/statedep.py`](../../../eventd/statedep.py)).
3. The bridge is **byte-faithful to the Python eventd oracle**: the entry/chain bytes a Rust writer produces are accepted by `python3 -m eventd verify` and are byte-identical to a committed golden, exactly as `vakedz` is pinned to `vakedc` ([`vakedz/test/crossverify.sh`](../../../vakedz/test/crossverify.sh)).

This sprint is the **integration glue between two already-specified layers**, not a new encoding. It is dependency-blocked (see ¬ß7) and is delivered as a planning artifact now.

### Non-goals
- Not a new chain format. `eventd/core.py` is **FROZEN** (its module docstring, lines 17‚Äì37) and cross-verified against `tools/ralph/ralphcore.py` and `vakedz/cache.zig`. S5 must not alter it.
- Not the Zig eventd daemon. That is the #15-pattern port, scheduled WP4-S4 ("Zig port (WP4-S4) replaces by Sep", kickoff line 47); the Python eventd is its oracle and ours.
- Not RFC 0004's compiler passes (DAG validation = order 5, lives in `vakedc`) nor the zero-copy scan path (order 8). RFC 0004 ¬ß8.
- Not perf tuning (WP3-S6) nor stress hardening (WP3-S7).

## 3. The central design decision (surface, do not silently pick)

There is a **documented contradiction in the repo** about *what bytes eventd hashes*:

- **RFC 0003 ¬ß10 + ¬ß4.6 + ¬ß11 item 9** state eventd chains `H(prev ‚Äñ frame-bytes)` ‚Äî over the *raw hcpbin header+body* the wire delimited ("the chain covers header and body together", line 1164; "`eventd` chains `H(prev ‚Äñ frame-bytes)`", line 403).
- **`eventd/core.py` (FROZEN) + `eventd/statedep.py` note** state eventd chains `sha256(prev_hex ‚Äñ canonical_json(payload))` over a **JSON payload**, and that "eventd payload bodies are JSON; the `.hcplang` frames in RFC 0004 ¬ß2 are the wire form, this is the logged form" (`statedep.py` lines 19‚Äì24).

**Working decision for S5 (oracle-faithful):** eventd's chain stays exactly `sha256(prev_hex ‚Äñ canonical_json(payload))` over a JSON **logged form**. The integration *projects* each admitted Litany frame into a JSON payload dict (`{"kind": ‚Ä¶, "v": ‚Ä¶, ‚Ä¶}`), precisely as `statedep.py`/`runtime.py` already do. We do **not** redefine eventd to hash raw `frame-bytes`.

Rationale: the task names "Python eventd is the byte oracle"; `core.py` is FROZEN and triple-cross-verified (`ralphcore`, `vakedz/cache.zig`); and the statedep/runtime modules already commit to the JSON-logged-form model. Changing the chain to hash frame-bytes would (a) break the frozen oracle, (b) invalidate every existing eventd entry/test, and (c) make the chain unverifiable by `python3 -m eventd verify`. The litany‚ÜíJSON projection is the bridge; the wire's `frame-bytes` is what is *transmitted*, the JSON logged form is what is *chained*.

**Open reconciliation item OR-1 (must be filed, see ¬ß8):** RFC 0003 ¬ß10 / ¬ß4.6 / ¬ß11.9 (chain over `frame-bytes`) vs `eventd/core.py` + `statedep.py` (chain over `canonical_json(payload)`). These are mutually exclusive byte contracts. One must be amended. S5 implements the oracle (JSON); the RFC text needs a corrective erratum or a deliberate decision to bump the eventd format (a breaking change that re-digests all chained frames ‚Äî explicitly out of S5 scope). Until reconciled, S5 carries an optional `frame_bytes_sha256` *field inside the JSON payload* (see ¬ß4.2) so the raw frame digest is still auditable without changing the chain function.

### 4.2 The projection: Litany frame ‚Üí eventd logged form

A frame admitted by the wire (RFC 0003 ¬ß10 admission rule: full `frame-bytes` strictly canonical per RFC 0002 ¬ß6.8) is projected to a JSON payload dict that the eventd chain then hashes. The projection is **total and deterministic** (same frame ‚áí same payload ‚áí same chain bytes).

**Header fields** (RFC 0003 ¬ß4.4.1): the wire-layer header `kind`/`corr`/`stream`/`seq`/`end` map into payload keys:

| Header field (¬ß4.4.1) | hcpbin wire form | Logged-form JSON key | JSON value |
|---|---|---|---|
| `kind` @0 | LEB128 u8 0‚Äì4 | `"kind_class"` | one of `"request"`,`"response"`,`"event"`,`"control"`,`"error"` (the ¬ß4.4.1 enum 0‚Äì4 decoded to its name; kept distinct from the payload's own `"kind"` discriminator) |
| `corr` @1 | 16 raw UUID bytes (RFC 4122 big-endian) | `"corr"` | canonical lower-case hyphenated UUID string |
| `stream` @2? | LEB128 u64 if present | `"stream"` | integer, **omitted if absent** |
| `seq` @3? | LEB128 u64 if present | `"seq"` | integer, **omitted if absent** (note: this is the *frame header* seq, distinct from the eventd entry `seq` which `make_entry` assigns) |
| `end` @4 | `0x01` if true, omitted if false | `"end"` | boolean, **omitted if false** (mirrors the ¬ß4.4.1 / RFC 0002 ¬ß6.2 default-omission rule) |

Omission must mirror hcpbin default-omission so that the JSON logged form carries exactly the information the canonical header did ‚Äî no `null`/`0` placeholders (RFC 0003 ¬ß4.4.1 "Optional fields are omitted, not encoded with null/zero markers").

**Body fields:** decoded from the negotiated schema (RFC 0003 ¬ß4.4 / ¬ß5.4 schema-digest negotiation) into their natural JSON scalars. The mapping is the RFC 0002 ¬ß7.1 codegen table read in reverse:
- `u8‚Ä¶u64`, `i8‚Ä¶i64`, `enum` ‚Üí JSON integer (statedep/runtime already ban floats in these payloads, `statedep.py` `_require_int`).
- `bool` ‚Üí JSON boolean.
- `string` ‚Üí JSON string (NFC per RFC 0002 ¬ß6.4 was already applied at encode time; the logged form preserves it as written ‚Äî eventd does **no** further normalization, `core.py` line 32).
- `bytes`, `hash`, `uuid` ‚Üí JSON string in a fixed textual form (base64-std-no-pad for `bytes`; `"<algo-hex>:<digest-hex>"` for `hash` per RFC 0002 ¬ß6.5 registry; canonical UUID string for `uuid`). The textual form is part of OR-2 (see ¬ß8) and MUST be fixed before implementation so the bytes are deterministic.
- `timestamp` ‚Üí JSON string (RFC 3339, UTC, fixed sub-second precision) ‚Äî fixed under OR-2.
- `list` ‚Üí JSON array (order preserved, RFC 0002 ¬ß6.6); `map` ‚Üí JSON object (keys already sorted by canonical key bytes on the wire; the JSON encoder re-sorts by code point per `core.py`, so the two sorts must be reconciled ‚Äî captured in OR-2).
- nested `record`/`union` ‚Üí nested JSON object (`union` carries an arm-tag field).

**Schema versioning:** every projected payload carries `"v": <schema version int>` (the `@since`/version discipline, RFC 0002 ¬ß8) exactly as `statedep.py`/`runtime.py` set `"v": STATEDEP_V` / `RUNTIME_V`. The wire's negotiated schema digest (RFC 0003 ¬ß5.4) is recorded once per connection in the `connection_open` lifecycle entry (below), not per frame.

**OR-1 bridge field:** the projected payload MAY include `"frame_bytes_sha256": "<hex>"` ‚Äî `sha256` of the exact canonical `frame-bytes` the wire delimited (¬ß4.1). This keeps the raw-frame digest auditable inside the JSON-hashed chain without changing the chain function. It is part of the logged form (so it *is* covered by the entry hash); whether it is required is gated on OR-1.

### 4.3 Redaction (`@redact`, RFC 0002 ¬ß4.4, normative)

A body field declared `@redact` MUST NOT have its plaintext value placed in the logged form. The projection replaces it with a fixed sentinel object `{"redacted": true, "len": <n>}` (length of the redacted value's canonical bytes, so size is auditable without content). This is applied **before** `canonical_json`, so the redaction is part of the hashed bytes ‚Äî a downstream `litanydump`/`eventd state` never sees the plaintext, matching the ¬ß4.4 requirement that `@redact` fields are redacted "in `eventd` projections." A regression test (see ¬ß6) must prove plaintext never appears in the on-disk `.jsonl`.

### 4.4 Connection-lifecycle entries (RFC 0003 ¬ß10)

Beyond frames, the bridge logs the connection shape. These are eventd payloads with their own `kind` discriminator (new `KIND_*` in a `litany_wire` logged-form module mirroring statedep/runtime):

```
connection_open   {kind, v, conn, peer, wire_version, max_frame, schema_digests:[...]}
connection_close  {kind, v, conn, reason: "bye"|"goaway"|"half_close"}
transport_fault   {kind, v, conn, fault, detail}     # ¬ß9.1 abortive close / framing violation
```

A **framing violation** (RFC 0003 ¬ß4.5: non-minimal/overlong `length`, `length` > `max_frame`, `length`==0, un-parseable header, mid-frame EOF) is recorded *only* as a `transport_fault` connection-end entry and the connection is aborted ‚Äî the offending bytes are never admitted as a frame (RFC 0003 ¬ß10, lines 1166‚Äì1169). A header-clean-but-non-canonical *body* is a frame-level `error` (¬ß9.2): its `frame-bytes` are excluded from the chain, and the bridge logs a `frame_rejected {kind, v, conn, corr, reason}` entry rather than the frame itself.

### 4.5 Write path (single-writer discipline, mirror `eventd/log.py`)

The Rust `eventd_writer.rs` reproduces the `EventLog` discipline exactly (`eventd/log.py` lines 149‚Äì214):
1. **Open as writer:** take the exclusive `flock(LOCK_EX | LOCK_NB)` **first**, then load+verify the existing chain under the lock (the Codex P1 ordering fix, `log.py` lines 161‚Äì173). A second writer ‚áí refuse with the WriterLockError-equivalent (exit-code 5 contract).
2. **Boot verify:** `verify_chain` the loaded entries; a broken chain is a hard error (TamperError-equivalent, exit-code 4) ‚Äî the writer is closed without ever holding the lock on a refused log (`log.py` lines 182‚Äì184).
3. **Append:** `entry = make_entry(tail_hash, len, payload)`; write `entry_line(entry)` (`json.dumps(entry, separators=(",",":"), sort_keys=True, ensure_ascii=False, allow_nan=False) + "\n"`); `flush`; `fsync` before returning (`log.py` lines 205‚Äì214). The `flush+fsync` per append is the durability contract (eventd design "Daemon shape", spec lines 52‚Äì62).
4. **Readers** map the log read-only and fold (`DependencyIndex.from_entries`, `RuntimeState.fold`) ‚Äî no change; the existing Python readers consume the Rust-written log unchanged, which is the whole point of byte parity.

The single-writer owner in the real runtime is `agent-supervisord` (eventd design line 55); in S5 the litany crate is the writer when it is the process appending. Multi-writer (per-fiber sub-logs) stays the eventd design's open question.

### 4.6 Fold integration (no new fold)

Because projected `DependencyRegistration`/`ConsumerCheckpoint`/`RewindEvent`/`consumer_evicted` payloads are byte-identical to what `statedep.py` constructors emit, they fold into the existing `DependencyIndex` with **zero changes**: GC floor (¬ß4), cold-start verify (¬ß6), eviction liveness (¬ß4.2) all work as-is. S5's job is to make the *wire* the producer of those payloads (replacing hand-written `python3 -m eventd append`), and to enforce RFC 0004 ¬ß3.1 (registration logged before consumption) structurally at the wire boundary.

## 5. Worked projection example (grounding)

A RFC 0004 ¬ß2 `DependencyRegistration` control frame on the wire:

- Header (RFC 0003 ¬ß4.4.1), single-shot control, no stream: `kind=3 (control)`, `corr=<uuid>`. Frame-bytes = `03 01 <16 uuid bytes> <body...>`.
- Body (RFC 0002 ¬ß6.2, ascending tags): `consumer` uuid @1, `producer` uuid @2, `consumer_step` u64 @3, `producer_step` u64 @4, `producer_step_hash` hash @5, `topology_epoch` u64 @6.

Projects to the eventd logged form (identical to `statedep.dependency_registration(...)`, `statedep.py` lines 64‚Äì73, with the header keys added):

```json
{"consumer":"<uuid>","consumer_step":4,"corr":"<uuid>","kind":"dependency_registration","kind_class":"control","producer":"<uuid>","producer_step":7,"producer_step_hash":"01:<64hex>","topology_epoch":2,"v":1}
```

(keys shown unsorted for readability; `canonical_json` sorts them by code point before hashing). The chain entry is then `{"seq":N,"prev":"<64hex>","payload":<above>,"hash":sha256("<prev>"+canonical_json(payload))}`, one LF-terminated line. `python3 -m eventd verify` over the resulting file MUST exit 0, and `python3 -m eventd coldstart <consumer>` MUST behave identically to a log written by the Python constructor.

## 8. Open reconciliation items (must be filed as issues before implementation)

- **OR-1 ‚Äî chain-byte contradiction.** RFC 0003 ¬ß10 / ¬ß4.6 / ¬ß11 item 9 (chain over `frame-bytes`) vs `eventd/core.py` FROZEN + `statedep.py` note (chain over `canonical_json(payload)`).
