## 1. Objective

Take the Litany wire stack from "passes the happy-path compliance tests" (S2–S5)
and "has a perf baseline" (S6) to **robust under adversarial and high-load
input**, with the wire's two highest-stakes correctness invariants demonstrably
held:

1. **Canonical framing is unforgeable** — every malformed / non-canonical /
   overlong / oversized input is rejected exactly as RFC 0003 §4.5 + §9 + RFC 0002
   §6.8 mandate, and a framing violation aborts **exactly one** connection without
   mid-stream resync (RFC 0003 §4.5, §9.1, §11 item 2).
2. **The connection and control plane never deadlock** — a `credit`/`ping`/
   `pause`/`resume`/`goaway` control frame is always emittable and receivable even
   when every data stream is at zero credit (RFC 0003 §7.2, §8.3(1), §8.3(3),
   §11 items 5–7).

Plus DoS-resistance under load (RFC 0003 §14.4): `max_frame` bounds per-frame
allocation, per-stream credit bounds per-stream buffering, liveness bounds
idle-peer cost — all three must survive a stress harness without unbounded memory
growth or wedging.

Success is **not** "no crashes." Success is: a differential/round-trip oracle
proves canonicality (RFC 0002 §6.8 fixed point), every documented rejection path
fires under fuzzing, and a concurrency stress harness exercises the
deadlock-freedom argument empirically.

## 2. Inputs / oracles

| Oracle | Source of truth | What it proves |
|--------|-----------------|----------------|
| **Round-trip fixed point** | RFC 0002 §6.8: `encode(decode(encode(v))) == encode(v)` | Canonicality of `hcpbin` bodies + frame records |
| **Golden vectors (positive)** | RFC 0002 §6.1.1, §6.3.1, §6.6.1, §6.7.1, §10; RFC 0003 §4.6 (framing), §5.8 (handshake) | Bytes match the spec exactly |
| **Rejection vectors (negative)** | RFC 0002 §6.1 (overlong/non-minimal varint), §6.4 (bool ≠ 0x00/0x01), §6.5 (known-algo hash len mismatch), §6.2 (out-of-order/dup tags); RFC 0003 §4.1–§4.3 (zero/overlong/oversize length), §4.4.1 (header tag rules), §9.2.1 (`error_kind` taxonomy) | Every malformed input fails, and fails in the documented category |
| **Decision-rule oracle** | RFC 0003 §9.3 | A given malformed input lands as transport-fault (abort) **xor** frame-level `error` (connection survives) — never the wrong bucket |
| **Existing adversarial-independence suite** | ``protocol/hcp/hcpbin/tests/golden.rs`` (`../../../protocol/hcp/hcpbin/tests/golden.rs`) — "written BLIND to the implementation" | S7 extends this discipline; fuzz targets drive the same public symbols (`decode_uvarint`, `decode_bool`, `decode_bytes`, `decode_hash`, `decode_i8..i64`) |
| **S6 perf baseline** | WP3-S6 deliverable (latency vs grpc/cbor; `protocol/hcp/litany/benches/`) | Regression comparand for criterion (relative) on M1 and the ≤10µs gate on `dev-cx53` |

No oracle invents bytes the RFC does not give. Where RFC 0003 Open Question 1
leaves the `hcp.wire` schema digest and the official conformance vectors "TBD
until the layout lands" (RFC 0003 §5.6, §11), S7 **must not invent a digest** — it
pins those tests behind the dependency in §7.

## 4. Algorithm / design

### 4.1 Differential / round-trip fixed-point harness (the canonicality oracle)

The single most important hardening artifact. RFC 0002 §6.8 states the conformance
test verbatim: `encode(decode(encode(v))) == encode(v)` MUST be a fixed point, and
two conforming implementations MUST agree byte-for-byte.

- **proptest generators** produce typed values for each `hcplang` type the stack
  supports at S7 (scalars → records → unions → list/map → frames). For each
  generated `v`: assert the fixed point. For union/hash with unknown arm/algo
  (RFC 0002 §6.7, §6.5) generate opaque-byte payloads and assert byte-for-byte
  round-trip preservation (RFC 0002 §7.2).
- **Map ordering** (RFC 0002 §6.6): generate maps with deliberately-scrambled
  insertion order; assert the encoding is sorted by canonical encoded-key order
  and is insertion-order-independent.
- **Default omission** (RFC 0002 §6.3, §6.3.1): generate all-default nested
  records and present-but-empty optional aggregates; assert the omit/emit choice
  matches the §6.3 table exactly (this is the subtle "two valid encodings" hole).

### 4.2 Fuzz targets (the rejection oracle)

cargo-fuzz / libFuzzer targets, one per decode entry point. Invariant for every
target: **never panic, never over-read, reject non-canonical input as a typed
`DecodeError`** (the crate's existing error type,
[`protocol/hcp/hcpbin/src/lib.rs`](../../../protocol/hcp/hcpbin/src/lib.rs)).

- `uvarint_strict` / `ivarint_width`: feed arbitrary bytes to `decode_uvarint`,
  `decode_i8..i64`; assert no overlong/non-minimal varint is ever accepted
  (RFC 0002 §6.1), and any value exceeding the declared width `k` is rejected.
- `bool_strict`: any byte ≠ `0x00`/`0x01` is rejected (RFC 0002 §6.4) — never
  coerced to `true`.
- `bytes_len` / `hash_selfdelim`: a length/`len` prefix larger than the remaining
  buffer is a clean error, never an over-read; a **known** `algo-id` with the
  wrong `len` is rejected, an **unknown** `algo-id` is skipped by `len` and the
  raw digest preserved (RFC 0002 §6.5, §6.5.1).
- `roundtrip_fixedpoint`: if `decode(b)` succeeds, `encode(decode(b))` must equal
  the canonical form, and decoding *that* must succeed and be stable (RFC 0002 §6.8).
- (litany) `frame_record_decode` / `frame_header_decode` / `handshake_preamble`:
  drive the frame-record length prefix (RFC 0003 §4.1–§4.3), the `@0..@4` header
  decoder invariant (RFC 0003 §4.4.1: `kind` in 0–4; `@4` valid only if `@2`
  present; out-of-order / duplicate `@4` are framing violations), and the magic +
  HELLO preamble (RFC 0003 §5.2, §5.7).

Each fuzz target seeds its corpus from the **positive** golden vectors (RFC 0002
§6.1.1/§6.3.1/§6.6.1/§6.7.1/§10, RFC 0003 §4.6/§5.8) so the fuzzer starts from
valid frames and mutates outward.

### 4.3 Negative golden table (the decision-rule oracle)

`rejection_vectors.rs` (codec) and `decision_rule.rs` (frame) are explicit tables:
each row is `(hex bytes, expected outcome, RFC §)`. Outcome is one of:

- **codec**: `Err(DecodeError::...)` of the right kind;
- **frame, transport fault** (RFC 0003 §9.1): abort connection, no resync, fault
  recorded — for framing violations (overlong/zero/oversize length §4.1–§4.3,
  header decode failure §4.5, bad magic §5.2, credit overrun §8.4);
- **frame, frame-level `error`** (RFC 0003 §9.2): connection survives, `error`
  echoes `corr` with the right `error_kind` (RFC 0003 §9.2.1 taxonomy:
  `malformed_frame`, `schema_digest_mismatch`, `seq_gap`, `chapter_closed`,
  `stream_reset`, `invalid_correlation`, ...) — for header-clean-but-non-canonical
  bodies and schema/authority failures.

This table operationalizes the RFC 0003 §9.3 decision rule ("is the byte-stream's
frame alignment still trustworthy?") and is the regression guard against the
classic wire bug of conflating the two categories. **Non-canonical bodies are
never admitted to the `eventd` chain** (RFC 0003 §9.2, §10, §11 item 9) — the
table asserts the offending `frame-bytes` is rejected before any chain append.

### 4.4 Concurrency / deadlock-freedom harness

`deadlock_freedom.rs` runs two `litany` endpoints over an in-process tokio duplex
(or a UDS loopback — RFC 0003 §13.2) and reproduces the RFC 0003 §8.3 argument
empirically:

1. Open a stream, drain its credit to zero (RFC 0003 §8.1–§8.2; default initial
   credit 16 frames, §8.2).
2. With the producer blocked on zero credit, the consumer side sends a `ping`
   (RFC 0003 §7.1) and a `credit` grant; assert both are **received and acted on**
   within a bound (RFC 0003 §8.3(1)(b), §11 item 5) — the receiver never stops
   reading (§8.3(3)).
3. From the blocked producer, send a `ping`/`credit`/`close` control frame; assert
   it is **emitted** even while a data write is buffered and unsendable
   (RFC 0003 §7.2 sender-side obligation, §11 item 6).
4. Assert per-stream buffered octets stay ≤ `granted × max_frame` (RFC 0003
   §8.3(4)) and that a second stream keeps draining while the first is paused
   (§8.3(2), §8.4.1 worked example).
5. Liveness independence: with a stream paused, `ping`/`pong` still flow and the
   idle timer is not suspended (RFC 0003 §7.5).

### 4.5 Stress binaries (DoS-resistance under load)

Long-running binaries under `litany/stress/` (not `cargo test` units):

- `flood`: N connections × M streams of chunked frames; assert total resident
  memory stays bounded by `Σ granted × max_frame` + fixed overhead (RFC 0003
  §8.3(4), §14.4). Inject a hostile peer that (a) claims `length > max_frame`
  (assert abort without reading the body, RFC 0003 §4.3) and (b) sends a data
  frame at zero credit (assert `credit-overrun` teardown, RFC 0003 §8.4).
- `credit_stall`: hold one stream permanently un-replenished (a *permitted*
  deliberate-backpressure outcome, RFC 0003 §8, §11 item 7) and assert the
  connection still makes progress and is observably-stalled via liveness, not
  wedged.
- `soak`: sustained encode/decode/round-trip throughput for a fixed wall-clock
  window; sample RSS to detect leaks/unbounded growth (no allocation on the hot
  path is the kickoff target, line 54).

### 4.6 Performance gate (arch-sensitive — see §5)

The kickoff success criterion is **≤10µs per frame encode/decode on `dev-cx53`
(8-core, no-alloc hot path)** (kickoff line 54). This is an **absolute** number on
a **specific Linux x86_64 host**; it is **not portable to M1 (aarch64-darwin)** —
different microarchitecture, memory subsystem, and allocator behaviour make an
absolute µs threshold meaningless cross-arch. On M1, criterion is used **only for
relative regression** against the S6 baseline checked into
`protocol/hcp/litany/benches/`; the absolute gate runs on `dev-cx53`.

## 5. Test plan — M1-local vs `dev-cx53`/Linux

Toolchain verified on this host (2026-06-14): `cargo 1.95.0`, `cargo-fuzz 0.13.1`,
`arch = arm64`. `dev-cx53` is OFF-LIMITS for the current 6h autoresearch window
(orchestrator note); everything below that is marked M1-local must pass without it.

### 5.1 Runs fully on M1 (aarch64-darwin) — correctness & robustness

| Check | Command | Why M1 is sufficient |
|-------|---------|----------------------|
| Round-trip fixed point (§4.1) | `cargo test -p hcpbin` + `cargo test -p litany` | Pure logic; arch-independent |
| Property tests (proptest, §4.1) | `cargo test -p hcpbin --test fixed_point` | Deterministic given a seed; arch-independent |
| Negative golden / rejection tables (§4.3) | `cargo test -p hcpbin --test rejection_vectors` / `-p litany --test decision_rule framing_rejection` | Byte-exact assertions from RFC §§ |
| Fuzz **smoke** (§4.2) | `cargo +nightly fuzz run uvarint_strict -- -runs=100000` (and each target) | libFuzzer builds on aarch64-darwin (nightly + sanitizer); short campaigns find shallow panics/over-reads locally |
| Deadlock-freedom harness (§4.4) | `cargo test -p litany --test deadlock_freedom` | tokio in-process / UDS loopback — no Linux-specific syscalls |
| Relative perf regression (§4.6) | `cargo bench -p litany` (criterion vs S6 baseline) | Relative deltas are meaningful on any single host |
| Zig codegen target (RFC 0002 §7) | `zig build -Dtarget=x86_64-linux` (compile-only) | Compiles on M1; namespace syscalls do **not** run here |

### 5.2 Requires `dev-cx53` (Linux x86_64) — authoritative perf & integration

| Check | Why it cannot gate on M1 |
|-------|--------------------------|
| **≤10µs/frame absolute gate** (kickoff line 54) | Absolute µs threshold is arch-specific; aarch64-darwin numbers are not comparable to the x86_64 target |
| **Long fuzz campaign** (hours, large corpus) | Authoritative coverage/throughput belongs on the canonical build host; M1 runs smoke only |
| **Soak / sustained-throughput memory stability** (§4.5 `soak`) | Long-run RSS behaviour depends on the production allocator + kernel on the target host |
| **vsock stress** (RFC 0003 §13.3) | `AF_VSOCK` is Linux-only; no macOS equivalent |
| **Namespace-isolated daemon integration** (Zig enforcement daemons, RFC 0002 §7) | Linux namespaces/seccomp run only on Linux |
| **eventd end-to-end chaining under load** (RFC 0003 §10) | Exercises the runtime `eventd` daemon ([`docs/superpowers/specs/2026-06-12-eventd-design.md`](2026-06-12-eventd-design.md)); integration host is Linux |

## 6. Acceptance criteria

S7 is done when **all** hold:

1. **Round-trip fixed point** (RFC 0002 §6.8) passes for every supported type via
   proptest with ≥ a fixed case budget, plus all positive golden vectors
   (RFC 0002 §6.1.1/§6.3.1/§6.6.1/§6.7.1/§10, RFC 0003 §4.6/§5.8) — M1-local.
2. **Every fuzz target** runs a smoke campaign on M1 with **zero panics, zero
   over-reads, zero accepted non-canonical inputs**; the authoritative long
   campaign on `dev-cx53` finds no new crashers.
3. **Negative golden table** is complete: at least one vector per cited rejection
   rule (RFC 0002 §6.1/§6.2/§6.4/§6.5; RFC 0003 §4.1–§4.3/§4.4.1), each asserting
   the right outcome **and** the right §9.3 category (fault vs frame-level
   `error`) — M1-local.
4. **Deadlock-freedom harness** passes: control frame emittable + receivable under
   full data-credit stall, per-stream memory bound ≤ `granted × max_frame`, second
   stream drains while first is paused, liveness independent of pause
   (RFC 0003 §7.2, §7.5, §8.3, §11 items 5–7) — M1-local.
5. **DoS-resistance**: `flood` keeps memory bounded and tears down hostile peers
   (`length > max_frame` → abort §4.3; zero-credit data → `credit-overrun` §8.4);
   `soak` shows no leak — perf/long-run legs on `dev-cx53`.
6. **Perf**: relative criterion regression vs S6 baseline within tolerance on M1;
   **≤10µs/frame absolute gate on `dev-cx53`** (kickoff line 54).
7. **Decision-rule separation** (RFC 0003 §9.3) holds for the whole negative
   corpus; non-canonical bodies are proven **never** chained to `eventd`
   (RFC 0003 §9.2, §10, §11 item 9) — M1-local.
8. Every new test file cites the RFC § it enforces, matching the
   adversarial-independence convention of
   ``protocol/hcp/hcpbin/tests/golden.rs`` (`../../../protocol/hcp/hcpbin/tests/golden.rs`).

### 7.1 Blocking risks / open questions to track

- **RFC 0003 Open Question 1** (RFC 0003 §5.6, §11): the `hcp.wire` schema digest
  is "value TBD until the layout lands" and the **official conformance vectors are
  a companion artifact that does not exist yet**. Tests that pin the control-schema
  digest or consume official handshake/credit-stall vectors are **blocked** on this
  landing. S7 **must not invent a digest**; until then, deadlock/handshake harnesses
  use locally-constructed `hcp.wire` frames and are upgraded to the pinned vectors
  when OQ1 closes.
- **`error_kind` normative set** (RFC 0003 §9.2.1): the taxonomy is "provisional;
  the normative set is in the `hcp.wire` schema" (RFC 0003 Appendix A) — also gated
  by OQ1. Assert the *category* (fault vs frame-level `error`) firmly; treat exact
  `error_kind` strings as provisional until OQ1.
- **Header tag model is contested** between RFC 0002 §4.2 ("@0 reserved, author
  tags @1+") and RFC 0003 §4.4.1 (header tags @0–@4, body begins at tags > @4) —
  a latent RFC-level incoherence (cf. [`tools/rfc-incoherence-hunter`](../../../tools/rfc-incoherence-hunter/)).
  S7 header tests follow **RFC 0003 §4.4.1**, since the header is a wire concern
  and 0003 is authoritative there; resolving the contradiction is not S7's job.
- **Capability defaults** (RFC 0003 §5.5, §8.2): initial credit (default 16),
  `max_frame` floor (64 KiB) / default (1 MiB), idle 15s / liveness 30s (RFC 0003
  §7.3) are the stress-harness constants; if S2–S4 negotiate differently, update
  the harness constants accordingly.
