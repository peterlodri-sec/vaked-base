## 1. Recursive framing

Slice 1 reverse-engineers the self-hosted inference runtime (`llama-cli`,
llama.cpp) running on `dev-cx53`. The decompiler-LLM is llm4decompile-6.7B,
itself an LLM, served on the same machine. This is deliberately recursive: one
LLM dissects the binary that runs another LLM. Because llama.cpp is FOSS (MIT),
the ground-truth C source is available at the exact build commit — producing an
objective fidelity metric that an arbitrary stripped binary would not give us.
This recursive, ground-truth-anchored design is what makes slice 1 a meaningful
testbed rather than a qualitative demonstration.

## 2. Data flow

```
llama-cli (nix-store binary)
       │
       ├─── ghidra_frontend.py ──────────────────────────────────────────────────┐
       │    PyGhidra (pyghidra_decompile.py, CPython 3)                          │
       │    out: {func_name: pseudo_c}                                           │
       │         │                                                               │
       │         └─── llm_refine.py ──────────────────────────────────────────┐ │
       │              llama-server POST /completion (temp=0)                   │ │
       │              out: {func_name: refined_c}                              │ │
       │                                                                       │ │
       ├─── dynamic_frida.py ──────────────────────────────────────────────────┤ │
       │    frida_driver.py (frida-python, findGlobalExportByName, stdio=pipe) │ │
       │    out: {func_name: {calls, timing_ms}}                               │ │
       │                                                                       │ │
       └─── watcher_client.py → /run/oracle-watcher.sock                      │ │
            │   (unix socket; root service)                                    │ │
            └─── watcher_daemon.py + bpftrace (PID-scoped)                    │ │
                 out: {syscalls, mmaps: [*.gguf paths], files}                 │ │
                                                                               ▼ ▼
                                        fidelity.py (vs llama.cpp ground-truth source)
                                        score(refined_c, ground_truth) → float [0,1]
                                               │
                                        schema.py → finding record (see §4)
                                               │
                                        oracle.py persist_finding → findings/<sha256>.json
                                               │
                                        bridge.py → observed_effects + transition_xref
                                               │
                                        ledger.py (hash-chained JSONL; ralphcore primitives)
```

All three evidence producers (ghidra/llm, frida, ebpf watcher) are
**independently nullable** — a finding is valid with any subset present. Failures
are recorded in the ledger as decision entries; the loop continues without
crashing.

## 3. Finding record schema

Defined in `tools/oracle/schema.py`. The canonical shape:

```jsonc
{
  "kind": "oracle_finding",
  "v": 1,
  "target": {
    "path":       "/nix/store/…/bin/llama-cli",
    "sha256":     "<hex64>",
    "source_ref": "<llama.cpp commit sha>"
  },
  "decompiler": {
    "model":        "llm4decompile-6.7b-v2",
    "model_sha256": "<hex64>",
    "temperature":  0
  },
  "functions": [
    {
      "name":         "llama_decode",
      "addr":         "0x…",
      "pseudo_c_sha": "<sha256 of pseudo_c>",
      "refined_c":    "int llama_decode(…) { … }",
      "fidelity": {
        "score":  0.87,
        "method": "normalized-token-similarity"
      },
      "dynamic": {
        "frida": { "calls": 12, "timing_ms": 3.14 },   // null if frida unavailable
        "ebpf":  { "syscalls": {"mmap": 2, "read": 48},
                   "mmaps":    ["/nix/store/…/llm4decompile-6.7b-v2.gguf"],
                   "files":    ["/nix/store/…"] }       // null if watcher unreachable
      }
    }
  ],
  "observed_effects": { "writes": [], "deletes": [] },
  "transition_xref": null,        // null until grounded; `oracle ground` populates it (slice 2)
  "confidence": 0.87
}
```

`validate_finding` in `schema.py` checks `kind`, `v`, required top-level
keys (`target`, `decompiler`, `functions`, `confidence`), and that `confidence`
is a float in [0, 1].

Chain fields (`prev_hash`, `this_hash`) are added by the ledger at append
time, not stored in the finding payload directly — the finding is the ledger
entry's `payload`.

## 4. Ralph-loop orchestration

`loop.py` runs a ralph-style decision loop over the target functions, reusing
`tools/ralph/ralphcore.py` chain primitives (never modified — import only).

**LoopState** (`policy.py`) carries the mutable view each tick needs:

```
LoopState(
  functions:   list[str],            # all target function names
  results:     dict[str, dict],      # accumulated per-function evidence
  iters:       int,                  # ticks consumed so far
  budget_iters: int,                 # hard cap (default 50)
)
```

**Policy** (`policy.next_action`) returns one of:

| Action | Condition |
|--------|-----------|
| `{"action": "decompile", "fn": name}` | Function not yet in `results` |
| `{"action": "refine", "fn": name}` | Function in `results` with `fidelity < THRESHOLD` and `refine_passes < MAX_REFINE` |
| `{"action": "finalize"}` | All functions at/above threshold, or budget exhausted, or `MAX_REFINE` reached for every pending function |

Constants: `FIDELITY_THRESHOLD = 0.75`, `MAX_REFINE = 2` (in `policy.py`).

**Control.** If `control_path` is set, `loop.py` reads a JSON file at the start
of each tick. `{"stop": true}` in that file halts the loop immediately and
appends a `control_stop` ledger entry — mirroring ralph's live pause/stop
mechanism. This lets a running RE session be gracefully interrupted without
corrupting the ledger.

**Producers are injected callables**, making the loop testable with fakes and
decoupled from the real tools:

```
decompile(fn)          -> (pseudo_c, refined_c, fidelity)
refine(fn, prev)       -> (refined_c, fidelity)
dynamic(fn)            -> (frida_dict | None, ebpf_dict | None)
```

In production, `oracle.py::cmd_run` wires in the real implementations
(`ghidra_frontend.run_ghidra`, `llm_refine.refine`, `dynamic_frida.run_frida` +
`watcher_client.query_watcher`).

## 5. eBPF watcher

The eBPF evidence path has two halves separated by a privilege boundary:

**Root side** — `oracle-ebpf-watcher.nix` + `watcher_daemon.py`:
- NixOS systemd service, `Type=simple`, `Restart=on-failure`.
- `RuntimeDirectory=oracle-watcher`; socket at `/run/oracle-watcher.sock`
  (`srw-rw----  root  oracle-watcher`).
- On each request: validates the `pid` field (must be > 0), calls
  `_run_bpftrace` which executes bpftrace with a PID-scoped program, parses
  the output with `parse_bpftrace`, and returns JSON over the socket.
- The bpftrace program traces two probe points for the target PID:
  `tracepoint:syscalls:sys_enter_*` (syscall histogram) and
  `tracepoint:syscalls:sys_enter_openat` (file-open trace, surfaces GGUF
  weight mmap paths). It runs for `duration_s` seconds then exits.
- `parse_bpftrace` strips the `tracepoint:syscalls:sys_enter_` prefix from
  syscall names so the output is a clean `{openat: 4, mmap: 2, …}` dict.

**Unprivileged side** — `watcher_client.py`:
- `query_watcher(sock_path, pid, duration_s)` opens the unix socket,
  serializes `{"pid": pid, "duration_s": duration_s}` as a JSON request, and
  deserializes the response.
- `decode_response` raises `RuntimeError` if `ok` is false, so failures
  propagate cleanly to `loop.py`'s nullable evidence path.
- `revdev` never executes bpftrace directly and never acquires `CAP_BPF` or
  `CAP_PERFMON`. The socket is the entire attenuation surface.

## 6. Open items

These items are known unknowns that the acceptance run (Task 17) will resolve:

1. **Exact GGUF variant.** The Taskfile and docs reference
   `llm4decompile-6.7b-v2.gguf` as the expected filename. The specific
   quantization level (Q4_K_M vs Q5_K_M vs Q8_0) and Hugging Face source repo
   (`LLM4Binary/llm4decompile-6.7b-v2-GGUF` or a mirror) must be confirmed
   on dev-cx53 before `model:fetch` can be scripted completely.

2. **Prompt template wording.** `llm_refine.py` uses:
   ```
   PROMPT_PREFIX = "# This is the decompiled pseudo-code:\n"
   PROMPT_SUFFIX = "\n# What is the original source code?\n"
   ```
   This follows the llm4decompile decompile-refine convention as described in
   the model card. The exact wording must be confirmed against the chosen GGUF's
   model card at first real-run time (the card wording varies between
   `-6.7b-v1` and `-6.7b-v2`).

3. ~~**analyzeHeadless absolute path.**~~ **RESOLVED.** The Jython/analyzeHeadless
   path is superseded by PyGhidra (CPython 3). `ghidra_frontend.run_ghidra` now
   invokes `pyghidra_decompile.py` via the `~/oracle/pgvenv` venv; the Taskfile
   derives `GHIDRA_INSTALL_DIR`, `JAVA_HOME`, and `ORACLE_LIBSTDCXX_DIR` from the
   nix store dynamically (no pinned path needed). Set up the venv once with
   `task -d tools/oracle pyghidra:setup`. Validated on ghidra 12.0.4 / openjdk-21
   / pyghidra 3.0.2 on dev-cx53.

4. **Fidelity method upgrade.** The slice-1 scorer is Dice over C token
   multisets (comments stripped, identifiers/keywords/operators extracted with
   a regex). A tree-sitter C AST diff is a planned upgrade (noted in
   `fidelity.py` module docstring) for a later cycle.

5. **bpftrace program.** The program in `watcher_daemon._bpftrace_program`
   uses `tracepoint:syscalls:sys_enter_*` (wildcard) and
   `tracepoint:syscalls:sys_enter_openat`. The wildcard form requires a
   sufficiently recent bpftrace on `dev-cx53`; if the wildcard tracepoint is
   unsupported, fall back to enumerating specific syscalls. Verify at first
   watcher run.

6. **`Date.now()` resolution in `frida_driver.py`.** Frida's `Date.now()` returns
   millisecond-resolution wall-clock time. The driver multiplies by `1e6` to
   express as nanoseconds (`dur_ns`), and `dynamic_frida.parse_frida_trace`
   divides by `1e6` to recover milliseconds. For very fast functions this
   quantizes to 1 ms; sub-ms timing requires Frida's `Process.hrtime()` (a
   later upgrade).
