## 3. Publication-ready section

### From release governance to artifact governance

The previous section established *when* a Vaked system is permitted to ship: the gates, approvals, and sign-offs that govern a release as a product decision. That governance is necessary but it operates one level above the things being shipped. A release is not a monolith; it is a closure over a set of versioned artifacts - prompts, container images, API schemas, documentation, semantic schemas, MLIR specifications, and the write-ahead log format itself. Release governance decides *whether* the bundle is allowed out. **Artifact governance decides whether every component of that bundle can be named, pinned, attested, and - when it goes wrong - rolled back to a known-good state.** This section is the bridge: it takes the release-level "yes" and grounds it in the integrity of the individual artifacts.

### The prompt is an artifact, and it is the hardest one

The instinct to treat a prompt as configuration text rather than as a governed software artifact is the central failure mode this section corrects. A prompt is source: it is the program text that, together with a model and a decoding policy, produces behavior. It deserves the same discipline as any other source artifact - a version, an identity, an owner, a changelog, and a rollback path.

But the prompt is harder than ordinary source, and the difference is the intellectual core of this bridge. In deterministic software, a version pin *fully determines* behavior: pin the commit, pin the inputs, and the output is fixed, so a reviewer can verify a change is safe by reading the diff. In an LLM system that property does not hold. Pinning the prompt version is *necessary but insufficient*: the model can drift underneath you, and sampling (temperature, top-p, seed) makes a single prompt a distribution of behaviors rather than a function. **You cannot prove a prompt change safe by reading it.** Regressions are discovered empirically - in evaluation, in canary, or in production - not statically in review.

That asymmetry is *why* fast, governed rollback matters more for prompts than for deterministic code, and why the rollback unit is not the prompt file alone. The integrity unit is a pinned tuple:

> **(prompt version, model digest, decode params + seed, eval baseline).**

Rolling back a prompt regression means restoring that whole tuple, not editing text back. This reframes two terms that recur loosely in prompt-governance writing and should not survive into a serious specification:

- [REFRAME] **"Stable outputs."** In a non-deterministic system you cannot promise stable outputs, and the phrase is unfalsifiable. What is governable is the *integrity unit above*, with **bounded, measured variance** against the eval baseline - a falsifiable claim ("p95 eval score stays within X of baseline at this seed policy"), not a marketing one.
- [REFRAME] **"Canonical deployment tracks."** Replace with **digest-pinned release channels backed by immutable provenance** - a channel is a named pointer; what makes it trustworthy is that each entry resolves to a content-addressed, attested artifact, not the word "canonical."

### The variance contract

"Bounded, measured variance" is only governance if it is falsifiable, which means it must name a metric, a statistic, a threshold shape, a seed policy, and a pass/fail rule. Vaked specifies the *shape* of that contract and leaves the numeric tolerance to per-deployment policy - there is no universal magic constant, because the acceptable variance for a code-generation prompt differs from that of a classification prompt. The contract is the following definition list.

- **Metric.** A configurable eval suite that produces one scalar score per run. The default is `tool-call-success-rate` (the fraction of attempted tool calls that succeed), which aligns the gate with the prompt-regression failure mode in Scenario A; deployments may substitute a task-appropriate scalar (pass-rate, rubric score) so long as the suite is fixed for the comparison.
- **Statistic and threshold.** Sample N runs over a fixed seed set, yielding N scores, and take p95 across them. The threshold is *directional against the recorded baseline*: the candidate's p95 must be no worse than the baseline's p95 by more than a tolerance delta (absolute, or relative as X%). The comparison is one-sided on purpose - an improvement is never a regression - and delta is set by deployment policy.
- **Seed policy.** The gate run uses the *same fixed seed set* as the recorded baseline, so the two p95 figures are computed over identical sampling and the comparison is reproducible. This is distinct from the production sampling policy, which is unconstrained (live traffic uses whatever decode params and seeds the deployment chooses); the fixed seed set exists only to make the gate a controlled experiment against the baseline.
- **Pass/fail rule.** The integrity tuple passes the gate iff p95 of the eval metric over the fixed seed set is within the threshold of the baseline. Otherwise the change is blocked before release, or - if already shipped - rolled back to the prior tuple.

Because the unit that passes or fails is the integrity tuple *(prompt version, model digest, decode params + seed, eval baseline)*, the rollback unit is that whole tuple, not the prompt text alone: a regression restores the last tuple whose p95 was within threshold. The tolerance is a policy parameter, set once per deployment and recorded alongside the baseline, so "within bounded variance" resolves to a concrete, re-runnable check rather than a promise.

### Mutable tags versus immutable digests

The container layer makes the mutable/immutable distinction concrete and standards-anchored, and it generalizes to every artifact class. A tag such as `:latest` is a **mutable** name: re-pulling it may return different bytes over time [STD, C2]. A digest reference, `image@sha256:…`, is **immutable**: the OCI image-spec defines the descriptor `digest` as a content identifier - a cryptographic hash enabling content-addressable storage - such that content from an insecure source can be re-verified by recomputing the hash, and content whose length or hash does not match SHOULD NOT be trusted [STD, C1]. Docker's own guidance is that pulling by digest "pins" an image so it is "always the same," whereas a tag may move [STD, C2].

The governance rule that falls out of this is a single sentence: **tags are for humans; digests are for deployments.** A release channel may be addressed by a friendly tag, but what is deployed, attested, and recorded must always be a digest. Vaked takes this further than convention. Per RFC 0007, every materialized image - NixOS closure, OCI image, or MirageOS unikernel - is content-addressed and carries a PQC-signed `provenance.json` (a *votive seal*); the *only* path to mutate a running system is a code change that re-derives and re-signs the image. Signatures default to ML-DSA (FIPS 204), hybrid with a classical primitive during migration [INTERNAL, C13]. "Image-as-code" is the digest discipline promoted from a recommendation to an invariant.

### Supply-chain integrity vocabulary

The container example is one instance of a general supply-chain integrity model with mature primary standards, which Vaked adopts rather than reinvents:

- **Provenance / attestations.** SLSA provenance is "verifiable information about software artifacts describing where, when and how something was produced," carried as an in-toto attestation with predicate type `https://slsa.dev/provenance/v1` (BuildDefinition + RunDetails) [STD, C3]. SLSA build levels L1-L3 grade the hardening of the build *platform* (L1: provenance exists; L2: hosted infrastructure with signed provenance; L3: isolated builds with signing keys protected from user-defined steps); at verification time the spec makes `builder.id` the intended sole determiner of the *claimed* level, because trust is anchored to the builder identity rather than to other provenance fields [STD, C4]. An in-toto attestation Statement binds a `subject` (artifact `name` + mandatory `digest`) to a `predicateType` URI - the generic frame under which provenance, SBOMs, and custom claims all travel [STD, C5].
- **SBOM.** SPDX is an open SBOM standard, recognized as ISO/IEC 5962:2021, with SPDX 3.0 adding Build, Security, and AI profiles [STD, C6]. CycloneDX is a peer standard - an OWASP project standardized as Ecma International ECMA-424 (TC54), spanning SBOM, SaaSBOM, HBOM, ML-BOM, CBOM, and VEX [STD, C10].
- **Signing + transparency.** Sigstore provides keyless signing (Fulcio short-lived certificates bound to OIDC identity) with every signing event recorded to Rekor, an immutable append-only transparency log; cosign is the client [STD, C7].
- **Policy baseline.** NIST SP 800-218 (SSDF v1.1) was developed in response to EO 14028 [STD, C8]; it prescribes specific artifact integrity and provenance controls - PS.2 "Provide a Mechanism for Verifying Software Release Integrity," and PS.3 "Archive and Protect Each Software Release," whose tasks require archiving "integrity verification information, provenance data" per release (PS.3.1) and to "Collect, safeguard, maintain, and share provenance data for all components of each software release (e.g., in a software bill of materials [SBOM])" (PS.3.2) [STD, C9].

Vaked's `eventd` log - an append-only, hash-chained, per-runtime event log - is the local analogue of a transparency log [INTERNAL, C12]: the votive seal of an image is recorded to it, so "which image is this runtime actually running, and who signed it?" stays answerable forever.

### Documentation is a versioned artifact too

Documentation versioning is not a courtesy; it is part of the integrity envelope. It expands into four governed sub-classes:

- **Release branches / migration docs** - versioned with the release, telling an operator how to move *between* pinned versions (the rollback and roll-forward runbooks).
- **Operator docs** - versioned with the runtime daemons they describe (OTP supervision plane, Zig enforcement daemons), so the surface matches the binary.
- **Compliance evidence** - the SBOMs, attestations, and provenance records, retained per release and addressable by the same digests as the artifacts they describe.

Documentation drift is a governance defect of the same kind as an unpinned tag: a doc that describes a version other than the one deployed is a mutable pointer masquerading as a fact.

## 4. Release-governance checklist for Vaked V1.0

Concrete and tickable. A release is blocked if any unchecked.

- [ ] Every deployed artifact is referenced by an **immutable identity** (digest / content hash / `StepHash`), never by a mutable tag. [C1,C2]
- [ ] Each container image has a **PQC-signed `provenance.json`** (votive seal) recorded to `eventd`. [C13]
- [ ] SLSA provenance is generated for the build, predicate type `slsa.dev/provenance/v1`, with a recorded `builder.id`. [C3,C4]
- [ ] An **SBOM (SPDX)** is produced and attached for the release. [C6]
- [ ] Signatures are recorded to a transparency log (`eventd` locally; Sigstore/Rekor for externally distributed artifacts). [C7]
- [ ] Every prompt change ships the full **integrity tuple** *(prompt ver, model digest, decode+seed, eval baseline)* and an eval run against baseline within bounded variance. [C11,C14]
- [ ] Each API change declares its **SemVer impact** and, if breaking, a deprecation window and migration doc.
- [ ] Each **MLIR/topology change** bumps the topology epoch; all dependency artifacts carry it. [C12]
- [ ] Any **WAL-format change** has a forward/backward read path and a tested rewind procedure. [C12]
- [ ] **Docs match the deployed version**: operator docs, migration docs, and compliance evidence are pinned to the release tag.
- [ ] A **rollback runbook** exists per artifact class, naming the rollback unit and the last known-good identity.

## 5. Rollback scenario examples

**Scenario A - Prompt regression.** A prompt update raises tool-call error rate in canary. Because the prompt is governed as a tuple, recovery is not "edit the text back" - it is restoring the last tuple *(prompt v1.4.2, model digest D, decode policy P, eval baseline B)* whose eval score was within bounded variance. The bad tuple's eval record stays in the log as evidence of *why* it was reverted, and the rollback is itself an `eventd` entry. The fix is empirical: re-evaluate, do not re-read. [C11,C14]

**Scenario B - Bad container digest.** A new image digest passes build but fails attestation verification (votive seal does not validate, or SLSA `builder.id` is unexpected). Deployment refuses the digest - there is no mutable `:latest` to fall through to, so the running system stays on its current content-addressed closure. Recovery is repointing the release channel to the prior verified `image@sha256:…`. The mutable-tag failure mode (silently pulling drifted bytes) is structurally impossible. [C1,C2,C13]

**Scenario C - API schema break.** A minor release accidentally removes a required field. Consumers fail contract tests, which are the attested evidence. Because each API major/minor is an immutable per-version contract, rollback is repinning callers to the last compatible version and re-issuing the change behind a SemVer-major bump with a deprecation window and migration doc - turning an accidental break into a governed one.

**Scenario D - WAL-format change.** An `eventd` format change makes a runtime unable to read prior history. Recovery uses RFC 0004 machinery: the format change must carry a read path for the prior format; failing that, the runtime rewinds to a pinned `StepHash` below the affected region (a `RewindEvent`), and dependency-aware GC guarantees no downstream consumer's checkpoint was pinned to truncated history. The hash chain self-attests that the recovered prefix is intact. [C12]
