# vakedz — the Zig front-end for Vaked

> Vaked declares. Nix materializes. OTP supervises. **Zig enforces.** eBPF
> testifies. CrabCC indexes. Surfaces reveal.

`vakedz` is a faithful, **content-addressed-cache-native** port of the `vakedc`
pipeline (`parse → check → lower`) to Zig. The Python `vakedc/` stays the
**reference**; `vakedz` is the production front-end that the runtime layer
("Zig enforces") will eventually carry.

It is also the first place Vaked's central research bet becomes a **native
primitive**: the `ralphloop-cache` (`src/cache.zig`) — an immutable,
content-addressed, hash-chained ledger — so the compiler's own history is a
replayable graph increment instead of a recomputed pass. Same source ⇒ same
key ⇒ replay, never recompute. This *closes the dogfooding loop*: the loop that
builds the language is itself built on the language's core idea (see
`tools/ralph/PURPOSE.md`).

| Stage | Status | Notes |

| **ralphloop-cache** (content-addressed, hash-chained) | ✅ shipped | format-compatible with `ralph`/`eventd` ledgers |
| **check** (0011 type system) | 🚧 scaffold | pipeline wired; 22 diagnostics not yet ported |
| **lower** (0012 emitters) | 🚧 scaffold | refuses to emit; emitter port is the backlog |

The full `parse → check → lower` port is the tracked v0.1→v1.0 backlog (see the
plan doc + GitHub issues). Every gap found by dogfooding becomes an issue, per
the repo convention — that issue stream *is* the backlog.

## Build & run (devshell)

The toolchains come from `nix develop` (Zig is not assumed global):

```bash
task vakedz-build              # → vakedz/zig-out/bin/vakedz
task vakedz-test               # zig build test
task vakedz-verify             # build + byte-parity gate vs. vakedc
task parse-zig -- vaked/examples/engines/zig.vaked

# direct:
cd vakedz && zig build && zig build test
./zig-out/bin/vakedz parse ../vaked/examples/operator-field.vaked --print
./zig-out/bin/vakedz cache verify
```

CI (`.github/workflows/vakedz-ci.yml`) pins Zig 0.14.0 (the `build.zig.zon`
floor), builds, runs the unit tests, and runs the **cross-verify gate** — the
single source of truth that the port is faithful.

## Layout

```
vakedz/
├── build.zig / build.zig.zon   # Zig 0.14 build; pinned floor
├── src/
│   ├── main.zig                # CLI: parse | check | lower | all | cache
│   ├── json.zig                # canonical JSON (byte-parity with vakedc/emit.py)
│   ├── lexer.zig               # v0.3 tokenizer (vakedc/lexer.py)
│   ├── parser.zig              # recursive-descent AST (vakedc/parser.py)
│   ├── graph.zig               # AST → LPG → canonical emit (resolve.py + graph.py + emit.py)
│   ├── cache.zig               # the ralphloop-cache primitive
│   ├── check.zig               # 0011 checker (scaffold)
│   └── lower.zig               # 0012 lowering (scaffold)
└── test/
    ├── golden/*.graph.json     # goldens generated by `vakedc parse` (parity targets)
    └── crossverify.sh          # the closed-loop dogfooding gate
```

## The byte-parity contract

The reference's canonical JSON is *not* a blanket key-sort. The structural
wrappers are **fixed order** — top-level `version,source,nodes,edges`; node
`id,kind,name,labels,props,provenance`; edge `from,to,label,props`; provenance
`file,decl,span`; span `byteStart,byteEnd,line,col`. Only the **props subtree**
is recursively key-sorted. Nodes sort by `id`; edges by `(from,label,to,props)`.
Strings escape like CPython's `json` (no `/` escaping, UTF-8 passthrough), with
a trailing newline. `src/json.zig` encodes exactly this; `src/graph.zig`
sorts only the props value (`sortRecursive`).
