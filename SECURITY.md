# Security Policy

## Scope

Vaked is a **research and experimental project**. It is not deployed in production, has no user accounts, and processes no personal data. There is no runtime to exploit beyond the local dev shell and the vakedos bare-metal NixOS target described in [`DEPLOY.md`](DEPLOY.md).

Security reports are welcome, but response time and remediation commitments are best-effort only — this project carries no SLA.

## What is in scope

- The `vakedc` compiler: miscompilation that would produce artifacts enabling privilege escalation or sandbox escape on the target host.
- The eBPF policy manifests generated by `vakedc lower`: policies that silently fail open or grant more capability than declared.
- The HCP / Litany wire protocol design (RFCs under `protocol/rfcs/`): authentication bypass, replay, or capability-escalation flaws in the protocol specification.
- The vakedos NixOS configuration (`hosts/vakedos/`): misconfigurations that weaken the security posture of the bare-metal host.

## What is out of scope

- Vulnerabilities in upstream dependencies (Nix, NixOS, the Linux kernel, Zig, Erlang/OTP, Rust). Report those to the respective upstream projects.
- Issues that only affect the stub/skeleton portions of the runtime that are not yet implemented.
- Denial-of-service against the development machine or CI environment.

## Reporting

Report security issues by email to **cabotage@protonmail.com**. Please include:

- A description of the issue and its potential impact.
- Steps to reproduce or a proof-of-concept.
- The affected file(s) or component(s).

**Do not open a public GitHub issue for security vulnerabilities.**

Disclosure timeline: this project follows coordinated disclosure. Once a report is received, a response will be sent within a reasonable time (no formal SLA). If no response is received within 14 days, the reporter may disclose at their discretion.
