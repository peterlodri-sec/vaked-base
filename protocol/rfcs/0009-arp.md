# RFC 0009 — ARP: behavioral primitives + model adapters for AI-lish

- **Status:** Draft
- **Created:** 2026-06-14
- **Track:** Agent protocol

## Abstract

**ARP (Agent Register Protocol)** is a thin behavioral layer that an agent emits
*alongside* [AI-lish V1](../../docs/ailish/2026-06-14-ailish-v1-rfc.md) frames. AI-lish
V1 already owns the hard parts — the SSA execution-graph grammar, the eight registers
and their monad rules, typed atoms, dataflow/justification edges, gates, and the
`gate(*:fail)` → `R:commit` freeze invariant — and ships a `nom` parser + guardrail +
formatter in the [`ailish`](../../ailish/) crate. ARP does **not** redefine any of that.

ARP adds the four *behavioral* signals that AI-lish V1 does not carry — **Stride**,
**Tension**, **Valence**, **Branch** — plus the **per-model adapter notes** needed to run
the register convention across model families. These were specified in issue #202 (ARP)
before AI-lish V1 landed; this RFC records the genuinely additive subset and explicitly
cedes the grammar to AI-lish V1.

> Scope note. An earlier parallel effort drafted a competing AIL grammar + a 3-register
> set. That is **withdrawn** in favor of AI-lish V1 (8 registers, `⊕` deliberately
> removed in favor of named funcs, SSA `%N`). ARP is what survives: the progress/affect/
> fork signals and the portability layer, riding on V1's syntax.

## Terminology

Aligned with [`docs/protocol/README.md`](../../docs/protocol/README.md). New here:

| Term | Definition |
|------|------------|
| ARP | Agent Register Protocol — the behavioral signals + model adapters in this RFC, layered on AI-lish V1. |
| Stride | A declared progress arc (`[STRIDE: a → b → c]`) emitted before acting; a retry/checkpoint anchor. |
| Tension | A goal-distance signal (`[T:N]`, 0..100) that drives compression and gives the harness an escalation threshold. |
| Valence | A polarity signal (`[+]` / `[-]` / `[!]`) emitted after a tool result. |
| Branch | An explicit fork (`[BRANCH: a \| b; condition: X]`) marking a retry/decision checkpoint. |
| AI-lish V1 | The execution-graph IR + grammar these signals ride on ([RFC](../../docs/ailish/2026-06-14-ailish-v1-rfc.md), [`ailish/`](../../ailish/) crate). |

## 1. Scope & relationship to AI-lish V1

ARP is **purely additive**. The division of ownership is strict:

| Concern | Owner |
|---------|-------|
| Grammar, EBNF, parser, formatter | AI-lish V1 (`ailish/` crate, [RFC §2/§5](../../docs/ailish/2026-06-14-ailish-v1-rfc.md)) |
| Registers (`R:think…R:bench`), monad rules, SSA `%N`, typed atoms, dataflow/justification, funcs | AI-lish V1 (RFC §2–§3) |
| Gates + the `gate(*:fail)` → `R:commit` freeze invariant | AI-lish V1 (RFC §3) |
| **Stride / Tension / Valence / Branch signals** | **ARP (this RFC, §2)** |
| **Per-model adapter notes** | **ARP (this RFC, §3)** |

ARP frames are emitted inline among AI-lish frames. They are advisory signals for the
harness and the human reader; they carry **no authority** and do not gate or mutate state
(that is AI-lish V1's `R:risk`/`R:commit` + the guardrail). If the AI-lish maintainers
choose to fold any ARP primitive into the V1 grammar, that is an AI-lish grammar change
(its own process), and the compact-form question in §5 applies.

## 2. The four behavioral primitives

Spellings are the long forms from issue #202. Compact forms are intentionally **not**
assigned here — AI-lish V1 already owns the `[!_]` compact namespace (RFC §5), so any
compact ARP tag must be reconciled against it (§5, open).

### 2.1 Stride — `[STRIDE: step → step → step]`

A declared atomic arc of intended progress, emitted *before* the steps run. It mirrors
AI-lish V1's `R:plan` schedule (`→ target`) but as a single self-contained signal, so a
harness has the intended arc for auto-retry if a step fails. Steps reference the same
action/atom vocabulary as V1; `→` here is "then", consistent with V1 reserving `→` for
dataflow inside graph frames (a Stride is a plan annotation, not a graph edge).

### 2.2 Tension — `[T:N]`, `N ∈ 0..100`

Goal-distance. High early, low as the task converges. Two uses: (1) it drives natural
compression — a near-done task needs little prose; (2) it gives the harness a threshold to
escalate on (e.g. tension stuck high across many turns). Empirically useful against models
that over-explain at low tension (DeepSeek R1, §3). Lexically distinct from V1's `[!T]`
(think) by the colon + value and the absence of `!`.

### 2.3 Valence — `[+]` / `[-]` / `[!]`

A polarity signal emitted after a tool result: `[+]` good, `[-]` bad, `[!]` attention. A
harness can threshold-escalate on a run of `[-]`. (Reconciliation note: distinct from V1's
`[!X]` compact registers, which are always bang+letter; see §5.)

### 2.4 Branch — `[BRANCH: a | b; condition: X]`

An explicit fork: alternative paths and the condition that selects between them. It marks a
checkpoint for auto-retry on the first path's failure, complementing V1's `depend()` /
graph structure without encoding dataflow.

## 3. Per-model adapters

The register convention works because it exploits register-switching capacity models
already have; the failures observed in the cuc-bench five-model run are *adapter* problems,
not protocol problems. (Bench: gpt-4o-mini 61.8%, llama-3.3-70b 73.2%, deepseek-r1 50.8%,
minimax-m3 48.9% token savings, gate held; morph-v3 and qwen3 failed the gate — addressed
below.)

- **Claude / Anthropic** — full system-message support; inject the register declarations +
  the artifact-gate rule in the system prompt. Maps cleanly to the CUC activation phrase.
- **DeepSeek (R1, V3)** — system messages supported; the `<think>` chain runs in the
  planning register automatically. Inject an explicit low `[T:N]` to curb over-explanation.
- **Gemini** — system instruction via the `system_instruction` field (not
  `messages[0].role=system`). Map the artifact register to an explicit "standard English
  only for file/commit/PR text" instruction; Stride declarations help Gemini not abandon
  multi-step chains.
- **Single-turn (e.g. `morph/morph-v3-large`)** — no system-message channel; fold the
  register declaration + gate into the first user message. Mid-conversation register
  switching is unavailable, so state the gate upfront. (Explains morph's bench gate failure.)
- **Thinking models with non-English CoT (e.g. `qwen/qwen3-235b`)** — the chain-of-thought
  activates in Chinese and bleeds CJK into artifact outputs (qwen's bench gate failure).
  Adapter: suppress CJK in `<think>` for artifact output, or use the provider's "thinking
  off" when the artifact register is active. AI-lish V1's `R:artifact` posture + the
  guardrail are the backstop.

## 4. Composition with HCP

ARP, like AI-lish V1, is in-context text — no wire, no encoding. It composes one level
below HCP/Litany: HCP declares capability *between* agents; ARP/AI-lish describe behavior
*within* one agent's turn. An HCP session grant MAY carry an initial register/tension
assignment. ARP needs none of HCP's machinery, and HCP does not depend on ARP.

## 5. Open questions

1. **Compact forms.** AI-lish V1 owns the `[!_]` compact namespace (RFC §5). If ARP
   primitives earn compact forms, they must be assigned *by the AI-lish maintainers*
   without colliding with `[!T]`/`[!P]`/…; until then ARP uses only the long forms above.
2. **Fold into V1 grammar?** Should Stride/Tension/Valence/Branch become first-class V1
   productions (a V1 grammar revision in the `ailish` effort), or remain advisory
   annotations? This RFC assumes advisory; promotion is the maintainers' call.
3. **Emitted vs injected.** Is `[T:N]` a model self-report or a harness-injected control?
   The cuc-bench register-compliance metric should settle which is load-bearing.
4. **Security.** ARP carries no authority and MUST NOT be used to grant capability (that is
   `preceptord`/HCP and V1's guardrail). A harness consuming ARP from untrusted content
   treats these signals as advisory, never as privilege.
