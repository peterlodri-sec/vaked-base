AIL-0 Design Grounding: Is a Register-Tagged Compression IL for LLM Agents Sound?

1. Bottom-line verdict

The design is sound in principle but its core novelty is the least-evidenced part. A small, parseable intermediate language that (a) removes low-information content and (b) routes work into a compact structured representation has solid, token-measured backing: algorithmic compressors hit 20x token reduction with ~1.5pt accuracy loss (LLMLingua), structured pseudocode DSLs cut input tokens 55-87% and output tokens 41-70% (CodeAgents), and gating only the final artifact while leaving reasoning free recovers accuracy that rigid schemas destroy (Let Me Speak Freely). The single biggest caveat: AIL-0's distinctive mechanism — hand-authored, register-tagged ([R:plan] [R:tool] [R:artifact]) compression — has no direct token-savings evidence in this dossier. Every supporting result is either orthogonal (algorithmic, not stylistic compression) or single-source self-reported (CodeAgents on its own benchmark). The register tags themselves are pure token tax with no measured payback. AIL-0 is therefore a defensible hypothesis that must be benchmarked, not a settled win — and the cleanest cautionary tale (CJK glyph density, below) shows exactly how a "compression" that looks dense to humans can increase token count.

2. Main-topic synthesis

What AIL-0 should compress vs keep exact. The evidence cleanly separates two compression mechanisms with opposite track records. Algorithmic low-information removal — perplexity/self-information pruning of tokens — is repeatedly token-measured and robust: LLMLingua reaches up to 20x token compression on GSM8K losing only ~1.52 EM points (78.85 to 77.33) [supported], LLMLingua-2 does task-agnostic 2x-5x [supported], Selective Context cuts ~50% context with proportional memory/latency drops that can only come from fewer tokens [supported], and Gist tokens hit up to 26x [supported]. By contrast, stylistic hand-authored compression (telegraphic/"caveman" English, CJK shorthand) is either unmeasured or actively undercut. The design implication is stark: AIL-0 should lean on algorithmic-style information density (drop low-signal tokens, prefer compact structured forms) and must NOT assume that terse-looking authoring converts to token savings.

The structured-DSL strand is AIL-0's strongest direct analog and its thinnest evidence base simultaneously. CodeAgents is the one source reporting genuine token reduction from codifying agent reasoning into a structured pseudocode/DSL: 55-87% input and 41-70% output token cuts with 3-36pp accuracy gains [supported]. This is the closest published proxy for what AIL-0 proposes. But it is abstract-only, single-source, self-reported on the authors' own suite, with no independent replication, and the reduction is partly definitional (a DSL is mechanically more compact than the NL it replaces). Adjacent agent-DSL work supports format but not token claims: CodeAct beats text/JSON action formats by 20.7pp success rate and uses up to 30% fewer actions/turns [supported] — but "fewer turns implies fewer tokens" is an unsupported, likely-false extrapolation, because one CodeAct turn composes multiple tool calls and can be more token-dense [mixed]. AIL-0 should claim CodeAgents-style token savings only as a hypothesis to validate, and should never cite CodeAct as token evidence.

Reasoning must stay free-form; only artifacts may be gated. The strongest design constraint in the dossier is that constraining the reasoning channel costs accuracy. Forcing a rigid JSON schema across the whole response collapses GSM8K accuracy for Claude-3-Haiku from 86.51% to 23.44% (~63pp) [supported], with the mechanism being field-order: 100% of GPT-3.5-Turbo JSON-mode responses put answer before reason, collapsing chain-of-thought into direct answering [supported]. Forcing monolingual-Chinese reasoning costs -5.6pp on MATH500 (p=0.0017) — measured on QwQ-32B-Preview, NOT DeepSeek-R1 [supported in output-language-gate]. The fix is the artifact gate: reason in natural language first, then convert to schema ("NL-to-Format"), which recovers nearly all lost accuracy [supported]. This directly validates AIL-0's register split: [R:plan] reasoning should be loosely-constrained free text, while [R:tool]/[R:artifact] outputs can carry tight structure.

Constrained decoding is a real tool with real, framework-specific costs. Decode-time grammar masking guarantees structural validity by construction and can match-or-beat unconstrained accuracy when prompts are matched and a reasoning span is preserved (JSONSchemaBench, Llama-3.1-8B: GSM8K 80.1→83.8 with Guidance) [supported]. The "format degrades reasoning" result is contested — dottxt/Outlines (an interested vendor) shows matched-prompt structured generation winning (0.77→0.78 GSM8K) [supported]. But the costs are concrete: coverage collapses on complex schemas (GitHub-Hard empirical coverage: Outlines 3%, XGrammar 28%, llama.cpp 39%, Guidance 41%) [supported], grammar compilation is framework-dependent overhead (Outlines ~3.48s vs Guidance ~0.00s) [supported], and per-token latency ranges from 53% faster (Guidance) to 151% slower (Outlines) [supported]. If AIL-0 is enforced via a grammar, the parser/grammar must stay simple enough to avoid the coverage cliff, and the framework choice is load-bearing.

The honest accounting on token economics. AIL-0 saves tokens only where it (1) removes low-information content algorithmically or (2) replaces verbose NL with a genuinely more compact representation that the tokenizer rewards. It costs tokens via register tags ([R:plan] etc. are added tokens), via any CJK/rare-symbol glyphs (byte-fallback inflation, below), and potentially via constrained-decoding compile/latency overhead. The net is unknown and must be benchmarked per-tokenizer. The claim that per-character savings track per-token savings is explicitly [mixed] and unmeasured in every primary source.

3. Per-strand evidence

3.1 prompt-compression (algorithmic — the solid core)

  • [supported] [high] LLMLingua compresses prompts up to 20x in tokens with little performance loss across GSM8K, BBH, ShareGPT, Arxiv-March23; ratio is token-defined (tau = L̃/L). 20x is the favorable maximum, not the headline average. (2310.05736)
  • [supported] [high] At 20x token compression on GSM8K, EM drops only ~1.52pt (78.85 to 77.33, 2,366→117 tokens) — and 20x is the worst row shown (14x scores 77.41, 5x beats baseline at 79.08). (html v2)
  • [supported] [high] LLMLingua-2: task-agnostic 2x-5x token ratios, 3x-6x faster than prior methods, 1.6x-2.9x lower end-to-end latency. Extractive token-classification, so the ratio is token-based. (2403.12968)
  • [mixed] [medium] LLMLingua-2 retaining ~98.7% accuracy at "14x" on GSM8K is a single hand-picked case study on a marketing page, not a dataset mean; actual ratio is 13.3x rounded up to 14x. Token reduction real; framing is the issue. (llmlingua.com)
  • [supported] [high] Selective Context: token-level pruning, ~50% context reduction, 36% memory and 32% latency cuts — those memory/latency drops are only achievable by feeding fewer tokens, proving genuine token (not character) savings. (2310.06201)
  • [supported] [medium] Selective Context's marketing "2x more content, 40% memory/GPU savings" is unit-ambiguous ("content", not tokens/chars); the underlying mechanism is token pruning. (GitHub)
  • [supported] [high] Gist tokens compress prompts up to 26x (original prompt tokens → learned gist tokens), up to 40% FLOPs reduction, 4.2% wall-time speedup, "minimal" (qualitative) quality loss. Token-based. (2304.08467)
  • [mixed] [medium] These ratios are all token-savings; no primary source reports any character ratio, so the claim that per-char savings are "generally lower" is unverifiable inference, not measurement. (sources above)

3.2 nl-style-compression (stylistic — undercut or unmeasured)

  • [supported] Classical Chinese (Wenjian): the ~28% reduction was character-counted, not token-counted. Measured token effect: -13.2% on o200k_base but +6.4% WORSE on cl100k_base, and only vs the AAAK compact baseline (not English prose). Cleanest char≠token case in the set. (ai.rs)
  • [supported] Older BPE splits most CJK chars into 2-3 tokens; newer o200k represents CJK better — explaining the cl100k-negative/o200k-positive flip. Token efficiency depends on tokenizer vocab, not language terseness. (ai.rs, 2604.14210)
  • [supported] Controlled coding study: Chinese was NOT more token-efficient — 1.28x tokens (MiniMax-2.7), 1.09x (GPT-5.4-mini), 0.98x (GLM-5) vs English. Model/tokenizer-dependent, arithmetically self-consistent. (2604.14210)
  • [supported] Switching to Chinese also degraded task success: -4.5pp (MiniMax), -9.9pp (GPT-5.4-mini), -9.5pp (GLM-5). The one model where Chinese saved tokens (GLM-5, 0.98x) dropped most in quality. Token savings can carry a quality cost. (2604.14210)
  • [supported] The telegraphic/"caveman" 40-60% token-reduction claim is ASSERTED, not measured — no tokenizer named, no before/after counts, quality is anecdotal developer observation. (trend.hulryung.com)
  • [supported] Dropping function words plausibly cuts English tokens (English BPE ~1.1-1.35 tokens/word) but no rigorous before/after telegraphic-vs-prose token measurement exists — an evidence gap asymmetric to the rigorously-debunked CJK side. (tiltstack, OpenAI cookbook)
  • [mixed] Algorithmic (not stylistic) removal demonstrably cuts tokens (LLMLingua 20x/-1.52pt; BBH 7x/-13.2pt; Arxiv 9x) — but the claim's "ShareGPT 9x" is wrong (source says 3.3x; 9x is Arxiv). Mechanism = small-LM budget controller + iterative perplexity compression, explicitly NOT a writing style. (html v2, GitHub)

3.3 tokenizer-symbols-cjk (the byte-fallback tax)

  • [supported] cl100k_base tokenizes most CJK chars as 2-3 tokens (not 1) because a ~100k vocab can't hold a dedicated token per 97,000+ CJK ideograph. Token claim, vocab-pressure mechanism well-supported. (tonybaloney)
  • [supported] GPT tokenizers are byte-level BPE: text → UTF-8 bytes → BPE merges on a 256-byte base, so any char is representable without <unk>. Mechanism, not numeric. (fast.ai, HF)
  • [mixed] Rare Unicode symbols (→ ⇒ ∵ ∴ ⊕) can fall back to ~1 token/byte (2-4 tokens each) — but whether these specific symbols are absent from the merge vocab is assumed, not measured; → (U+2192) is common in code/math and is a plausible single-token counterexample. Mechanism supported, 5-symbol application unverified. (HF, fast.ai, tiktoken)
  • [supported] SentencePiece (Llama, Mistral) runs BPE on code points; byte_fallback represents rare/OOV chars as UTF-8 bytes (multiple tokens). (fast.ai)
  • [mixed] character_coverage (0.99995 Llama 2) governs byte fallback — but it is coverage by occurrence frequency, not "0.0005 of codepoints" by count; minor metric misstatement plus a slight HF misattribution. Substance correct. (fast.ai)
  • [mixed] o200k_base ~doubles vocab vs cl100k (199,997 vs 100,256), improving CJK compression. Vocab-doubling supported; the specific Japanese "9→8 tokens" figure is not on the cited primary URLs (single-source, unverified). (grokipedia, llm-calculator)
  • [mixed] CJK costs ~1.7x-2.4x more tokens than English (Mandarin 1.76x, Japanese 2.12x, Korean 2.36x) — page-grounded ratios supported. The secondary "200-300 tokens/100 chars vs ~130/100 words" has a baseline mismatch (apples-to-oranges). (tonybaloney)
  • [mixed] Chinese-trained tokenizers (Qwen, DeepSeek, GLM) add CJK vocab so Chinese can tokenize more efficiently (GLM 0.923 ZH/EN) — substance independently corroborated, but the named source URLs do not contain these numbers (low provenance). (llm-calculator, biggo)

Token-vs-char takeaway for this strand: glyph density is not token density. CJK and rare symbols can cost extra tokens via byte fallback on the wrong tokenizer; the same text flips between savings and loss purely by tokenizer choice.

3.4 cjk-reasoning-leakage (why a free-reasoning channel can drift)

  • [supported] DeepSeek-R1's chain-of-thought exhibits language mixing, worse with multilingual RL prompts; may reason/respond in English even for non-EN/non-ZH queries. (2501.12948)
  • [supported] R1's mitigation is a "language consistency reward" (proportion of target-language words in CoT). (2501.12948)
  • [supported] That reward slightly degrades task performance but improves readability/human preference (qualitative, no number). (2501.12948)
  • [supported] RLVR-trained models mix most, then pre-trained base, then SFT/RLHF least. Single source. (2507849)
  • [mixed] MODEL MISATTRIBUTION: forcing monolingual-Chinese reasoning costs -5.6pp on MATH500 (p=0.0017) — but the intervention ran on QwQ-32B-Preview, NOT DeepSeek-R1 (R1 is excluded because it enforces language consistency). (2507849)
  • [supported] Benefit of mixing is dataset-dependent: monolingual constraint hurts MATH500 but constrained monolingual-Chinese beats unconstrained bilingual on Gaokao Cloze. (Underlying model QwQ, not R1.) (2507849)
  • [supported] A lightweight probe guiding code-switch decoding gains up to +6.25pp (Gaokao Cloze max), smaller MATH500 average (+1.62±1.01 — error bar near zero), +3.33pp cross-dataset AIME2024. (2507849)
  • [supported] QwQ-32B-Preview officially documents language mixing/code-switching as a known limitation leaking into final outputs; users report Chinese chars in Russian and Vietnamese responses. (HF model card, discussion #16)
  • [mixed] System-prompt language pinning is reported with inconsistent efficacy (one user: "didn't work"); the "commonly recommended" framing is an overstatement from one thread. (discussion #16)

3.5 grammar-constrained-decoding (enforcement costs)

  • [supported] JSONSchemaBench: six frameworks (Guidance/llguidance, Outlines, llama.cpp GBNF, XGrammar, OpenAI, Gemini) over ~10,000 real schemas, 10 complexity-tiered datasets. (2501.10868)
  • [supported] Coverage diverges widely on hard schemas (GitHub-Hard, Llama-3.2-1B): Guidance 41%, llama.cpp 39%, XGrammar 28%, Outlines 3%; GitHub-Medium Guidance 69%/Outlines 29%. Empirical coverage; 1B model is a deliberately hard low-end slice. (2501.10868)
  • [supported] Per-token latency can drop (Llama-3.1-8B, llama.cpp): Guidance 6.37ms TPOT vs 15.40ms unconstrained, while Outlines (30.33ms) and llama.cpp grammar (29.98ms) ~2x slower. Genuine ms/token. (2501.10868)
  • [supported] Grammar-compilation is a distinct framework-dependent overhead: Outlines ~3.48s vs Guidance ~0.00s on GlaiveAI. (2501.10868)
  • [supported] XGrammar claims up to 100x speedup of the mask-generation engine, "near-zero overhead" when integrated — a vendor/author self-claim. (2411100)
  • [mixed] Constraints don't uniformly help accuracy: JSONSchemaBench shows ~+3pp when a separate reasoning field is reserved (GSM8K 80.1→83.8); CRANE notes constraints "can reduce functional correctness." Numbers supported; the unifying causal rule ("helps iff reasoning field reserved") is synthesized, not demonstrated. (2501.10868, 2502.09061)
  • [supported] CRANE (reason-then-constrain interleaving) beats both plain-constrained and unconstrained on symbolic reasoning by up to ~10pp (GSM-Symbolic +9pp, FOLIO +6.9pp). (2502.09061)
  • [supported] OpenAI Structured Outputs (strict mode, gpt-4o-2024-08-06): constrained decoding guarantees schema adherence, 100% on OpenAI's eval vs <40% for gpt-4-0613 — single-origin vendor figure, widely repeated, not independently measured. (OpenAI, Protecto)

3.6 agent-dsls (the structured-representation case)

  • [supported] CodeAct (executable Python as unified action space) beats next-best (text) action format by 20.7pp success rate on M3ToolEval. Success-rate, not token, claim. (2402.01030)
  • [mixed] CodeAct uses up to 30% fewer actions and 2.1 fewer turns (gpt-4-1106) — but the headline "more token-efficient" rests on the unsupported, likely-false inference that fewer turns → fewer tokens. The paper measures no token count; a CodeAct turn can be more token-dense. Do not cite as token evidence. (2402.01030)
  • [supported] ReAct (interleaved reasoning + actions) beats imitation/RL baselines by 34% (ALFWorld) and 10% (WebShop) with 1-2 in-context examples. Success-rate claim. (2210.03629)
  • [supported] CodeAgents (structured pseudocode/DSL for multi-agent reasoning) cuts input tokens 55-87% and output tokens 41-70% while improving planning accuracy 3-36pp (GAIA/HotpotQA/VirtualHome). Abstract says "token usage" / "token-aware metrics" → genuine token reduction. Caveats: abstract-only, single-source, self-reported, reduction partly definitional, no independent replication. AIL-0's closest published analog. (2507.03254)
  • [supported] "Let Me Speak Freely?": whole-response JSON-with-schema drops GSM8K accuracy 86.51%→23.44% (Claude-3-Haiku), 75.99%→49.25% (GPT-3.5), 75.13%→48.90% (LLaMA-3-8B); two-step NL-then-convert recovers nearly all. Accuracy claim. (2408.02442)
  • [supported] That degradation is contested: dottxt/Outlines (interested vendor) argues it stemmed from JSON-mode setup + mismatched prompts, and their grammar-constrained generation matches/beats unstructured (77% vs 73% vs 68%). Supported as "the result is contested," not as "dottxt is right." (dottxt)
  • [supported] Grammar/schema-constrained decoding eliminates malformed tool calls by construction — a structural (format) guarantee, explicitly NOT semantic correctness. Mechanism claim, low-confidence by author. (tianpan.co)

3.7 output-language-gate (the artifact-gate pattern — AIL-0's register split)

  • [supported] Whole-response structured output degrades reasoning by ~26-42pp: GSM8K LLaMA-3-8B -25.83pp, Last Letter LLaMA-3-8B -42.07pp, Last Letter GPT-3.5 -31.54pp. Measures prompt-instruction/JSON-mode with non-matched prompts; contested by next claim. (2408.02442)
  • [supported] Contested/mechanism-dependent: with matched prompts + true decode-time masking, structured matches or beats unconstrained (GSM8K 0.77→0.78, Last Letter 0.73→0.77, Shuffle 0.41→0.44). Vendor rebuttal (dottxt), but methodology critique is sound. (dottxt)
  • [supported] Artifact-gate architecture (reason free-form, constrain only the FINAL output) recovers nearly all lost performance via NL-to-Format; 100% of GPT-3.5 JSON-mode responses put answer before reason, collapsing CoT. Directly validates AIL-0's [R:plan]-free / [R:artifact]-constrained split. (2408.02442)
  • [supported] Independent matched-prompt benchmark (JSONSchemaBench, Llama-3.1-8B): constrained decoding does NOT hurt and slightly improves (GSM8K 80.1→83.8 Guidance, etc.). No CIs reported → robust reading is "does not degrade"; the +3pp is the best-case (Guidance) framework. (2501.10868v3)
  • [supported] Dominant real-world failure is coverage collapse on complex schemas, not invalid output: GitHub-Hard Outlines 3% / XGrammar 28% / llama.cpp 39% / Guidance 41%; OpenAI 9% coverage but 100% compliance on accepted; GlaiveAI >86% for all. (2501.10868v3)
  • [supported] Latency is framework-specific, not intrinsic: GitHub-Easy TPOT unconstrained 15.83ms, Guidance 7.44ms (53% faster), llama.cpp 27.22ms (72% slower), Outlines 39.78ms (151% slower); compile Guidance ~0s, Outlines 3.71s. (2501.10868v3)
  • [supported] Constraining the reasoning language costs accuracy (evidence FOR leaving reasoning free): forcing monolingual-Chinese reasoning -5.6pp MATH500 (p=0.0017, QwQ-32B-Preview); probe-guided code-switching +1.62 to +2.92pp. (2507849v2)
  • [thin] The exact bullseye — "reason freely in any language, force the FINAL answer into target language, maintains/improves accuracy" — is feasible and directionally supported, but affirmative answer-only-gate quantification is genuinely absent and thinner than the schema-gate evidence. (2505.05408)

3.8 constrained-gen-metrics (how the bench must measure)

  • [supported] IFEval: strict/loose accuracy at prompt-level and instruction-level; "loose" applies text transforms to cut false negatives. (2311.07911, inspect_evals)
  • [supported] IFEval = 25 verifiable instruction types across ~500 prompts. (2311.07911, HF)
  • [thin] GPT-4 (0613) ~77% strict prompt-level on IFEval — exact decimal rests on a single unverifiable secondary source; directional point (frontier models high-70s) plausible. (2406.12793)
  • [supported] JSONSchemaBench metrics: declared coverage, empirical coverage, compliance rate (empirical/declared), plus GCT/TTFT/TPOT efficiency. (2501.10868)
  • [supported] Empirical coverage degrades sharply with complexity: GitHub-Hard Guidance ~0.41 (llama.cpp 0.39, Outlines 0.03) vs GlaiveAI ~0.96. (2501.10868v3)
  • [supported] Help-or-hurt depends on constraint design, not constraining per se: answer-before-reasoning schema degrades; a schema permitting a reasoning span preserves/improves. Synthesis across two papers, qualitative. (2408.02442, 2501.10868v3)
  • [supported] Answer-before-reasoning schema can cut GSM8K accuracy ~63pp for Claude-3-Haiku (86.51→23.44; high variance sd 22.9). (2408.02442)
  • [mixed] Constrained decoding guarantees grammar validity by construction (per-step masking), but NOT full schema validity (empirical coverage 0.41/0.39/0.03 on hard; compliance 0.98 even on easy ≠ 100%) and never semantic correctness. "100% structural validity" is overstated if read as schema compliance. (2501.10868v3)

4. Design implications for AIL-0

  1. Compress by algorithmic information density, not by terse authoring. Token-measured wins (LLMLingua 20x/-1.52pt, Selective Context ~50%, Gist 26x, §3.1) all come from removing low-information tokens algorithmically. Stylistic terseness (caveman 40-60%, telegraphic English) is [supported]-as-unmeasured (§3.2). AIL-0 should treat its compact forms as candidates for programmatic low-signal removal, and must not ship a "40-60% savings" number without a tiktoken before/after on the target tokenizer.

  2. Do NOT rely on CJK glyphs or rare Unicode for compression. [strand:nl-style-compression] shows wenyan's 28% is character-counted, flipping to -13.2% (o200k) vs +6.4% worse (cl100k) in tokens (§3.2); [strand:tokenizer-symbols-cjk] shows CJK costs 1.76x-2.36x tokens and rare symbols fall back to per-byte tokens (§3.3). Register sigils and any non-ASCII should be validated against every target tokenizer; assume byte-fallback inflation until measured.

  3. Keep [R:plan] reasoning free-form; gate only [R:tool]/[R:artifact]. The artifact-gate / NL-to-Format result (Claude-Haiku 86.51→23.44 under whole-response schema, recovered by reason-then-convert, §3.7) and the -5.6pp monolingual-reasoning cost on QwQ-32B-Preview (§3.4, §3.7 — note: QwQ, not DeepSeek-R1) both say: never put structure in the reasoning channel. AIL-0's register split is the right architecture if tags don't leak constraints into [R:plan].

  4. If you grammar-enforce AIL-0, keep the grammar simple and pick the framework deliberately. Coverage collapses on complex schemas (Outlines 3% on GitHub-Hard) and latency ranges 53% faster to 151% slower by framework (§3.5, §3.7). A small EBNF is an asset here — it stays on the high-coverage, low-compile-cost side. Guidance/llguidance and XGrammar are the low-overhead end per the dossier; Outlines compile (~3.48s) is a real tax.

  5. Treat register tags as a token tax that must be earned back. Tags like [R:plan]/[R:tool]/[R:artifact] are added tokens with no measured savings in the dossier. The only direct token-saving analog is CodeAgents (55-87% input / 41-70% output, §3.6) — single-source and self-reported. Budget the bench to prove tags + structure net-save against a plain-NL baseline.

  6. Never cite CodeAct/ReAct as token evidence; cite them for format/success-rate only. CodeAct's 20.7pp and 30%-fewer-actions are success/step metrics; "fewer turns → fewer tokens" is [mixed]-unsupported and likely false (§3.6). AIL-0's token claim must rest on its own measurement, not borrowed step-efficiency.

  7. Guarantee only structural validity, validate semantics separately. Constrained decoding gives grammar-validity by construction but not schema-compliance (compliance 0.98 ≠ 1.0 even on easy) and never semantic correctness (§3.8, §3.6). AIL-0's parser can guarantee a parse; it cannot guarantee the [R:tool] call is correct.

5. Open questions / thin evidence the bench must settle

  1. Does AIL-0's register-tagged format net-save tokens vs plain NL on the target tokenizer? No dossier source measures hand-authored register-tagged compression. CodeAgents is the only token-saving DSL result and is single-source/self-reported (§3.6). Bench must report input+output token deltas, per tokenizer (cl100k vs o200k at minimum), against a plain-NL baseline.
  2. The English-side stylistic-compression gap. Function-word dropping is word-plausible but token-unmeasured (§3.2). Bench a tiktoken before/after of telegraphic vs full-prose AIL-0.
  3. Affirmative answer-only-gate quantification. The exact "reason freely, force final artifact into target form, accuracy preserved" claim is [thin] (§3.7) — no compliance percentages located. Bench must measure both artifact-schema compliance and downstream accuracy under the gate.
  4. Does the help-or-hurt rule actually hold for AIL-0's schema? The "helps iff a reasoning span is reserved" mechanism is [mixed]-synthesized, not demonstrated (§3.5, §3.8). Run a controlled with/without-reasoning-field ablation on AIL-0 itself.
  5. Per-char vs per-token relationship is unmeasured everywhere (§3.1). If AIL-0 markets any character-based density figure, it must separately report the token figure — they can diverge in sign (CJK case).
  6. Grammar coverage/latency for the actual AIL-0 EBNF. Coverage and TPOT are framework- and complexity-specific (§3.5, §3.7); none measured for AIL-0. Bench the chosen framework on AIL-0's real grammar, reporting declared coverage, empirical coverage, compile time, and TPOT.
  7. Frontier instruction-following ceiling is uncertain (IFEval GPT-4 ~77% is [thin], §3.8) — don't assume models will follow AIL-0 syntax perfectly; measure IFEval-style strict/loose adherence to the format itself.
