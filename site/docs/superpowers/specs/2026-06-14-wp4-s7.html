WP4-S7 — vakedos deploy test (#114)

Status

Implementation spec (2026-06-14). Sprint WP4-S7 of the WP4 daemon-MVP track
(docs/superpowers/plans/2026-06-14-wp4-kickoff.md,
table row "WP4-S7 | Sep 18–Sep 30 | vakedos deploy test (#114)"). It is the
last sprint of WP4: prove that the bare-metal host
(hosts/vakedos/, nixosConfigurations.vakedos in
flake.nix) with the WP4 daemon roster layered on top
deploys and comes up operational.

This is a planning artifact. The end-to-end deploy it specifies is multiply
blocked (see §7) — most of what it tests does not exist yet (the
nixosModules.<runtime> wiring layer, the emitted systemd units, two of the
three daemons). S7 therefore specifies the harness, oracles, and gates
completely and concretely, and marks the end-to-end run blocked-on those
dependencies, exactly the way
2026-06-14-wp4-s2.md gated its config-driven path on
design step 0 (see its §1.2 and criterion 7). Over-claiming a deploy that cannot
run is the failure mode this spec is written to prevent.

Build / verification environment. dev-cx53 (Linux x86_64, Nix 2.34.7, kernel
6.x) is the WP4 build target (kickoff lines 21–22) and is OFF-LIMITS for the
current 6h autoresearch window. Note dev-cx53 is a build host, not the
deploy target — the deploy target is the Vultr bare-metal EPYC 4345P with a ZFS
mirror and Legacy/BIOS boot (DEPLOY.md §"Target
hardware"), which no CI host reproduces. Everything below marked M1-local
must pass on aarch64-darwin (zig 0.16.0, cargo 1.95.0, arch arm64, verified
2026-06-14) without dev-cx53; everything marked Linux-VM needs an x86_64
KVM-capable Linux host; everything marked bare-metal needs the box itself
(#114 / spring-2027 horizon).

1. Objective

Produce a deploy test that answers, with mechanical oracles rather than
eyeballing, two questions for nixosConfigurations.vakedos + the WP4 daemon
roster:

  1. Does it build/switch? — the config evaluates, the x86_64-linux toplevel
    realizes, and nixos-rebuild switch (or its VM equivalent) succeeds. This is
    the kickoff success line "nixos-rebuild switch succeeds on bare metal"
    (ROADMAP_2026-2027.md WP4-7 row).
  2. Are the daemons operational? — agent-supervisord, sandboxd, and
    eventd come up under systemd and reach a protocol-defined operational
    state, not merely systemctl is-active. The oracle for "operational" is
    RFC 0004 §6 (cold-start verifies dependency freshness before RUNNING) and
    RFC 0001 §5 (write-ahead-before-effect; only canonical frames on the eventd
    chain), §2.1 below.

The primary deliverable is a nixosTest (pkgs.testers.runNixOSTest,
exposed as checks.<system>.vakedos-deploy) that boots the config in a VM and
asserts (1) and the VM-observable subset of (2). The nixosTest is what bridges
"dev-cx53 off-limits / the box does not exist yet" and "still mechanically
verifiable." The bare-metal-only properties (ZFS mirror, BIOS/GRUB, ECC) stay a
checklist test against DEPLOY.md §"5. Verify" for the
day the hardware is provisioned.

1.1 In scope

  • A nixosTest (checks.<system>.vakedos-deploy) that boots vakedos-in-a-VM,
    asserts the system reaches multi-user.target, and asserts daemon
    operational state via the RFC oracles (§2.1, §4).
  • A pure-eval gate (nix flake check + module-option assertions) that runs
    on M1 without realizing the closure (§5.1).
  • A bare-metal verify checklist test: a structured, machine-checkable
    encoding of the DEPLOY.md §5 steps, tagged
    VM-coverable vs box-only (§4.3, §5.3).
  • A systemd-unit interface contract test: assert that the units the daemons
    need are declared with the dependency order RFC 0001 §5 / RFC 0004 require —
    written against the contract so it fails loudly the moment the deferred
    emitter (lowering §7) lands wrong (§4.2).
  • The M1-vs-Linux-VM-vs-bare-metal split, named per check (§5).

1.2 Out of scope (explicit, with provenance so a reader does not think they were missed)

  • nixosModules.<runtime> itself. S7 consumes the runtime wiring module
    (the layer that installs gen/ artifacts and points daemons at them, lowering
    docs/language/0012-lowering.md §4.3); it
    does not author it. Today flake.nix defines only
    nixosModules.vaked-telebot — the runtime module does not exist. This is
    S7's primary blocker (§7).
  • Emitting the systemd units. The systemd.units lowering target is
    interface-stubbed / deferred (lowering §7 table: "unit details depend on
    the daemon packaging, deferred with the daemons … The OTP supervision tree
    itself is no longer deferred: otp.supervision (§3.4) emits it"). S7 does not
    write the emitter; it writes the contract test the emitter must satisfy.
  • Building the daemons. agent-supervisord and eventd daemon dirs are
    MISSING (daemons/agent-supervisord/, daemons/eventd/ do not exist);
    daemons/sandboxd/ is a WP4-S1 stub. S1–S6 are hard prerequisites (§7); S7
    assumes their artifacts and tests them in situ.
  • WP4-S6 integration test. No S6 spec exists in this dir (only s2/s3). S7
    does not re-specify "3 daemons running together" — that is the S6
    deliverable (kickoff row "WP4-S6 | … | integration test (3 daemons running on
    vakedos)"). S7 layers the deploy/boot dimension on top of whatever S6
    produces and consumes its integration harness as a building block (§7).
  • eventd byte-format / hash-chain encoding. The canonical frame encoding and
    the SHA-256 chain are owned by eventd (WP4-S4) and the hcpbin spec (WP3 /
    RFC 0002 §6, worked examples §6.1.1 etc.) — not S7. S7 asserts only
    "eventd is up and a write-ahead append advances the chain" via the RFC 0001
    §5 / RFC 0004 §6 behavioural oracles, never the byte layout. (For the record,
    the kickoff/issue phrasing "RFC 0002 §4 / Appendix A" is stale: the encoding
    is RFC 0002 §6 with no Appendix A, and frame headers are RFC 0003 — S7
    cites neither because it stays out of the encoding.)
  • eBPF policy manifest, OTel collector config, surface launcher — the other
    three deferred lowering §7 targets. The vakedos substrate exposes their
    kernel/system facilities (bpftool, nft, etc. — DEPLOY.md §5), but wiring the
    daemon configs onto them is downstream of the runtime module (§4.3 note: "the
    artifacts themselves are no-ops today").

2. Inputs and oracles

2.1 "Operational" is an RFC oracle, not systemctl is-active

A deploy test must define operational mechanically. For the WP4 roster the
definition comes from the protocol, not from systemd liveness:

Oracle Where What S7 asserts from it
Cold-start verifies before RUNNING RFC 0004 §6 "Invariant III — cold start verifies before RUNNING"; algorithm §6.1 (provisional) After boot, agent-supervisord runs cold-start verification for each direct producer dependency before a consumer enters RUNNING; a stale/absent anchor leaves the consumer in stale_dependency (paused), not RUNNING (RFC 0004 §6, lines 570–618; stale_dependency defined line 51). "Operational" = the supervisor performed cold-start and the no-dependency daemons reached RUNNING.
Write-ahead before effect RFC 0001 §5 "Supervision and state dependencies: agent-supervisord" (lines ~425–467) Every applied control action is logged to eventd before it takes effect (agent → … → agent-supervisord → eventd → effect). S7's smoke asserts a control verb produces an eventd append that precedes the observable effect.
Only canonical frames on the chain RFC 0001 §5 "Event log and tamper-evidence: eventd" (lines ~444–462) eventd is up and an append advances the hash chain (S7 asserts advancement + tamper-evidence behaviour, never the byte encoding — §1.2).
Roster + membrane mapping docs/runtime/README.md "Daemons" / "Membrane mapping" The three WP4 daemons map to the expected membranes (process → sandboxd; control plane → agent-supervisord; audit spine → eventd) and the systemd dependency order reflects it.

2.2 Host-config inputs (present today, no host change in S7)

  • flake.nix nixosConfigurations.vakedos (modules:
    disko.nixosModules.disko, ./hosts/vakedos/configuration.nix).
  • hosts/vakedos/configuration.nix —
    nixpkgs.hostPlatform.system = "x86_64-linux", gcc.arch = "znver4", Legacy
    GRUB (boot.loader.grub.efiSupport = false), cgroup-v2, sysctls.
  • hosts/vakedos/disko.nix — ZFS mirror,
    1 MiB EF02 BIOS-boot partition, datasets (/nix /var /home /build /var/lib/vaked /var/lib/vaked per DEPLOY.md §5).
  • DEPLOY.md §5 "Verify" — the bare-metal oracle: the
    exact post-install checklist (zpool status rpool, zfs list, free -g,
    nproc, lscpu, ras-mc-ctl --status, systemctl status sshd irqbalance,
    bpftool/wasmtime/nft, SIGILL check, nix build nixpkgs#hello, survives
    reboot).

Until these land, S7 runs against a fixture runtime module

4. Algorithm / design

4.1 The nixosTest (nix/tests/vakedos-deploy.nix) — primary deliverable

pkgs.testers.runNixOSTest { nodes.machine = …; testScript = …; } boots a single
VM that imports hosts/vakedos/configuration.nix plus the runtime module
(fixture today, generated later). Because the nixosTest VM uses its own
qemu-friendly disk/boot, the test overrides the bare-metal-only pieces that a
VM cannot honour and records each override (so they are visibly not tested
here, only in §4.3 / bare-metal):

  • replace ./disko.nix (ZFS mirror + BIOS partitions) with the default
    virtualisation disk — ZFS-mirror / BIOS-GRUB / by-id disks are NOT covered
    by the VM     (box-only, §4.3);
  • drop nixpkgs.hostPlatform.gcc.arch = "znver4" for the VM run (the VM CPU is
    not guaranteed Zen 5; the global--march from-source rebuild is a build
    concern, asserted separately in §5.2) — SIGILL/-march correctness is
    box-only    ;
  • keep cgroup-v2, sysctls, sshd, and the daemon units intact — these are
    what the VM exists to test.

testScript (Python) asserts, in order:

machine.wait_for_unit("multi-user.target")        # the system switched / booted
machine.wait_for_unit("eventd.service")            # audit spine up first (roster order)
machine.wait_for_unit("agent-supervisord.service") # control plane up
machine.wait_for_unit("sandboxd.service")          # isolation daemon up
# operational, per RFC oracle (§2.1) — NOT mere is-active:
machine.succeed("vaked-supervisorctl status")      # supervisor reports its tree
# RFC 0004 §6: a no-dependency daemon reaches RUNNING; cold-start ran
assert "RUNNING" in machine.succeed("vaked-supervisorctl state eventd")
# RFC 0001 §5: write-ahead — a control verb appends to eventd BEFORE effect
chain0 = machine.succeed("vaked-eventctl head")     # chain head pre-action
machine.succeed("vaked-supervisorctl pause sandboxd")
chain1 = machine.succeed("vaked-eventctl head")     # chain advanced (append happened)
assert chain0 != chain1
assert "paused" in machine.succeed("vaked-supervisorctl state sandboxd")  # effect followed

The exact vaked-supervisorctl / vaked-eventctl surface is whatever S3/S6 ship
(S7 consumes it); the assertions above are the contract those tools must make
observable. Where S6's integration harness already exposes equivalent
introspection, S7 calls it rather than reinventing.

Why a nixosTest and not a script: it is the only way, without the box, to
exercise "switch succeeds + daemons operational" against the real unit
graph — it boots the actual config, starts the actual systemd units, and runs
assertions inside the guest. It needs x86_64 KVM (Linux / dev-cx53 lane),
not M1.

4.2 Systemd-unit interface contract (test_module_contract.py)

Independent of booting anything, assert the runtime module declares the units
and dependency order the protocol requires. Source of truth:
fixtures/vakedos-deploy/expected-units.json. Assertions (each cites its
oracle):

  • eventd.service, agent-supervisord.service, sandboxd.service all declared.
  • agent-supervisord.service has After=eventd.service and
    Requires=eventd.service — RFC 0001 §5 write-ahead: the supervisor logs to
    eventd before effect, so eventd must be up first.
  • sandboxd.service After=agent-supervisord.service — sandboxd is the exec
    mechanism the supervisor drives (runtime README membrane map: process →
    sandboxd; supervisor orchestrates the Zig daemons).
  • supervisor unit type/Type=notify (or equivalent) so systemd's notion of
    "started" aligns with the cold-start-complete moment (RFC 0004 §6).

Implementation: nix eval --json the module's
config.systemd.services.<name>.{after,requires,…} and diff against
expected-units.json. Pure eval ⇒ M1-local (§5.1). Written against the
contract, so it fails the instant the deferred §7 emitter wires the order wrong.

4.3 Bare-metal verify checklist (test_deploy_checklist.py + deploy-checklist.toml)

Encode DEPLOY.md §5 as {cmd, expect, tier} rows. tier
∈ {vm, box}:

Item (DEPLOY.md §5) tier Why

daemon units active + operational (§4.1) vm the nixosTest covers this

On M1/CI the box-tier rows are collected and reported as skipped (box-only),
not silently dropped — so the checklist stays an honest record of what is not
yet verified. They become live assertions only when run on the provisioned host.

4.4 Reconciling vakedz lower vs vakedc lower (named, not silently picked)

The kickoff success criterion says daemons start "from systemd units generated by
vakedz lower" (kickoff line 73); the Taskfile.yml deploy: target uses
vakedc lower. These are the Zig and Python front-ends of the same compiler;
their reconciler is the vakedz-verify byte-parity gate (Taskfile.yml
vakedz-verify: — "assert byte-parity with vakedc on the goldens"). S7 does not
need to pick: whichever front-end emits the units, the bytes are gate-equal, so
the contract test (§4.2) is front-end-agnostic. Crucially, both emit nothing
for systemd.units today — that target is deferred (lowering §7) — so S7's
unit source is the §2.3 fixture until the emitter lands.

5. Test plan — M1-local vs Linux-VM vs bare-metal

Three tiers, because a deploy test does not split cleanly into two. The trap is
the build: evaluating the config is pure and M1-local; realizing the
x86_64-linux closure is not.

5.1 Runs fully on M1 (aarch64-darwin) — pure evaluation, no closure realized

Check Command Why M1 is sufficient
Config evaluates nix flake check (eval leg) pure Nix eval; arch-independent
Toplevel resolves nix eval .#nixosConfigurations.vakedos.config.system.build.toplevel.drvPath resolves the derivation path without building it
Unit interface contract (§4.2) pytest tests/spec/vakedos_deploy/test_module_contract.py → nix eval --json …systemd.services pure eval of module options
Deploy checklist parse + vm-tier dry rows (§4.3) pytest …/test_deploy_checklist.py box-tier rows reported skipped; parse/shape checked

Boundary the build-step makes explicit: nix build .#nixosConfigurations.vakedos.config.system.build.toplevel (DEPLOY.md §3
"Dry-build") does not plainly run on M1 — realizing an x86_64-linux closure
needs a Linux builder (nix-darwin linux-builder VM or a remote builder). Do
not treat DEPLOY.md §3 as M1-local: it assumes a Linux laptop. On M1 the
dry-build is eval-only (drvPath), and realization moves to the Linux-VM tier.

5.2 Requires a Linux x86_64 host (dev-cx53 lane, KVM) — closure realize + VM boot

Check Why it cannot gate on M1
Realize the toplevel (nix build …toplevel) builds an x86_64-linux closure; the global znver4 -march rebuilds from source (no cache) — needs a Linux builder
nixosTest boot + daemon operational (§4.1) runNixOSTest boots a qemu guest and runs assertions inside it; needs x86_64 KVM
--vm-test (DEPLOY.md §3, nixos-anywhere --vm-test) boots the install in a VM; Linux/KVM only
daemon units reach RUNNING + write-ahead chain advance (§2.1) exercises real systemd + the running daemons

dev-cx53 is the host for this lane but is off-limits for the current window —
so during the window S7's Linux-VM checks are authored and self-reviewed but
run only when dev-cx53 frees up (or on any other x86_64 KVM Linux host). dev-cx53
also cannot stand in for the bare-metal tier (§5.3): it is a build host, not
the Vultr EPYC with the mirror/BIOS/ECC layout.

5.3 Requires the bare-metal box (Vultr EPYC 4345P) — #114 / spring-2027 horizon

The box-tier checklist rows (§4.3): ZFS mirror health, disko datasets, BIOS/GRUB
boot survival, real RAM/CPU/core counts, ECC via ras-mc-ctl, and the SIGILL /
-march correctness check that only the real Zen 5 CPU running the from-source
znver4 closure can confirm. A nixosTest VM cannot cover these. They are the
DEPLOY.md §5 checklist, run once on the provisioned host.

5.4 CI wiring (.github/workflows/wp4-s7-deploy.yml)

  • Job eval (any runner, incl. M1): §5.1 checks. Gates every PR.
  • Job vm (x86_64 KVM Linux): realize toplevel + nixosTest. Runs on the
    WP4 Linux lane (nix develop on the build host); skipped where KVM absent.
  • Every new test file cites the RFC § / DEPLOY.md § / lowering § it enforces (the
    WP3-S7 adversarial-independence convention,
    2026-06-14-wp3-s7.md criterion 8; carried by S2 §6.3).

6. Algorithm summary (deploy-test control flow)

vakedos-deploy:
  # tier 1 — M1 / any runner (pure eval)
  flake_check()                            # evaluates
  drv = eval(toplevel.drvPath)             # resolves, does NOT build
  assert_unit_contract(runtime_module)     # RFC 0001 §5 / RFC 0004 §6 order
  parse_checklist(); report box-rows skipped

  # tier 2 — Linux x86_64 + KVM (dev-cx53 lane; off-limits now)
  build(drv)                               # realize x86_64-linux closure
  nixosTest:                               # runNixOSTest, VM boot
    wait_for multi-user.target
    wait_for eventd, agent-supervisord, sandboxd
    assert cold-start ran + no-dep daemon RUNNING       # RFC 0004 §6
    assert control verb → eventd append BEFORE effect    # RFC 0001 §5
    (override ZFS/BIOS/-march for the VM; record as box-only)

  # tier 3 — bare-metal box (#114 / spring 2027)
  run DEPLOY.md §5 checklist box-rows on the provisioned EPYC

8. Acceptance criteria

S7 is done when all hold:

  1. Pure-eval gate (M1-local). nix flake check passes and
    nix eval .#nixosConfigurations.vakedos.config.system.build.toplevel.drvPath
    resolves on aarch64-darwin without realizing the closure (§5.1).
  2. Unit interface contract (M1-local). test_module_contract.py asserts the
    three daemon units exist with RFC-correct ordering
    (agent-supervisord After/Requires eventd per RFC 0001 §5; sandboxd After agent-supervisord) against expected-units.json; it is written so it
    fails if the deferred §7 emitter (or the fixture) gets the order wrong
    (§4.2).
  3. nixosTest exists and is correct-by-construction (authored M1, runs
    Linux-VM).     nix/tests/vakedos-deploy.nix is a runNixOSTest exposed as
    checks.<linux>.vakedos-deploy; its testScript asserts multi-user.target,
    the three units up, cold-start-ran + no-dependency daemon RUNNING (RFC 0004
    §6), and a control verb appending to eventd before its effect (RFC 0001
    §5). The VM overrides (ZFS/BIOS/-march) are recorded as box-only (§4.1).
  4. Operational is RFC-defined, not is-active. Every "operational"
    assertion cites RFC 0004 §6 or RFC 0001 §5; no assertion is mere
    systemctl is-active standing in for operational (§2.1).
  5. 3-tier split is explicit and honest. Each check is tagged
    M1 | linux-vm | box; the box-tier DEPLOY.md §5 rows are reported
    skipped (box-only)     off-hardware, never silently dropped (§4.3, §5.3). The
    nix build …toplevel realize step is correctly placed in the linux-vm
    tier, not M1 (§5.1 boundary note).
