WP4-S4 — eventd Zig port: core + eBPF skeleton

WP4-S4 — eventd Zig port: core + eBPF skeleton

Status: Spec (2026-06-14). Planning artifact — work is dependency-blocked
(see §8). Author target: Track B "immutable leg" (#17/#18), runtime roster role
eventd (docs/runtime/README.md:16).

Scope in one line: port the byte-fixing core of the Python eventd
reference (eventd/core.py, eventd/log.py) to Zig and prove it produces
byte-identical output to Python across the 11 golden vectors that
tests/spec/test_eventd.py pins, plus a compile-only eBPF witness skeleton.

1. Objective

Produce eventdz, a Zig daemon/library that is the production carrier of
Vaked's tamper-evident audit spine ("eventd witnesses",
docs/runtime/agent-guardd.md:3-4). This sprint delivers the #15 port
pattern for eventd: Python fixes the bytes; Zig reproduces them
(eventd/__init__.py:3-6). Concretely:

  1. Canonical JSON + hash chain (core) — reproduce
    eventd.core.{canonical_json, chain_hash, make_entry, verify_chain, entry_line} (eventd/core.py) byte-for-byte, including the cross-language
    hash contract spelled out in the core.py module docstring
    (eventd/core.py:17-37).
  2. Daemon shape (log) — reproduce eventd.log.EventLog + load_entries
  3. repair_truncate_tail semantics (eventd/log.py): single-writer
    flock, fsync-on-append, boot-time chain verification as a hard error,
    torn-tail / malformed-line refusal, crash-atomic repair.
  4. eBPF witness skeleton — a compile-only BPF_PROG_TYPE_CGROUP_SKB-class
    scaffold mirroring the agent_guardd reference (agent_guardd/bpf.py),
    establishing the kernel datapath that will append Event.Ebpf testimony
    into the eventd chain (docs/runtime/README.md:29,
    docs/runtime/agent-guardd.md:33-53). Load/attach/verifier runs are
    Linux-only and deferred (§8/§7).

The byte-identity bar (explicit): byte-identity is measured on the
on-disk log line (entry_line, eventd/core.py:83-89), not merely the
hashed payload bytes. That is the stricter, cleaner gate and it is what "two
runs ⇒ byte-identical logs" already means for the reference
(tests/spec/test_eventd.py:210-228). It implies the Zig writer MUST emit the
entry wrapper in the reference's sorted key order (see §4.4).

1.1 Non-goals (drawn explicitly)

  • RFC 0004 state-dependency fold is NOT in S4. eventd.statedep's own
    docstring defers the zero-copy scan path to "a Zig-port concern, explicitly
    after correctness" (eventd/statedep.py:16-17). DependencyIndex,
    gc_floor, verify_cold_start, eviction semantics — out of scope; see §6.
  • Runtime operational fold is NOT in S4. eventd.runtime
    (RuntimeState.fold, run/step lifecycle) is a later port; see §6.
  • CLI surface (verify/append/replay/state/floor/coldstart/
    runs/repair)     — S4 ports only the two CLI verbs needed to drive the
    parity gate (verify, append) plus repair --truncate-tail. The fold-
    dependent verbs (state, floor, coldstart, runs) ride on §6.
  • No Rust component. This is a Zig port. cargo exists in the environment
    only because other sprints (e.g. Rust-side tooling) share the tree; S4 ships
    no Rust.

2.2 The 11 golden vectors (clarification)

"11 golden vectors" = the 11 entries of _VECTORS in
tests/spec/test_eventd.py:50-64. These are NOT the RFC 0002 §6.1.1 / §6.3.1
/ §6.6.1 / §6.7.1 worked examples. Those examples encode hcpbin (the wire
binary form); eventd's on-disk format is the JSONL hash chain of
core.py, a different layer (see §3.1). The kickoff/issue wording "RFC 0002 §4
/ Appendix A" is stale on both counts: RFC 0002 has no Appendix A, and its byte
encoding lives in §6 (hcpbin encoding rules,
protocol/rfcs/0002-hcplang.md:439), while the frame header is in RFC 0003
(WIRE), explicitly not hcpbin (0002 §4.2).

3. The canonical-JSON dialect — the cross-language hash contract

Source of truth: eventd/core.py:17-37. Restated as Zig acceptance rules:

  1. UTF-8 output, ensure_ascii=False. Non-ASCII is raw UTF-8 bytes, never
    \uXXXX. Only JSON-mandatory escapes: \", \\, and control chars
    < 0x20 as \uXXXX (lowercase hex, zero-padded to 4). / is NOT escaped.
    json.zig's writeString (vakedz/src/json.zig:98-121) already does
    exactly this (its test :149-155 proves CPython parity) — reuse it.
  2. Object keys sorted by Unicode code point; separators (",",":"), no
    insignificant whitespace. Sort is the full recursive key sort of the
    payload tree (json.zig:sortRecursive, :70-83). ⚠️ The vakedc graph
    path only sorts the props subtree — do NOT inherit that; for payloads the
    whole tree sorts (vector 1 sorts nested too).
  3. Integers in full decimal, arbitrary precision (vector 7 is 10²⁰).
  4. Floats in Python repr / json.dumps shortest-round-trip form
    (§3.2 — the real work).
  5. NaN / Infinity rejected (allow_nan=False → ValueError;
    core.py:51-52). Zig equivalent: return error.NonFiniteFloat.
  6. No Unicode normalization (vectors 8 vs 9). This is deliberately the
    opposite     of hcpbin (§3.1).

3.1 What this is NOT — hcpbin (RFC 0002 §6)

eventd's hashed bytes are the JSONL canonical form above. RFC 0002 §6
(hcpbin encoding rules, protocol/rfcs/0002-hcplang.md:439) is a different
layer and on three points says the opposite of the eventd vectors. Cite §6
only to disclaim it:

hcpbin §6.4 (:671) eventd JSONL (the oracle)
-0.0 folds to +0.0 preserved as -0.0 (vector 4)
strings NFC-normalized, Unicode 15.1 pinned no normalization (vectors 8≠9)
NaN canonical quiet-NaN bits rejected
integers LEB128 varint, zig-zag for signed (§6.1, :451) decimal ASCII

Letting §6.4 leak into the port fails vectors 4, 8, 9. The hcpbin encoder is a
separate, future Zig artifact (it belongs with the RFC 0002/0003 wire
stack, not S4).

3.2 Float formatting — the single biggest risk

The vectors require CPython repr-shortest-round-trip floats. Zig's
std.fmt does not match Python's grammar out of the box (exponent
threshold and exponent zero-padding differ). Verified renderings from the
oracle (json.dumps, 2026-06-14):

0.0        -> 0.0
-0.0       -> -0.0          (sign preserved)
1.0        -> 1.0           (always a ".0", never "1")
-3.5       -> -3.5
1e-6       -> 1e-06         (lowercase e, sign, 2-digit zero-padded exponent)
1e-5       -> 1e-05
0.0001     -> 0.0001        (fixed form down to 1e-4)
0.00001    -> 1e-05         (switches to exponent at < 1e-4)
1e16       -> 1e+16         (positive exponent gets explicit '+')
1e17       -> 1e+17
1e21       -> 1e+21
1.5e-10    -> 1.5e-10
123456789.123 -> 123456789.123

Required Zig behavior (a new floatRepr routine — not in json.zig
today):

  • Shortest decimal that round-trips the IEEE-754 f64 (Ryū/Grisu-class;
    Zig's std.fmt.format with {e}/{d} is insufficient — it neither
    matches the exponent threshold nor guarantees Python's shortest form).
  • Emit fixed-point notation for exponents in roughly [-4, 16), scientific
    otherwise — match CPython's float_repr thresholds exactly (the table
    above is the acceptance oracle; extend it with more probe values during
    implementation rather than trusting any single library).
  • Scientific form: <mantissa>e<sign><≥2-digit exponent>, lowercase e,
    explicit +/-, exponent zero-padded to at least 2 digits.
  • A whole-number float still carries .0 (1.0, not 1).
  • Preserve -0.0.
  • NaN/±Inf → error.NonFiniteFloat.

Because this is the highest-risk code, it gets its own module + its own
micro-golden test (§7.1) seeded with the table above plus boundary probes
(9999999999999998.0, 1e15, 5e-324, etc.).

3.3 Integer width — bignum

Vector 7 is 100000000000000000000 (10²⁰), which exceeds both i64 and
u64. The contract is arbitrary precision; for the 11 vectors i128
(max ≈ 1.7e38) is sufficient — say so explicitly. The JSON parser that
reads payloads back (for verify/append-after-load) must therefore decode
large integer literals into i128 (or a small bignum) without lossy float
coercion. json.zig's Value is int: i64 today (json.zig:26) — widen to
i128 (or add an bigint: []const u8 decimal-string variant carried
verbatim). Note for the record that a truly unbounded big integer would need a
decimal-string passthrough; i128 is the S4 floor, larger is a tracked follow-up.

4. Algorithm / design

4.1 Reuse map (what already exists in Zig)

vakedz/src/json.zig and vakedz/src/cache.zig already implement the same
frozen chain ("format-compatible with ralph/eventd ledgers",
cache.zig:23-24). Reuse, do not reinvent:

Need Reuse Caveat
sha256 hex cache.zig:sha256Hex (:39-43) none
chain_hash(prev,payload) cache.zig:chainHex (:47-54) none — identical to eventd.core.chain_hash
CPython string escaping json.zig:writeString (:98-121) none
recursive key sort json.zig:sortRecursive (:70-83) apply to the whole payload, not just props
torn-tail verify cache.zig:verify + rawPayload (:213-261) adapt: eventd refuses (TamperError), cache returns prefix
genesis const cache.zig:GENESIS (:34) none

Do NOT copy cache.zig:put's entry serialization. It builds the entry
wrapper in insertion order {seq,prev,payload,hash} and never sorts the
top level (cache.zig:192-197). eventd's entry_line is sorted
{hash,payload,prev,seq} (core.py:83-89; confirmed in §2.3). S4 needs a new
entry serializer that sorts the wrapper.

4.2 Module layout (paths to create)

A sibling Zig project under daemons/eventd/ (mirroring daemons/sandboxd/),
keeping the production daemon out of the vakedz compiler front-end:

S4 (avoid coupling the compiler front-end's build to the daemon build now).

4.3 core.zig API (matches eventd/core.py)

pub const GENESIS: [64]u8 = "0"**64;  // = cache.GENESIS

// canonical JSON of an already-built (and key-sorted) payload Value.
pub fn canonicalJson(alloc, payload: *json.Value) ![]u8;   // sortRecursive then writeCanonical

pub fn chainHash(prev_hex: []const u8, payload_canonical: []const u8, out: *[64]u8) void; // = cache.chainHex

pub const Entry = struct { seq: i64, prev: [64]u8, payload_canonical: []const u8, hash: [64]u8 };
pub fn makeEntry(prev_hex, seq, payload) !Entry;

// entry_line: sorted wrapper {hash,payload,prev,seq} + '\n'
pub fn entryLine(alloc, e: Entry) ![]u8;

pub fn verifyChain(entries) bool;   // seq 0..n; prev links; hash recomputes (re-canonicalizing payload)

verifyChain MUST re-canonicalize each entry's payload from its parsed form
and recompute chainHash(prev, canon) (exactly verify_chain,
core.py:67-80) — it must NOT trust the literal payload substring on disk
beyond parsing it (a tamper that re-sorts keys but leaves a stale hash must
fail). The rawPayload substring trick from cache.zig:253-261 is acceptable
only if the on-disk payload is already canonical; for a hardened audit spine,
parse → re-canonicalize → compare (matches the Python reference; slower but
correct). Document the choice; default to parse-and-recanonicalize.

4.4 Wrapper key order (load-bearing)

entry_line = json.dumps(entry, separators=(",",":"), sort_keys=True, ensure_ascii=False, allow_nan=False) + "\n" (core.py:88-89). With
sort_keys=True over keys {seq,prev,payload,hash} the on-disk order is
hash, payload, prev, seq. The Zig entry serializer builds the wrapper Value
with those four keys and runs the same sort (the payload sub-Value is itself
already canonical). Confirmed against §2.3 goldens.

4.5 log.zig — daemon shape (matches eventd/log.py)

State machine and refusal paths to reproduce exactly:

  • Open (reader): EventLog(path) → load_entries then verifyChain;
    failure → error.Tamper before any entry is served (log.py:175-181).
  • Open (writer): take LOCK_EX | LOCK_NB FIRST, then load+verify under
    the lock (log.py:164-184). A second writer → error.WriterLock. A refused
    (tampered) writer-open MUST release the lock (log.py:182-184;
    test_eventd.py:146-162).
  • append(payload): chain off tail_hash, write entryLine, flush,
    fsync before return (log.py:205-214).
  • load_entries byte-oriented: split on \n, drop empty; a line that is
    bad JSON or undecodable UTF-8 or valid-JSON-but-not-an-object →
    error.Tamper (NOT a decode traceback) (log.py:46-68,
    test_eventd.py:136-173). Zig: read bytes, attempt UTF-8 decode + JSON
    parse per line; map both failures to error.Tamper.
  • Single-writer is cross-process (flock, test_eventd.py:175-200). On
    macOS M1 flock(2) is available and advisory-exclusive — testable locally
    (§7). Note flock portability caveat for the eventual Linux deployment is a
    tracked open item (eventd-design.md:122).
  • repair_truncate_tail (log.py:96-146): take the writer lock; compute
    the longest valid prefix (_longest_valid_prefix, :71-93 — reuse
    cache.zig:verify's walk, but stop-on-first-break and keep the prefix);
    dropped = byte_lines - prefix; if dropped<=0 → {repaired:false};
    else append a log_repair entry {"kind":"log_repair","v":1, "dropped":N} chained off the prefix tail, written crash-atomically
    (temp file → fsync → os.replace → dir fsync; log.py:131-143). Result
    {repaired, dropped, tail_seq}. Idempotent on a clean log
    (test_eventd.py:764-767). A break anywhere truncates everything after it
    (a chain cannot resume past a broken link; log.py:99-106).

The log_repair payload key order: canonical JSON sorts to
{"dropped":N,"kind":"log_repair","v":1} — the Zig writer gets this for free
from sortRecursive.

4.6 eBPF witness skeleton (ebpf.zig) — compile-only

Defined role (grounded, not invented): eventd is the witness/sink for
the eBPF testimony datapath. docs/runtime/README.md:29 —
ebpf → agent-guardd (...) + eventd; docs/runtime/agent-guardd.md:33-36 —
eBPF testifies → Event.Ebpf payload per verdict → eventd append → hash-chained log.jsonl. So eventd's eBPF leg is the kernel-side complement to the
userspace chain: a minimal BPF_PROG_TYPE_CGROUP_SKB-class program whose
verdicts (drop/pass per destination) become Event.Ebpf payloads appended via
log.zig:append. agent-guardd owns policy compilation/attach; eventd owns the
witnessing of verdicts onto the immutable chain.

S4 deliverable (scaffold only):

  • ebpf.zig reproduces the agent_guardd/bpf.py reference shape in Zig:
    the bpf() syscall ABI constants (__NR_bpf=321 x86-64,
    BPF_PROG_LOAD=5, BPF_PROG_ATTACH=8, BPF_PROG_TYPE_CGROUP_SKB=15,
    BPF_CGROUP_INET_EGRESS=1; bpf.py:33-39), the 8-byte BPF instruction
    encoder (<BBhi: op, (src<<4)|dst, off, imm; bpf.py:51-53), and a
    compilePosture() that assembles a deny/pass-by-default program
    (MOV64_IMM + EXIT; bpf.py:42-43).
  • Event.Ebpf payload constructor: a Zig fn producing the verdict payload
    dict (fields TBD with agent-guardd; minimally {kind:"ebpf", verdict, dest, ...}) that routes through core.canonicalJson → log.append. This
    ties the skeleton to the byte contract from day one.
  • A bpf() wrapper that, on a non-Linux/incapable host, degrades to the
    reference datapath     rather than crashing — exactly the
    agent_guardd/bpf.py:19-22 discipline ("a kernel without BPF degrades
    cleanly").

Explicitly deferred to Linux/dev-cx53 (§8): real BPF_PROG_LOAD
(in-kernel verifier run), BPF_PROG_ATTACH at BPF_CGROUP_INET_EGRESS, and
any live verdict capture. S4's gate for this leg is "compiles for
x86_64-linux", nothing more.

5. Test plan

Two-tier, matching the environment constraints (dev-cx53 off-limits 6h;
macOS M1).

5.1 Verifiable locally on M1 (zig version = 0.16.0 confirmed present)

  1. floatrepr micro-gate (test/floats.golden): floatrepr.zig over the
    §3.2 table + boundary probes → exact-string match. Run as zig build test.
    This is the gate that de-risks the hardest code.
  2. canonical-JSON parity (test/golden/canon.txt): for each of the 11
    vectors, core.canonicalJson == the committed canonical line, and
    chainHash-chained hash == the committed hash. zig build test +
    crossverify.sh.
  3. entry-line byte parity (test/golden/vectors.jsonl): chaining the 11
    vectors from genesis produces a log file byte-identical to the committed
    vectors.jsonl (which the Python oracle generated; §2.3). This is the
    "byte-identical to Python ref" headline.
  4. determinism: write the 11 vectors twice → identical bytes
    (mirrors test_eventd.py:210-228).
  5. log discipline (all work on macOS — flock/fsync/open are POSIX):
  6. append + reopen verifies; entry count preserved.
  7. flip one payload byte → reopen raises error.Tamper
    (test_eventd.py:126-135).
  8. malformed line / torn-UTF-8 tail / non-object tail → error.Tamper,
    not a decode traceback (test_eventd.py:136-144,769-807).
  9. second writer (in-process) refused; refused-writer-open releases the lock
    (test_eventd.py:114-162).
  10. cross-process second writer refused (spawn a holder process, mirror
    test_eventd.py:175-200).
  11. repair --truncate-tail: torn tail refuses pre-repair; repair drops it,
    logs log_repair, re-verifies; idempotent on a clean log
    (test_eventd.py:738-767).
  12. drift guard (in crossverify.sh, like vakedz/test/crossverify.sh:50-58):
    re-derive the goldens from the live Python oracle and warn/fail if the
    committed goldens drift — pins Zig and Python to the same bytes forever.
  13. CLI exit-code parity for the ported verbs against the stable oracle
    contract (eventd/__main__.py:40-44): verify → 0 (ok) / 4 (tampered);
    append → 0 / 4 / 5 (locked); repair → 0 / 5.

5.2 Compile-only on M1 (cross-compile)

  1. eBPF skeleton compiles for Linux: zig build -Dtarget=x86_64-linux
    builds ebpf.zig (and the daemon) clean. The bpf() syscall path is
    referenced but not executed. (Namespace/syscall behavior is a no-op on
    macOS; only compilation is asserted — matches the orchestrator constraint.)

5.3 Requires dev-cx53 / a Linux host (DEFERRED — §8)

  1. Real BPF_PROG_LOAD (in-kernel verifier accepts the posture program;
    capture "processed N insns").
  2. BPF_PROG_ATTACH at BPF_CGROUP_INET_EGRESS on a capable host; verdict →
    Event.Ebpf → eventd append, end-to-end.
  3. Any test asserting kernel namespace/cgroup syscall behavior (compile-only
    on M1; behavior only on Linux).

5.4 The parity gate command

# M1, no dev-cx53 needed:
cd daemons/eventd && zig build && zig build test          # tiers 1,2,4,5,7 + floatrepr
zig build -Dtarget=x86_64-linux                           # tier 8 (eBPF compile-only)
./test/crossverify.sh ./zig-out/bin/eventdz               # tiers 2,3,6 vs Python oracle

7. Acceptance criteria

S4 is done when, on a clean macOS M1 checkout (no dev-cx53):

  1. daemons/eventd/ exists with the §4.2 layout; zig build (Zig 0.16.0)
    produces zig-out/bin/eventdz.
  2. 11-vector byte-identity: chaining the 11 _VECTORS from genesis
    produces a file byte-identical to the committed test/golden/vectors.jsonl
    (the §2.3 entry lines), AND each hash equals the §2.3 value. The committed
    golden re-derives identically from the live Python oracle (drift guard
    green).
  3. float-repr gate green: floatrepr.zig matches every line of
    test/floats.golden (§3.2 table), including 1e-06, 1e+16, -0.0,
    1.0, the 1e-4 ↔ 1e-5 threshold.
  4. Hostile cases pass: vectors 8 ≠ 9 hash (no NFC); -0.0 preserved
    (vector 4); 10²⁰ exact (vector 7); NaN/Inf → error.NonFiniteFloat.
  5. Log discipline green (tier 5.1 #5): tamper/malformed/torn-tail/
    non-object-tail → error.Tamper; single-writer enforced in-process and
    cross-process; refused writer-open releases the lock; repair drops the
    unverifiable suffix, logs log_repair, re-verifies, idempotent.
  6. CLI exit codes match the oracle table (__main__.py:40-44) for the
    ported verbs.
  7. eBPF skeleton compiles for x86_64-linux
    (zig build -Dtarget=x86_64-linux clean); the Event.Ebpf payload
    constructor routes through the byte contract; the bpf() path degrades
    cleanly off-Linux. Real load/attach explicitly out of S4 (tier 5.3).
  8. crossverify.sh exits 0 (all parity + drift checks pass).
  9. README documents the byte contract, the "Python fixes the bytes, Zig
    reproduces them" #15 pattern, and the §3.1 hcpbin-is-a-different-layer note.

Out of acceptance (tracked, not S4): statedep/runtime folds, the full CLI,
real BPF load/attach, hcpbin encoding, shared-json-module convergence,
true-bignum (>i128) integers, snapshot/compaction/group-commit
(eventd-design.md:122-133).

8. Risks & open items

  • Float repr (highest): Zig std.fmt ≠ CPython float_repr. Mitigation:
    dedicated floatrepr.zig + exhaustive micro-goldens; treat §3.2 table as
    the acceptance oracle, not memory.
  • Wrapper key order: easy to inherit cache.zig's insertion order and
    silently fail tier 3 while tier 2 (hash) still passes. Mitigation: tier 3
    asserts the on-disk line, not just the hash.
  • i128 vs true bignum: i128 satisfies the 11; a payload integer > i128
    would need decimal-string passthrough. Flagged as a follow-up; the contract
    is arbitrary-precision.
  • Event.Ebpf schema not yet frozen — ebpf.zig constructor is a
    placeholder pending the agent-guardd co-design (agent_guardd/evidence.py).
  • flock portability to the eventual Linux deployment (eventd-design.md:122)
    — out of S4 (S4 only needs it working on M1 for the tests).
