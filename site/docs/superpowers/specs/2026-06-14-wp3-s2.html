WP3-S2 — Litany frame layer (RFC 0003 §4) + round-trip suite (design)

WP3-S2 — Litany frame layer (RFC 0003 §4) + round-trip suite (design)

Status

Design (2026-06-14). WP3 sprint S2 (Jul 9–Jul 23) of the HCP wire epic
(docs/superpowers/plans/2026-06-14-wp3-kickoff.md). Depends on WP3-S1
(protocol/hcp/hcpbin, codec types already landed). Creates the new
protocol/hcp/litany/ Rust crate. Convention here: design → plan → impl; this
is the design.

This sprint delivers the synchronous frame codec: the canonical
delimitation of Votive Frames on a byte-stream (RFC 0003 §4) and the frame
header encode/decode (RFC 0003 §4.4 / §4.4.1), plus a round-trip + golden +
violation test suite. It is pure bytes → (header, opaque body) → bytes with
no I/O and no connection state machine, so the whole deliverable is
cargo test-verifiable on the M1 dev machine.

Scope decision (load-bearing — read first)

RFC 0003 (protocol/rfcs/0003-litany-wire.md) is the entire wire stack:
transport contract (§3), frame delimitation (§4), handshake/negotiation (§5),
chapter/stream lifecycle (§6), liveness (§7), flow control (§8), errors (§9),
evidence (§10), conformance (§11), transport bindings (§12). That is many
sprints of work and most of it requires async I/O, a connection state machine,
and (for vsock/namespaces) Linux. A 2-week sprint whose stated dependency is
"WP3-S1 codec types" and whose stated deliverable is a "round-trip suite" can
only own the part that is a pure function over bytes.

Discriminator: anything that is bytes ↔ frame-record with no socket and no
state is IN; anything that needs a socket, a timer, or a multi-frame state
machine is OUT.

IN (this sprint)

  • §4.1 frame record — length := varint(byte-len of frame-bytes) followed by
    frame-bytes := header-bytes ++ body-bytes; back-to-back, no padding (§4.1).
  • §4.2 length prefix — unsigned LEB128 varint, reusing hcpbin's varint
    verbatim; strict overlong/non-minimal rejection; the max_frame-width overlong
    bound (§4.2, §4.3).
  • §4.3 max-frame guard — length > max_frame is a framing violation rejected
    without reading the body; length == 0 is illegal (§4.1, §4.3).
  • §4.4 / §4.4.1 frame header — encode/decode of kind/corr/stream/seq/
    end (the reserved @0..@4 space, RFC 0002 §4.2), byte-exact per the resolved
    layout in §"Frame-header byte layout" below.
  • §4.4.1 decoder invariant — header/body boundary, optional-field omission,
    out-of-order / duplicate-tag / out-of-range rejection.
  • Frame-stream codec — encode a Vec<Frame> to one byte string and decode a
    byte string to Vec<Frame> (the canonical-framing fixed point of §4, §10).
  • §4.5 violation taxonomy — every framing violation classified and surfaced
    as a typed FrameError; no mid-stream resync (§4.5).
  • Round-trip + golden + violation test suite (the named deliverable).

OUT (deferred — later sprints, with reason)

Deferred Reason Owner
Handshake / version / schema-digest negotiation (§5) live exchange + hcp.wire digest is TBD (RFC 0003 Open question 1) WP3-S2.5 / S3
Chapter & stream lifecycle, multiplexing runtime (§6) connection state machine WP3-S3 (RFC 0004 routing)
Liveness timers (§7), flow-control runtime / credit (§8) async + timers WP3-S4 (RFC 0005 control)
Transport-fault vs orderly close at the connection level (§9.1/§9.1a) needs a live connection; S2 produces the typed framing-violation values only WP3-S3+
eventd admission / hash chain (§10) integration WP3-S5
Transport bindings stdio / unix / vsock (§12) I/O + Linux (vsock) WP3-S6+
hcp.wire control-frame bodies (Appendix A schema) digest TBD; needs RFC 0002 §6.2/§6.3 record codec WP3-S2.5
≤10µs/frame perf gate (kickoff success criterion) threshold is dev-cx53 hardware WP3-S6

The frame header codec is in scope; frame body parsing is not — the wire
treats the body as an opaque self-delimiting hcpbin blob (RFC 0003 §4.1, §4.4:
"the wire treats frame-bytes as opaque except for parsing the header it itself
wrote"). S2 carries the body bytes verbatim and never decodes them.

Objective

A litany crate exposing a synchronous, allocation-conscious frame codec such
that:

  1. decode_frame(&[u8], max_frame) -> Result<(Frame, consumed), FrameError> reads
    exactly one frame record (§4.1) and reports how many octets it consumed.
  2. encode_frame(&Frame) -> Vec<u8> produces byte-identical output to the
    delimitation of §4.1/§4.2 (canonical framing — §4, §10).
  3. The pair is a fixed point: encode(decode(bytes)) == bytes for every
    canonical input, and decode(encode(frame)) == frame for every frame (the
    §6.8 / §7.2 conformance shape, applied to the framing layer).
  4. Every framing violation in the §4.5 set is rejected with a typed error and
    no attempt at resynchronization.

Inputs / oracles

Primary normative oracles (cite in tests):

  • RFC 0003 §4.1–§4.6 (protocol/rfcs/0003-litany-wire.md) — frame record,
    length prefix, max-frame guard, header position, violation set, worked example.
  • RFC 0003 §4.4.1 — the frame-header field table, canonical-encoding list,
    and decoder invariant. This is the byte-exact oracle for the header, with
    the three resolutions below.
  • RFC 0002 §4.2 (protocol/rfcs/0002-hcplang.md) — the reserved header
    fields (kind/corr/stream/seq/end) and the @0 reservation; the
    statement that the header is "supplied and validated by the Litany Wire layer."
  • RFC 0002 §6.1, §6.2, §6.3, §6.8 — varint/minimal-decode, ascending-tag /
    default-omission rules the header obeys, canonicality requirement.
  • WP3-S1 codec (protocol/hcp/hcpbin/src/lib.rs) — the Reader/Writer
    primitives and free functions S2 builds on (see §Dependencies).

Worked-example oracles to reproduce as golden vectors:

  • RFC 0003 §4.4.1 single-shot and multi-frame header examples.

Frame-header byte layout (normative for this crate — resolves RFC 0003 gaps)

RFC 0003 §4.4.1 is internally inconsistent in three places. The spec resolves
each by ranking normative prose + RFC 0002 §6.2 above example annotations,
and pins the result with a byte-exact golden vector. These are flagged as RFC
0003 errata to fold back when the RFC leaves Draft.

The header is a record encoded under hcpbin canonicality (RFC 0002 §6.2):
field := varint(tag) value-bytes, fields in strictly ascending tag order.

Tag Name Type Wire form Present when
@0 kind enum u8 (0–4) varint(0) varint(kind) always (errata R2)
@1 corr uuid varint(1) + 16 raw bytes (big-endian, NOT length-prefixed) always
@2 stream u64? varint(2) varint(stream) only if stream present
@3 seq u64? varint(3) varint(seq) only if stream present (R3)
@4 end bool varint(4) 0x01 only if end == true

Errata R1 — kind is fully tagged (00 ++ varint(kind))

§4.4.1's own annotated examples show kind bare (00 = kind 0; 02 = kind 2,
no tag byte), but §4.4 prose says the header is "encoded with the same hcpbin
rules as the body," and RFC 0002 §6.2 fixes field := varint(tag) value-bytes.
The §5.8 handshake transcript agrees: it shows 01 01 for "tag @0: kind=1" (two
bytes: tag 0x00? — see note). Resolution: every header field is fully
tagged. kind is 0x00 (tag @0) followed by varint(kind). The bare-byte
annotations in §4.4.1 are a documentation shorthand, not the wire form.

Note on §5.8: that transcript writes 01 01 and labels it "tag @0: kind=1"; the
leading 01 there is mislabeled (tag @0 is byte 0x00). The correct encoding of
kind=1 is 00 01 (tag @0, value 1). This spec uses 00 ++ varint(kind). (R1
erratum covers both the bare-byte and the §5.8 mislabel.)

Errata R2 — kind and corr are exempt from default-omission

RFC 0002 §6.3 would omit an enum field whose value equals case 0, and would
omit any field at its default. The frame class must always be explicit on the
wire (the wire routes by it, §6.3 of RFC 0003), and §4.4.1's "Canonical
encoding" list emits kind unconditionally and corr unconditionally.
Resolution: kind (@0) and corr (@1) are always emitted, overriding
the generic §6.3 default-omission for these two header fields. stream/seq/
end do obey omission (absent / false ⇒ omitted), per §4.4.1 "Optional fields
are omitted." (corr is a uuid, which RFC 0002 §6.3 already lists as
"no implicit default, always emit," so its always-emit is consistent; the
exception is only really needed for kind.)

Errata R3 — header/body boundary; seq requires stream; end requires stream

§4.4.1 step 6 says "any tag > @4 begins the frame body," but §4.6 and RFC 0002
§6.2 restart body tags at @1. A monotonic single tag space across header and
body is impossible (body @1 < header @4). Resolution:

  • The header is the leading run of fields with tags in {@0, @1, @2, @3, @4}
    in ascending order; the body is the remainder, an opaque hcpbin blob whose
    own tags restart at @1 (RFC 0002 §6.2, §4.6). The decoder stops reading header
    fields after it has consumed the @0..@4 prefix; everything after is body bytes
    it does not parse.
  • The decoder distinguishes "header @1 (corr)" from "body @1" by position:
    corr is the first field after kind and is mandatory, so exactly one @1 is a
    header field (the one immediately after @0). After the optional @2/@3/@4
    prefix, the next field (whatever its tag, including a second @1) is body.
  • seq (@3) MUST NOT appear unless stream (@2) is present (a per-stream
    sequence number with no stream is meaningless). end (@4) MUST NOT appear
    unless stream (@2) is present (§4.4.1 step 5 normative rule: "@4 is INVALID if
    @2 is absent"). Violations ⇒ FrameError::HeaderViolation.
  • Out-of-order header tags (e.g. @4 before @2) or duplicate header tags are
    framing violations (§4.4.1 step 5, RFC 0002 §6.2 rule 1).

Worked header golden vectors (byte-exact, normative for S2)

Let CORR = 11 22 33 44 55 66 77 88 99 aa bb cc dd ee ff 00 (16 bytes).

G-H1 — single-shot request (kind=0, corr, no stream/seq/end):

00 00                                  # @0 kind = varint(0)
01 11 22 33 44 55 66 77 88 99 aa bb cc dd ee ff 00   # @1 corr (16 raw bytes)

header-bytes length = 2 + 17 = 19 octets.

G-H2 — multi-frame event, final chunk (kind=2 event, stream=5, seq=9, end):

00 02                                  # @0 kind = varint(2)
01 <CORR 16 bytes>                     # @1 corr
02 05                                  # @2 stream = varint(5)
03 09                                  # @3 seq = varint(9)
04 01                                  # @4 end = true (0x01)

G-F1 — full frame record (G-H1 header + RFC 0002 §10 body), reproducing
RFC 0003 §4.6:

<len>                                  # varint(19 + 13) = varint(32) = 0x20
00 00                                  # header: @0 kind=0
01 <CORR 16 bytes>                     # header: @1 corr
01 07 66 73 2e 72 65 61 64             # body: @1 string len 7 "fs.read"
02 03 6b 65 79                         # body: @2 bytes len 3 0x6b 0x65 0x79

frame-bytes = 19 (header) + 13 (body) = 32 octets ⇒ length = 0x20.
Total record = 1 + 32 = 33 octets. (Body bytes are the RFC 0002 §10 vector,
carried opaque.)

These three vectors are committed as the canonical fixtures; the test suite
encodes to them and decodes from them.

Algorithm / design

In-memory model (frame.rs)

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum FrameKind { Request = 0, Response = 1, Event = 2, Control = 3, Error = 4 }

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct Frame {
    pub kind: FrameKind,         // @0, always present (errata R2)
    pub corr: [u8; 16],          // @1, always present (uuid, 16 raw bytes)
    pub stream: Option<u64>,     // @2
    pub seq: Option<u64>,        // @3 (only valid when stream is Some)
    pub end: bool,               // @4 (only true when stream is Some)
    pub body: Vec<u8>,           // opaque hcpbin body bytes (NOT parsed by S2)
}

body is the bytes after the header in frame-bytes; the codec never inspects
or validates it (RFC 0003 §4.1, §4.4). Whether the body is canonical hcpbin is a
frame-level concern (RFC 0003 §9.2) owned by a later sprint; S2 round-trips it
verbatim.

Header codec (header.rs)

corr is the gap WP3-S1 leaves: hcpbin has no uuid type (confirmed —
hcpbin/src/lib.rs implements only varint/bool/bytes/hash; uuid is a 16-byte raw
field per RFC 0003 §4.4.1 "16 raw bytes … not length-prefixed"). RFC 0002 §6.2/
§6.3 tagged-record + default-omission are also explicit TODOs in lib.rs (lines
494–495). Decision: hand-roll a bespoke 5-field header codec in litany using
hcpbin's Reader/Writer primitives, rather than extend hcpbin. Rationale
(Simplicity First): the header is a fixed 5-field shape with non-generic rules
(R1–R3), the only consumer is the wire, and leaving the general record/uuid codec
to whichever sprint owns hcp.wire bodies keeps S2 surgical and keeps hcpbin's
scope (RFC 0002 §6) clean.

encode_header(&Frame, &mut hcpbin::Writer):
1. w.put_uvarint(0); w.put_uvarint(kind as u64); // @0, always
2. w.put_uvarint(1); w.put_raw(&corr); // @1, always, 16 raw bytes
3. if stream Some: w.put_uvarint(2); w.put_uvarint(stream);
4. if stream Some && seq Some: w.put_uvarint(3); w.put_uvarint(seq);
5. if end: w.put_uvarint(4); w.put_bool(true); // bool true = 0x01

Encoder invariants asserted before emit (debug + release): if seq.is_some() or
end then stream.is_some() (R3); kind numeric in 0..=4. These are the
"wire layer guarantees" of §4.4.1; a Frame that violates them is a programmer
error, not decodable input — return FrameError::HeaderViolation from a checked
encode entry point rather than panicking.

decode_header(&mut hcpbin::Reader) -> Result<Frame-without-body, FrameError>:
1. Read tag (uvarint); MUST be 0 ⇒ else HeaderViolation. Read kind (uvarint),
MUST be ≤ 4 ⇒ else KindOutOfRange (a framing violation, §4.4.1 step 1).
2. Read tag; MUST be 1 ⇒ else HeaderViolation. Read 16 raw bytes into corr
(EOF ⇒ UnexpectedEof).
3. Peek next tag. If 2: read stream (uvarint). If absent, stream = None and
steps 4–5 must not see @3/@4.
4. If next tag 3: require stream present (R3) else HeaderViolation; read seq.
5. If next tag 4: require stream present (R3) else HeaderViolation; read end
(hcpbin get_bool, which already rejects non-0x00/0x01, §6.4).
6. Enforce ascending order: a tag ≤ the last header tag read ⇒ HeaderViolation
(duplicate / out-of-order, §4.4.1 step 5). Once a tag > 4 (or a second @1, or
anything not continuing the @2→@3→@4 run) is seen, the header is finished and
that field begins the body.
7. Remaining Reader bytes are the opaque body.

Header decode is driven inside the frame-bytes slice the record layer already
delimited, so "remaining bytes = body" is well-defined.

Record codec (record.rs)

encode_frame(&Frame, max_frame) -> Result<Vec<u8>, FrameError>:
1. Encode header + append body into a scratch Writer ⇒ frame-bytes.
2. if frame_bytes.len() == 0 || frame_bytes.len() as u64 > max_frame { Err(FrameTooLarge) }
(length 0 illegal §4.1; > max_frame §4.3).
3. Prepend varint(frame_bytes.len()) ⇒ record.

decode_frame(&[u8], max_frame) -> Result<(Frame, usize), FrameError>:
1. len = get_uvarint(input) — strict/minimal (hcpbin already enforces non-minimal
and overlong rejection, §6.1; reused verbatim per §4.2). On NonMinimal/
Overlong ⇒ map to FramingViolation.
2. Overlong-by-construction bound (§4.2): reject any length varint encoded
in more bytes than the minimal width of max_frame, even if numerically valid
— the canonical-width guard. Compute minimal_width(max_frame) once.
3. if len == 0 ⇒ FramingViolation (§4.1).
4. if len > max_frame ⇒ FrameTooLarge — without reading the body (§4.3).
5. Slice exactly len octets as frame-bytes; short input ⇒ UnexpectedEof
(mid-frame EOF is a framing violation, §4.5).
6. decode_header over the slice; the rest is body.
7. Return (Frame, 1?+len) = length-prefix-bytes + len.

Frame-stream helpers (the round-trip target of §4, §10):

pub fn encode_frames(frames: &[Frame], max_frame: u64) -> Result<Vec<u8>, FrameError>
pub fn decode_frames(bytes: &[u8], max_frame: u64) -> Result<Vec<Frame>, FrameError>

decode_frames loops decode_frame, advancing by consumed, until the input is
exhausted; back-to-back with no padding (§4.1). A trailing partial record ⇒
UnexpectedEof. encode_frames concatenates encode_frame outputs. The pair is
the canonical-framing fixed point of §4 / §10.

Error model (lib.rs)

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum FrameError {
    UnexpectedEof,                  // mid-frame / mid-header EOF (§4.5)
    FramingViolation,               // non-minimal/overlong/zero length (§4.1, §4.2, §4.5)
    FrameTooLarge { len: u64, max: u64 }, // length > max_frame (§4.3)
    KindOutOfRange(u64),            // kind not in 0..=4 (§4.4.1 step 1)
    HeaderViolation,                // out-of-order/dup tag, seq/end without stream (R3, §4.4.1)
    Codec(hcpbin::DecodeError),     // wrapped primitive-decode failure
}

Every variant maps to a §4.5 "framing violation": the record envelope or header
could not be trusted, so (in the live wire, later sprints) the connection would
abort with no resync (§4.5). S2 only returns the typed value; it performs no
I/O and tears nothing down. This is the clean boundary the spec keeps: S2 owns
classification, the connection runtime (S3+) owns reaction (abort vs orderly
close, §9.1/§9.1a).

Explicitly NOT in S2's error model: the frame-level error Votive Frame
(RFC 0003 §9.2) and non-canonical-body rejection (§9.2, RFC 0002 §6.8). The body
is opaque here; body canonicality is a later sprint that decodes bodies.

Test plan

All tests are cargo test in protocol/hcp/litany/, fully runnable on the M1
dev machine (pure functions, zero I/O, no Linux dependency). Follow the
adversarial-independence convention of hcpbin/tests/golden.rs: expectations are
derived only from the RFCs (and the errata resolutions above), written against
the public API, citing the exact RFC clause.

Local (M1, cargo test) — the entire deliverable

  1. Golden header (golden_header.rs) — encode produces G-H1, G-H2 byte-exact;
    decode of G-H1/G-H2 yields the expected Frame fields. Cite §4.4.1.
  2. Golden record (golden_record.rs) — G-F1 byte-exact, reproducing RFC 0003
    §4.6 (length = 0x20, header + opaque §10 body). Decode recovers the frame and
    the verbatim body bytes. Cite §4.1, §4.6.
  3. Round-trip fixed point (roundtrip.rs) —
  4. decode_frame(encode_frame(f)) == f for a matrix of frames: every
    FrameKind; corr all-zero / all-0xff / mixed; stream None vs Some; seq None
    vs Some(0)/Some(large); end true/false; body empty / small / spanning a
    multi-byte length prefix (≥128 octets, forcing a 2-byte length).
  5. encode_frame(decode_frame(bytes)) == bytes for the golden byte fixtures
    (the §6.8 re-encode fixed point applied to framing).
  6. decode_frames(encode_frames(v)) == v for multi-frame streams (back-to-back,
    §4.1) including interleaved kinds and a stream id retired then a fresh frame.
  7. Violations (violations.rs) — each rejected with the right FrameError:
  8. length == 0 ⇒ FramingViolation (§4.1).
  9. non-minimal length (0x80 0x00, i.e. 0 overlong) ⇒ FramingViolation
    (§4.2; mirrors hcpbin uvarint_strict_reject_overlong).
  10. length overlong-by-construction (more bytes than minimal width of
    max_frame) ⇒ FramingViolation (§4.2 width bound).
  11. length > max_frame ⇒ FrameTooLarge, asserting the body slice is not
    read (e.g. truncated input after the length still errors as too-large, not
    EOF) (§4.3).
  12. mid-frame EOF (input shorter than declared length) ⇒ UnexpectedEof (§4.5).
  13. kind tag missing / not @0 ⇒ HeaderViolation; kind value 5 ⇒
    KindOutOfRange (§4.4.1 step 1).
  14. missing corr / short corr (<16 bytes) ⇒ UnexpectedEof/HeaderViolation.
  15. out-of-order header tags (@2 then @1-as-header, @4 before @2) ⇒
    HeaderViolation (§4.4.1 step 5).
  16. duplicate header tag (two @2) ⇒ HeaderViolation.
  17. seq (@3) without stream (@2) ⇒ HeaderViolation (R3).
  18. end (@4) without stream (@2) ⇒ HeaderViolation (§4.4.1 step 5 / R3).
  19. end byte 0x02 ⇒ Codec(InvalidBool(0x02)) (hcpbin §6.4, reused).
  20. No-resync assertion — a frame-stream with a corrupt record in the middle:
    decode_frames errors at that record and does not attempt to scan forward
    (assert it returns the error rather than recovering later frames) (§4.5).
  21. zig build is N/A for S2. No Zig in this sprint. Cross-language Zig⇄BEAM
    round-trip is RFC 0002 §7.2, a later concern; do not add a Zig target here.
    (The orchestrator's zig build -Dtarget=x86_64-linux note applies to other
    work packages, not S2.)

Requires dev-cx53 / Linux — NONE of the deliverable; only the perf gate

  • Perf gate (≤10µs/frame encode+decode, kickoff success criterion): write a
    cargo bench (or a #[test]-gated timing harness) for encode_frame /
    decode_frame over the golden frames. It runs on M1 for sanity / regression
    signal    , but the ≤10µs threshold is defined against dev-cx53 (8-core, no-
    alloc hot path)     and is only gated there. S2 lands the benchmark and an M1
    baseline number; the threshold assertion is deferred to WP3-S6 on dev-cx53.
    (dev-cx53 is off-limits for this planning window; the bench is written now and
    the gate is run later — no S2 acceptance criterion depends on dev-cx53.)
  • Everything else in RFC 0003 that would need Linux (vsock binding §12.3,
    namespace-isolated transport) is OUT of S2 (see scope table); S2 has zero
    Linux-only verification.

Dependencies on other sprints

  • WP3-S1 (protocol/hcp/hcpbin) — hard dependency, already landed. S2 uses
    hcpbin::{Reader, Writer} (put_uvarint/get_uvarint, put_bool/get_bool
    with strict §6.4, put_raw/get_raw) and the strict varint decode (non-
    minimal/overlong rejection) that §4.2 mandates the length prefix reuse. S2 must
    not duplicate varint logic. Gaps S1 leaves that S2 fills locally (bespoke):
    the uuid 16-raw-byte field and the tagged-header record shape (S1 leaves
    records/§6.2/§6.3 as TODO).
  • WP3-S2.5 / S3 (downstream) — the live connection: handshake (§5),
    chapter/stream lifecycle (§6), the frame-level error frame and body
    canonicality (§9.2). They consume S2's Frame/encode_frame/decode_frame.
  • WP3-S5 (eventd) — admits canonical frame-bytes to the hash chain (§10);
    S2 produces exactly the delimited frame-bytes it will hash (§4.1, §10).
  • Blocked external input: the hcp.wire schema digest (Appendix A, "value
    TBD") and the RFC 0003 §11 conformance vectors (Open question 1) — needed for
    handshake conformance, not for S2. Do not invent a digest.

Workspace decision (flag for the pre-start gate): there is currently no
workspace Cargo.toml (the kickoff pre-start gate "Cargo.toml workspace wired
into dev shell" is unchecked, and find -name Cargo.toml finds only the hcpbin
crate). S2 ships litany as a standalone crate with a path-dependency on
../hcpbin (mirroring hcpbin's standalone setup) so it is buildable today
without blocking on the workspace. When the workspace lands, both crates become
members; the path-dep is forward-compatible with a workspace member. This is a
deliberate Simplicity-First choice to avoid coupling S2 to an unchecked gate.

Acceptance criteria

  1. protocol/hcp/litany/ exists with the layout above; cargo build and
    cargo test pass on the M1 dev machine with zero I/O and zero Linux
    dependency.
  2. encode_frame reproduces G-H1, G-H2, and G-F1 (the §4.6 record)
    byte-for-byte; decode_frame recovers the exact Frame and the verbatim
    opaque body.
  3. Round-trip fixed point holds: decode(encode(f)) == f over the test matrix,
    and encode(decode(bytes)) == bytes over the golden fixtures (§6.8 / §4
    canonicality). decode_frames(encode_frames(v)) == v for multi-frame streams.
  4. Every §4.5 + §4.4.1 violation in the test plan is rejected with the specified
    typed FrameError; no mid-stream resync is attempted (assertion #5 in the
    test plan).
  5. length reuses hcpbin's strict varint (no duplicated varint code); the
    max_frame guard rejects oversize lengths without reading the body; the
    overlong-by-construction width bound is enforced.
  6. The three RFC 0003 errata (R1 fully-tagged kind, R2 kind/corr always
    emitted, R3 header/body boundary + seq/end require stream) are encoded in
    the codec and pinned by golden vectors, and recorded in the spec for fold-back
    into RFC 0003 before it leaves Draft.
  7. A perf benchmark for encode_frame/decode_frame exists and reports an M1
    baseline; the ≤10µs/frame threshold is explicitly deferred to WP3-S6 on
    dev-cx53 (no S2 criterion depends on dev-cx53).
  8. Test files are adversarially independent (expectations from RFCs + errata,
    each citing its clause), matching the hcpbin/tests/golden.rs convention.

Risks / open items

  • RFC 0003 §4.4.1 inconsistencies (R1–R3) are resolved here by spec fiat
    ranking prose over example annotations. If the RFC authors resolve them
    differently when Draft closes, the golden vectors and codec must follow the RFC;
    the resolutions are isolated in header.rs + golden_header.rs to make that a
    contained change.
  • hcp.wire digest / §11 conformance vectors are TBD. S2 deliberately scopes
    below them; handshake conformance cannot be tested until Open question 1
    closes. This is a known, accepted gap, not a defect.
  • Body opacity. S2 never validates body canonicality (§9.2 / RFC 0002 §6.8);
    a non-canonical body round-trips verbatim through S2. That is correct for the
    wire framing layer (the body is opaque, §4.1) and is the later sprint's concern.
