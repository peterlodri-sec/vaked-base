0. Read first — language and citation notes

This daemon is Erlang/OTP, not Zig. The runtime roster
(docs/runtime/README.md) lists agent-supervisord | Erlang/OTP | process | "Control plane: supervision trees, lifecycle, restart strategy, orchestration of the Zig daemons", and the daemon design (docs/superpowers/specs/2026-06-12-agent-supervisord-design.md)
opens with "One BEAM application per host". Engineer B's "Zig/eBPF/Linux"
descriptor in the WP4 kickoff is the person's profile across the whole work
package; it is not this daemon's implementation language. The Zig daemons
(sandboxd, the eventd Zig port, mcp-brokerd) are what agent-supervisord
supervises and orchestrates — they are not this sprint. Do not propose a Zig
rewrite of the supervisor.

Citation scope. The stale "RFC 0002 §4 / Appendix A" wording that appears in
the WP3 kickoff concerns hcpbin byte golden-vectors — that is wire-protocol
(WP3) territory and is irrelevant to a supervisor spec. agent-supervisord has no
hcpbin golden-vector dependency. Its oracles are (a) the eventd Python oracle's
frozen exit-code contract and (b) the lowered OTP goldens under
vaked/examples/lowering/gen/otp/. The normative protocol citations below are
RFC 0004 (protocol/rfcs/0004-multi-agent-state-dependency.md) and RFC 0005
(protocol/rfcs/0005-control-frames.md), cited by section.

1. Objective

Stand up the hand-written BEAM application skeleton of agent-supervisord:
the top-level OTP application + supervisor that boots the machine-generated
runtime tree as one of its children, plus the three hand-written gen_server
processes the daemon design names. The deliverable of this sprint is the
application supervisor's own child spec list and restart strategy (the new,
hand-written supervision boundary), the oracle-port bridge that keeps byte
semantics in the Python oracle, and the control-file intake that drives the
ralph Control scheduling model — enough to boot, observe children alive, and
drive one pause → step → resume cycle observable on the eventd log.

This corresponds to Plan item 1 of the daemon design
(docs/superpowers/specs/2026-06-12-agent-supervisord-design.md §Plan: "Skeleton
BEAM app … application + the three gen_servers, oracle-port bridge, control-file
intake. (impl PR 1)") plus the explicit restart-strategy decision for the
hand-written tier.

2. Inputs / oracles

2.1 Normative inputs (RFCs)

  • RFC 0004 §6 / §6.1 (protocol/rfcs/0004-multi-agent-state-dependency.md:570)
    — the boot lifecycle STOPPED → BOOT_SCANNING → DEPENDENCY_VERIFYING → RUNNING | PAUSED(stale_dependency) is owned by agent-supervisord. S3
    must wire the transitions and the verify gate, but the verification itself is
    the oracle's coldstart verb. "operators cannot force-resume agents with stale
    dependencies" (§6) is a hard rule the control plane must honor.
  • RFC 0004 §7 / §7.1 (:685 / :698) — the supervisor owns the topology
    epoch counter     ("Epoch is supervisor-assigned, never caller-asserted"). S3
    holds the counter as control_plane state (single host, in-memory; bump
    logged write-ahead is a later concern once topology mutation is wired).
  • RFC 0004 §4 / §4.3 (:212 / :389) — GC floor / checkpoint liveness. S3
    does not compute floors; it shells to eventd floor. Listed as an input so
    the writer's append discipline is understood, not implemented here.
  • RFC 0005 §1 (:43) — the control-frame catalog and the ControlRefusal
    enum (denied=1, unknown_target=2, stale_epoch=3, not_paused=4, invalid_interval=5, invalid_rewind=6). S3 reproduces the semantics of
    pause/resume/slow/step (§2, §2.1, §2.2) via the control file; the wire frames
    themselves are out of scope (§1.1).
  • RFC 0005 §2.1 / §2.2 (:151 / :173) — graceful pause (in-flight
    completes, new work queues, no rollback) and non-transactional single-tick
    step. These are the ralph Control semantics the supervisor promotes.
  • RFC 0005 §3 (rewind, :261) / §4 (idempotency & logging, :271) — applied
    control actions are control_action eventd entries; the corr→outcome map is
    rebuilt from the log on supervisor restart    . §4 is the load-bearing fact for
    the restart-strategy decision in §3.2 (control-plane state is recoverable from
    the log    , not lost on crash).
  • RFC 0005 §6 (:437) — refusal taxonomy; refused actions are not logged
    (applied=false, no control_action entry). In S3 (file transport, no
    ControlAck channel) refusals are log-visible only as the absence of a
    control_action plus a daemon log line (design §"Control intake (reference)").

2.2 Code oracles (frozen contracts S3 reproduces, never re-derives)

  • eventd CLI (eventd/__main__.py:1-28) — the STABLE ORACLE CONTRACT.
    Exit codes are the API the BEAM port consumes:
  • 0 — ok / chain verified / dependencies verified ⇒ RUNNING
  • 2 — usage error
  • 3 — PAUSED(stale_dependency) — coldstart refused the RUNNING transition
  • 4 — tampered / malformed log — the audit spine is broken
  • 5 — writer refused — single-writer lock held elsewhere

Verbs S3 invokes over ports: verify <log> (boot precondition,
verify_on_boot), append <log> '<payload json>' (the single-writer funnel),
coldstart <log> <consumer-agent> (the §6 gate), state <log> [--at N]
(observability for the smoke test), floor <log> <producer> (reserved for the
later GC path). These verbs and exit codes are frozen — S3 must not fork them.
- Lowered OTP goldens — vaked/examples/lowering/gen/otp/operator_field_sup.erl
and vaked/examples/lowering/gen/otp/vaked_fiber_worker.erl (15→17-entry golden
manifest, #15 parity contract). The application supervisor boots a generated
<slug>_sup module as a child; S3 must treat these as read-only inputs and
never hand-edit them (the gen/ boundary, design §Shape). Generated strategy is
one_for_one, intensity => 3, period => 10 (operator_field_sup.erl:16).
- ralph Control model — tools/ralph/ralphcore.py (Control dataclass:
paused: bool, interval: float|None, step: bool; parse_control(dict)).
This is the proven scheduling model the supervisor promotes to control_plane
state. The control-file intake parses the same JSON shape.

2.3 Calibration target / example system

vaked/examples/agentfield-swe.vaked — the daily-use target. Its

gen/otp/agent_field_sup.erl (children transcriptMiner (fiber, config

3. Design

3.1 Process tree (what S3 builds)

S3 builds the hand-written supervisor that wraps the generated tree. The
generated tree is a child of the daemon, not the daemon itself (design §Shape):
the hand-written application supervises the machine-written supervisor, and the
hand/generated boundary is exactly the gen/ boundary.

agent_supervisord                (application; .app + application:start/2)
└── agent_supervisord_sup        (top-level supervisor — THE S3 deliverable)
    ├── [0] eventd_writer        (gen_server: serializes ALL appends via the
    │                             oracle port — the RFC 0004 single-writer funnel)
    ├── [1] control_plane        (gen_server: holds Control{paused,interval,step}
    │                             per target + the topology-epoch counter; intakes
    │                             control.json; emits control_action via writer)
    ├── [2] workflow_engine      (gen_server STUB in S3: registered, init/1 only;
    │                             body lands in impl PR 2)
    └── [3] agent_field_sup      (THE LOWERED TREE — generated module, started as a
        │                         child by start_link/0; never hand-edited)
        ├── worker 'transcriptMiner'  (vaked_fiber_worker — placeholder, heartbeats)
        └── worker 'fieldView'        (vaked_fiber_worker — placeholder, heartbeats)

3.2 The restart strategy for the top-level supervisor (the core S3 decision)

The generated tree's strategy is already fixed by Track C
(one_for_one, intensity => 3, period => 10,
operator_field_sup.erl:16). S3 must independently choose the top-level
agent_supervisord_sup strategy. The discriminating question (per the OTP
restart-coupling model): does any child cache another child's pid, or do
children reference each other by registered name and rebuild transient state from
the log?

Decision: one_for_one, intensity => 4, period => 10.

Rationale, child by child:

  • eventd_writer is the single-writer funnel. It holds no other child's pid:
    it talks to the Python oracle over a port and (later) holds the flock itself. If
    it crashes, restarting it alone re-opens the port and re-acquires the
    funnel; siblings keep their registered-name references and resume appending once
    it is back. No sibling state is invalidated by a writer restart.
  • control_plane holds Control{paused, interval, step} per target plus the
    epoch counter and the corr→outcome map. RFC 0005 §4 is the tie-breaker: that
    map is rebuilt from the log on supervisor restart — i.e. control-plane state
    is recoverable, not lost, on crash. So a control_plane crash does not require
    bouncing siblings; on restart it replays control_action entries (via
    eventd state) to rebuild its outcome map. one_for_one is correct.
  • workflow_engine (S3 stub) holds no cross-child pid; independent restart.
  • agent_field_sup is itself a supervisor. Its own children
    (vaked_fiber_worker placeholders) do not write eventd at all in v0
    (lowering design §"Worker contract": "No I/O, no eventd writes") — so a worker
    crash, or a restart of the whole generated tree, cannot corrupt the log or
    invalidate writer/control_plane state. one_for_one at the top means a crash of
    the generated supervisor restarts only the generated subtree.

Why not rest_for_one / one_for_all: those couple sibling lifecycles by
position. There is no shared transient pid-cache among these four children that a
crash would dangle (the cross-references are by registered name and the only
durable shared resource — the log — is reconstructable), so blanket sibling
restarts would only add restart amplification with no consistency benefit. This is
the same reasoning the lowering design used to reject rest_for_one for the
generated tree (2026-06-12-otp-supervision-lowering-design.md §"Strategy
mapping").

intensity => 4 (one more than the generated tree's 3): the top level tolerates
one extra restart in the window because a writer-port flap (oracle subprocess
restart) is an expected, recoverable event during the reference phase; period => 10 matches the generated tier for consistency.

Child specs (uniform with the generated tier where it makes sense):

Child restart shutdown type Notes
eventd_writer permanent 5000 worker start order [0]: must be up before any writer; shutdown last so in-flight control_action/step events still land during graceful shutdown
control_plane permanent 5000 worker start [1]; depends on writer being registered
workflow_engine permanent 5000 worker start [2]; stub in S3
agent_field_sup permanent infinity supervisor start [3], last; shutdown => infinity is the OTP rule for supervisor children (let it shut its own tree down)

Child start order is load-bearing, not cosmetic: OTP starts children
left-to-right and shuts them down right-to-left. eventd_writer first/last means
the writer is available before any sibling tries to append and is the last to go
during a graceful stop, so control actions emitted while the tree drains still
reach the log.

3.3 The oracle-port bridge (eventd_port + eventd_writer)

The single-writer discipline (RFC 0004) is enforced by funneling every append
through eventd_writer, which calls the Python oracle over an Erlang port. During
the reference phase the oracle takes the flock per append (design §Shape: "the
oracle takes the flock per append"); the BEAM-native flock holder is a later PR.

eventd_port responsibilities:
1. Spawn python3 -m eventd <verb> <args> via erlang:open_port/2
({spawn_executable, …}), capture stdout/stderr and the exit status.
2. Map the frozen exit codes (§2.2) to typed Erlang results:
0 → {ok, …}, 2 → {error, usage}, 3 → {paused, stale_dependency},
4 → {error, tampered}, 5 → {error, writer_locked}. The mapping table is
the contract eventd_port_SUITE.erl pins.
3. Surface {error, tampered} from verify as a hard boot failure
(verify_on_boot, design §Purpose.2) — the application refuses to start a tree
over a broken audit spine.

eventd_writer API (S3): append(Payload) -> {ok, Seq} | {error, Reason},
serialized through the gen_server mailbox so concurrent callers cannot race the
single writer. control_plane emits control_action entries exclusively through
this call.

3.4 Control-plane intake and the lifecycle gate

  • Transport (S3): control.json polled per tick (design §"Control intake
    (reference)"), parsed into the ralph Control shape
    (paused/interval/step). RFC 0005 wire frames replace the file later; both
    funnel into the same control_plane state. File-driven actions synthesize
    their corr     and refusals are log-visible only (no ControlAck channel until
    the wire).
  • Verbs (S3, semantics from RFC 0005 §2): pause/resume/slow/step
    mutate control_plane scheduling state only — never the log, never worker
    state. step on a non-paused target is a no-op refusal (not_paused, §2,
    §6) and is not logged. An applied pause/resume/slow/step writes one
    control_action entry via eventd_writer (RFC 0005 §4: "applied control
    actions are events").
  • Lifecycle gate (RFC 0004 §6): the boot transition for each worker is
    STOPPED → BOOT_SCANNING → DEPENDENCY_VERIFYING → RUNNING | PAUSED(stale_dependency). In S3 the workers are placeholders with no real
    dependencies, so DEPENDENCY_VERIFYING shells eventd coldstart <log> <agent>: exit 0 ⇒ permit RUNNING; exit 3 ⇒ hold PAUSED(stale_dependency).
    The transition machinery is wired in S3; the verification result comes from
    the oracle. The §6 hard rule (no force-resume past a stale dependency) is
    honored by routing every RUNNING transition through this gate.
  • Epoch counter (RFC 0004 §7): control_plane holds an in-memory u64 epoch,
    initialized at boot. S3 does not yet bump it (no topology-mutation path is
    wired); it exists so the later frame intake has a single owner. Caller-asserted
    epochs are never trusted (§7.1).
  • Authority (stub): no preceptord exists; S3 is allow-all + log the
    principal    , with the deny path stubbed but wired (design §Open). Every control
    action records the principal in the control_action payload so the later
    policy check has its input.

3.5 Rewind is explicitly deferred

The five-verb control model (control-plane design
2026-06-12-control-plane-design.md) includes rewind, which in RFC 0005 §3
composes RFC 0004 machinery (pause affected workers → append RewindEvent →
restart workers → cold-start re-anchors or parks them). S3 does not implement
rewind. It is the epic's acceptance-3 demo and lands once the writer + control
intake + cold-start gate (all S3) are in place plus the workflow/step path (PR2).
S3's job is to make the substrate rewind needs (writer funnel, restart of a
producer's worker via one_for_one, cold-start gate) exist and be observable.

4. Algorithm (boot + one control cycle)

application:start/2:
  1. read gen/eventd.json → log path L
  2. eventd_port: `python3 -m eventd verify L`
        exit 4 → crash the application (verify_on_boot precondition; tampered spine)
        exit 0 → continue
  3. start agent_supervisord_sup (strategy one_for_one, §3.2):
        a. eventd_writer  (opens the oracle port; ready to funnel appends)
        b. control_plane  (loads control.json shape; epoch := boot epoch)
        c. workflow_engine (stub)
        d. agent_field_sup:start_link/0   (the lowered generated tree)
             for each worker child:
               STOPPED → BOOT_SCANNING → DEPENDENCY_VERIFYING
               `eventd coldstart L <child>`:
                  exit 0 → RUNNING
                  exit 3 → PAUSED(stale_dependency)   (held; no force-resume)
  4. children alive; placeholders heartbeat (vaked_fiber_worker tick)

one pause → step → resume cycle (control.json poll tick):
  control.json {"paused": true}:
     control_plane sets target paused (graceful: in-flight completes, new queues)
     append control_action{verb=pause, target, corr=synth, principal} via eventd_writer
  control.json {"paused": true, "step": true}:
     if target paused: advance one scheduler tick, return to paused
        append control_action{verb=step, …}
     else: refuse not_paused (NO log entry)
  control.json {"paused": false}:
     control_plane resumes normal scheduling
     append control_action{verb=resume, …}

observe: `python3 -m eventd state L` shows the control_action entries (per-kind
counts), proving the cycle hit the log; refused not_paused step is absent.

5. Test plan

Two tiers. The split is governed by what each tier needs, and one fact decides
it: flake.nix (:21-43) provides erlang and elixir in the devshell for
all four systems including aarch64-darwin. So BEAM compilation and boot are
runnable on the M1 host via nix develop — they are not dev-cx53-only. The
only things genuinely needing dev-cx53/Linux are the real Zig-daemon ports the
supervisor will later spawn (namespace/cgroup syscalls), which are out of S3
scope (§1.1).

5.1 Tier A — local on M1, no BEAM required (Python oracle contract)

These pin the byte/verification contracts S3 delegates to. They are the honest
"verifiable locally today" floor and need only python3:

  • A1 — boot precondition (verify_on_boot). On a tampered/truncated log,
    python3 -m eventd verify exits 4. Regression: the spec asserts S3's port
    must treat exit 4 as a hard boot failure. Verify: existing eventd tamper tests
    under tests/spec/ already produce exit 4; A1 references them as the oracle
    contract.
  • A2 — cold-start gate. python3 -m eventd coldstart <log> <consumer> exits
    0 (⇒ RUNNING) on a verified anchor and 3 (⇒ PAUSED(stale_dependency)) on a
    missing/stale anchor. This is the exact gate control_plane consults; the test
    fixes the two exit codes S3 branches on.
  • A3 — single-writer refusal. Concurrent eventd append while the flock is
    held exits 5; the port maps it to {error, writer_locked}.
  • A4 — lowering structure (already shipped). tests/spec/test_otp_lowering.py
    asserts vakedc lower emits gen/otp/agent_field_sup.erl +
    vaked_fiber_worker.erl with the right children/strategy and is byte-
    deterministic. S3 depends on this output; A4 is the existing guard that the tree
    S3 boots is well-formed. Run: task test (python3 -m pytest tests -q).

Command surface for Tier A: python3 -m pytest tests -q and direct
python3 -m eventd … invocations. No cargo, no zig, no erl needed.

5.2 Tier B — local on M1 inside the devshell (nix develop, BEAM present)

These need erlc/erl, which the devshell provides on aarch64-darwin. They run
on the M1 dev machine under nix develop; they do not require dev-cx53.

  • B1 — compile. rebar3 compile (or erlc src/*.erl) under the devshell:
    the application + four gen_server/supervisor modules compile clean. Module
    name == filename for every module (OTP rule).
  • B2 — boot + children alive (extends task otp-smoke). The existing
    otp-smoke target (Taskfile.yml:67-85) lowers agentfield-swe, erlcs the
    generated gen/otp/*.erl, and boots the generated <slug>_sup directly. S3
    extends this to boot the application (agent_supervisord wrapping the
    generated tree) and assert all four top-level children + the two placeholder
    workers are alive after one heartbeat window (≈6s). This is the design's "Shape
    smoke" (2026-06-12-agent-supervisord-design.md §Verification.1).
  • B3 — one control cycle. Drive pause → step → resume via control.json
    and assert via python3 -m eventd state that control_action entries appear
    for the three applied verbs, and that a step issued while not paused
    produces no entry (not_paused, RFC 0005 §6). This is the §4 algorithm's
    control cycle, end to end, observable on the log.
  • B4 — restart strategy assertion. Common Test (agent_supervisord_SUITE.erl):
    kill eventd_writer and assert (a) it restarts, (b) control_plane and the
    generated tree keep their pids (proving one_for_one, not one_for_all); then
    kill a vaked_fiber_worker and assert only that worker restarts. This is the
    test that proves the §3.2 decision; it FAILS under one_for_all/rest_for_one
    and PASSES under one_for_one (regression-test discipline: the ordering proves
    it catches the wrong strategy).
  • B5 — determinism of generated inputs. Two vakedc lower runs produce
    byte-identical gen/otp/*.erl (already an A4/lowering guarantee); B5 just
    confirms the boot is reproducible from identical inputs.

5.3 What needs dev-cx53 / Linux (and is OUT of S3 scope)

  • Spawning the real Zig daemons (sandboxd, eventd Zig port) as supervised
    ports requires Linux namespaces/cgroups (unshare, cgroup-v2) — WP4-S5/S6, not
    S3. S3's workers stay vaked_fiber_worker placeholders, so S3 has no Linux
    syscall dependency    .
  • The eventd Zig port's byte-parity suite (WP4-S4) runs against the same goldens
    but is a separate sprint.

The orchestrator's M1 toolchain note (Rust via cargo test, Zig via zig build -Dtarget=x86_64-linux compile-only) is satisfied: S3 introduces no Rust/Zig
code (it is an Erlang daemon), so neither cargo nor zig is on S3's critical
path; the BEAM tooling it does need is M1-local via the devshell.

6. CI posture

CI has no Erlang (the lowering design and test_otp_lowering.py docstring both
state "CI has no Erlang"). Therefore:
- CI gate = Tier A (python3 -m pytest tests -q) plus a structure/lint check
of the new daemons/agent-supervisord/src/*.erl (presence, module/filename
agreement, no hand-edits to gen/).
- Tier B is a devshell/PR gate, recorded in the landing PR exactly as
otp-smoke output is recorded today (design §Verification.3, lowering design
§Verification). The PR that lands S3 includes the task otp-smoke-extended
transcript (boot + control cycle + restart assertion) run under nix develop on
the author's machine.

8. Acceptance criteria

S3 is done when:

  1. Subtree exists at daemons/agent-supervisord/ with the §3.1 layout; a
    README.md mirrors the agent_guardd/ / eventd/ per-daemon convention and
    the roster row.
  2. Compiles clean under the devshell (rebar3 compile / erlc src/*.erl),
    module==filename for all modules (B1).
  3. Boots the generated tree as child [3] of agent_supervisord_sup; all
    four top-level children and both agentfield-swe placeholder workers are alive
    after a heartbeat window (B2). verify_on_boot hard-fails on a tampered log
    (A1 + the port mapping).
  4. Restart strategy is one_for_one at the top level with the §3.2 child
    specs/order, and B4 proves it: an eventd_writer kill restarts only the writer
    (siblings keep pids); a worker kill restarts only that worker. The test FAILS
    under one_for_all/rest_for_one.
  5. One control cycle lands on the log: pause → step → resume via
    control.json produces three control_action entries visible via eventd state, and a step while not paused produces none (not_paused) (B3).
  6. Cold-start gate is wired: a worker whose eventd coldstart returns exit 3
    is held PAUSED(stale_dependency) and is not force-resumable; exit 0 permits
    RUNNING (A2 + transition wiring).
  7. Byte semantics stay in the oracle: no canonical-hash or verify_cold_start
    reimplementation on the BEAM (grep the subtree; the only path to those facts is
    eventd_port).
  8. CI green on Tier A; the landing PR records the Tier B devshell transcript
    (§6).

9. Open questions (carried from the daemon design, not resolved in S3)

  • One daemon per host supervising many runtimes, vs one per runtime (design
    §Open — lean per-host with one <runtime>_sup child per runtime; S3's
    single-example layout does not force the choice).
  • Where the preceptord authority check physically runs once it exists (design
    §Open — S3 leaves the deny path stubbed).
  • Whether eventd_writer should later hold the flock BEAM-native vs keep the
    oracle taking it per append (design §Shape — S3 keeps the oracle holding it).
  • Budget enforcement point for workflow steps (engine-side vs worker-side) —
    deferred with the workflow engine (PR2).
