Spec — WP3-S5: eventd integration (Litany ↔ append-only hash-chained log)

Spec — WP3-S5: eventd integration (Litany ↔ append-only hash-chained log)

Date: 2026-06-14 · Status: spec (planning artifact; dependency-blocked, not codeable now) · Sprint: WP3-S5 (Aug 21–Sep 3, docs/superpowers/plans/2026-06-14-wp3-kickoff.md line 28) · Owner: WP3 Engineer A (Rust)

Stale-reference correction (read first). The WP3 kickoff and issue #167 cite "RFC 0002 §4 / Appendix A" for the byte encoding and golden vectors. That wording is wrong. In the frozen RFC 0002 (protocol/rfcs/0002-hcplang.md):
- the byte encoding (hcpbin) is §6 ("hcpbin encoding rules"), not §4;
- the frame header (kind/corr/stream/seq/end) is reserved and implicit (§4.2) and is encoded by the WIRE layer — its normative byte layout is RFC 0003 §4.4 / §4.4.1, explicitly NOT in hcpbin (RFC 0002 §4.2 and the §6 scope note at lines 446–449);
- golden vectors live in the worked-examples subsections §6.1.1, §6.3.1, §6.6.1, §6.7.1, and §10 — there is no encoding "Appendix A" (Appendix A of RFC 0002 is generated-code examples, §A.1/§A.2, not byte vectors).

Implement against §6 + RFC 0003 §4, derive vectors from §6.1.1/§6.3.1/§6.6.1/§6.7.1/§10. The existing hcpbin crate already encodes this correctly (protocol/hcp/hcpbin/src/lib.rs docstring lines 22–23 make exactly this point).

1. Objective

Make the Litany wire (RFC 0003) feed the eventd append-only, hash-chained audit log (the immutable leg of the 1.0 epic, docs/superpowers/specs/2026-06-12-eventd-design.md) so that:

  1. Every admitted Litany frame, plus every connection-lifecycle moment, becomes a durable, tamper-evident eventd entry (RFC 0003 §10, §11 item 9).
  2. The state-dependency control frames of RFC 0004 §2 (DependencyRegistration, ConsumerCheckpoint, RewindEvent) — which MUST be durably logged before consumption (RFC 0004 §3.1) — are written through this same path and fold into the existing DependencyIndex / GC-floor / cold-start machinery (eventd/statedep.py).
  3. The bridge is byte-faithful to the Python eventd oracle: the entry/chain bytes a Rust writer produces are accepted by python3 -m eventd verify and are byte-identical to a committed golden, exactly as vakedz is pinned to vakedc (vakedz/test/crossverify.sh).

This sprint is the integration glue between two already-specified layers, not a new encoding. It is dependency-blocked (see §7) and is delivered as a planning artifact now.

Non-goals

  • Not a new chain format. eventd/core.py is FROZEN (its module docstring, lines 17–37) and cross-verified against tools/ralph/ralphcore.py and vakedz/cache.zig. S5 must not alter it.
  • Not the Zig eventd daemon. That is the #15-pattern port, scheduled WP4-S4 ("Zig port (WP4-S4) replaces by Sep", kickoff line 47); the Python eventd is its oracle and ours.
  • Not RFC 0004's compiler passes (DAG validation = order 5, lives in vakedc) nor the zero-copy scan path (order 8). RFC 0004 §8.
  • Not perf tuning (WP3-S6) nor stress hardening (WP3-S7).

3. The central design decision (surface, do not silently pick)

There is a documented contradiction in the repo about what bytes eventd hashes:

  • RFC 0003 §10 + §4.6 + §11 item 9 state eventd chains H(prev ‖ frame-bytes) — over the raw hcpbin header+body the wire delimited ("the chain covers header and body together", line 1164; "eventd chains H(prev ‖ frame-bytes)", line 403).
  • eventd/core.py (FROZEN) + eventd/statedep.py note state eventd chains sha256(prev_hex ‖ canonical_json(payload)) over a JSON payload, and that "eventd payload bodies are JSON; the .hcplang frames in RFC 0004 §2 are the wire form, this is the logged form" (statedep.py lines 19–24).

Working decision for S5 (oracle-faithful): eventd's chain stays exactly sha256(prev_hex ‖ canonical_json(payload)) over a JSON logged form. The integration projects each admitted Litany frame into a JSON payload dict ({"kind": …, "v": …, …}), precisely as statedep.py/runtime.py already do. We do not redefine eventd to hash raw frame-bytes.

Rationale: the task names "Python eventd is the byte oracle"; core.py is FROZEN and triple-cross-verified (ralphcore, vakedz/cache.zig); and the statedep/runtime modules already commit to the JSON-logged-form model. Changing the chain to hash frame-bytes would (a) break the frozen oracle, (b) invalidate every existing eventd entry/test, and (c) make the chain unverifiable by python3 -m eventd verify. The litany→JSON projection is the bridge; the wire's frame-bytes is what is transmitted, the JSON logged form is what is chained.

Open reconciliation item OR-1 (must be filed, see §8): RFC 0003 §10 / §4.6 / §11.9 (chain over frame-bytes) vs eventd/core.py + statedep.py (chain over canonical_json(payload)). These are mutually exclusive byte contracts. One must be amended. S5 implements the oracle (JSON); the RFC text needs a corrective erratum or a deliberate decision to bump the eventd format (a breaking change that re-digests all chained frames — explicitly out of S5 scope). Until reconciled, S5 carries an optional frame_bytes_sha256 field inside the JSON payload (see §4.2) so the raw frame digest is still auditable without changing the chain function.

The Rust eventd_chain.rs is a faithful re-implementation of the FROZEN eventd/core.py (it is not allowed to drift; the parity gate enforces this). It exists so the wire crate can chain entries without a Python dependency at runtime, exactly as vakedz/src/cache.zig re-implements the same chain in Zig.

4.2 The projection: Litany frame → eventd logged form

A frame admitted by the wire (RFC 0003 §10 admission rule: full frame-bytes strictly canonical per RFC 0002 §6.8) is projected to a JSON payload dict that the eventd chain then hashes. The projection is total and deterministic (same frame ⇒ same payload ⇒ same chain bytes).

Header fields (RFC 0003 §4.4.1): the wire-layer header kind/corr/stream/seq/end map into payload keys:

Header field (§4.4.1) hcpbin wire form Logged-form JSON key JSON value
kind @0 LEB128 u8 0–4 "kind_class" one of "request","response","event","control","error" (the §4.4.1 enum 0–4 decoded to its name; kept distinct from the payload's own "kind" discriminator)
corr @1 16 raw UUID bytes (RFC 4122 big-endian) "corr" canonical lower-case hyphenated UUID string
stream @2? LEB128 u64 if present "stream" integer, omitted if absent
seq @3? LEB128 u64 if present "seq" integer, omitted if absent (note: this is the frame header seq, distinct from the eventd entry seq which make_entry assigns)
end @4 0x01 if true, omitted if false "end" boolean, omitted if false (mirrors the §4.4.1 / RFC 0002 §6.2 default-omission rule)

Omission must mirror hcpbin default-omission so that the JSON logged form carries exactly the information the canonical header did — no null/0 placeholders (RFC 0003 §4.4.1 "Optional fields are omitted, not encoded with null/zero markers").

Body fields: decoded from the negotiated schema (RFC 0003 §4.4 / §5.4 schema-digest negotiation) into their natural JSON scalars. The mapping is the RFC 0002 §7.1 codegen table read in reverse:
- u8…u64, i8…i64, enum → JSON integer (statedep/runtime already ban floats in these payloads, statedep.py _require_int).
- bool → JSON boolean.
- string → JSON string (NFC per RFC 0002 §6.4 was already applied at encode time; the logged form preserves it as written — eventd does no further normalization, core.py line 32).
- bytes, hash, uuid → JSON string in a fixed textual form (base64-std-no-pad for bytes; "<algo-hex>:<digest-hex>" for hash per RFC 0002 §6.5 registry; canonical UUID string for uuid). The textual form is part of OR-2 (see §8) and MUST be fixed before implementation so the bytes are deterministic.
- timestamp → JSON string (RFC 3339, UTC, fixed sub-second precision) — fixed under OR-2.
- list → JSON array (order preserved, RFC 0002 §6.6); map → JSON object (keys already sorted by canonical key bytes on the wire; the JSON encoder re-sorts by code point per core.py, so the two sorts must be reconciled — captured in OR-2).
- nested record/union → nested JSON object (union carries an arm-tag field).

Schema versioning: every projected payload carries "v": <schema version int> (the @since/version discipline, RFC 0002 §8) exactly as statedep.py/runtime.py set "v": STATEDEP_V / RUNTIME_V. The wire's negotiated schema digest (RFC 0003 §5.4) is recorded once per connection in the connection_open lifecycle entry (below), not per frame.

OR-1 bridge field: the projected payload MAY include "frame_bytes_sha256": "<hex>" — sha256 of the exact canonical frame-bytes the wire delimited (§4.1). This keeps the raw-frame digest auditable inside the JSON-hashed chain without changing the chain function. It is part of the logged form (so it is covered by the entry hash); whether it is required is gated on OR-1.

4.3 Redaction (@redact, RFC 0002 §4.4, normative)

A body field declared @redact MUST NOT have its plaintext value placed in the logged form. The projection replaces it with a fixed sentinel object {"redacted": true, "len": <n>} (length of the redacted value's canonical bytes, so size is auditable without content). This is applied before canonical_json, so the redaction is part of the hashed bytes — a downstream litanydump/eventd state never sees the plaintext, matching the §4.4 requirement that @redact fields are redacted "in eventd projections." A regression test (see §6) must prove plaintext never appears in the on-disk .jsonl.

4.4 Connection-lifecycle entries (RFC 0003 §10)

Beyond frames, the bridge logs the connection shape. These are eventd payloads with their own kind discriminator (new KIND_* in a litany_wire logged-form module mirroring statedep/runtime):

connection_open   {kind, v, conn, peer, wire_version, max_frame, schema_digests:[...]}
connection_close  {kind, v, conn, reason: "bye"|"goaway"|"half_close"}
transport_fault   {kind, v, conn, fault, detail}     # §9.1 abortive close / framing violation

A framing violation (RFC 0003 §4.5: non-minimal/overlong length, length > max_frame, length==0, un-parseable header, mid-frame EOF) is recorded only as a transport_fault connection-end entry and the connection is aborted — the offending bytes are never admitted as a frame (RFC 0003 §10, lines 1166–1169). A header-clean-but-non-canonical body is a frame-level error (§9.2): its frame-bytes are excluded from the chain, and the bridge logs a frame_rejected {kind, v, conn, corr, reason} entry rather than the frame itself.

4.5 Write path (single-writer discipline, mirror eventd/log.py)

The Rust eventd_writer.rs reproduces the EventLog discipline exactly (eventd/log.py lines 149–214):
1. Open as writer: take the exclusive flock(LOCK_EX | LOCK_NB) first, then load+verify the existing chain under the lock (the Codex P1 ordering fix, log.py lines 161–173). A second writer ⇒ refuse with the WriterLockError-equivalent (exit-code 5 contract).
2. Boot verify: verify_chain the loaded entries; a broken chain is a hard error (TamperError-equivalent, exit-code 4) — the writer is closed without ever holding the lock on a refused log (log.py lines 182–184).
3. Append: entry = make_entry(tail_hash, len, payload); write entry_line(entry) (json.dumps(entry, separators=(",",":"), sort_keys=True, ensure_ascii=False, allow_nan=False) + "\n"); flush; fsync before returning (log.py lines 205–214). The flush+fsync per append is the durability contract (eventd design "Daemon shape", spec lines 52–62).
4. Readers map the log read-only and fold (DependencyIndex.from_entries, RuntimeState.fold) — no change; the existing Python readers consume the Rust-written log unchanged, which is the whole point of byte parity.

The single-writer owner in the real runtime is agent-supervisord (eventd design line 55); in S5 the litany crate is the writer when it is the process appending. Multi-writer (per-fiber sub-logs) stays the eventd design's open question.

4.6 Fold integration (no new fold)

Because projected DependencyRegistration/ConsumerCheckpoint/RewindEvent/consumer_evicted payloads are byte-identical to what statedep.py constructors emit, they fold into the existing DependencyIndex with zero changes: GC floor (§4), cold-start verify (§6), eviction liveness (§4.2) all work as-is. S5's job is to make the wire the producer of those payloads (replacing hand-written python3 -m eventd append), and to enforce RFC 0004 §3.1 (registration logged before consumption) structurally at the wire boundary.

5. Worked projection example (grounding)

A RFC 0004 §2 DependencyRegistration control frame on the wire:

  • Header (RFC 0003 §4.4.1), single-shot control, no stream: kind=3 (control), corr=<uuid>. Frame-bytes = 03 01 <16 uuid bytes> <body...>.
  • Body (RFC 0002 §6.2, ascending tags): consumer uuid @1, producer uuid @2, consumer_step u64 @3, producer_step u64 @4, producer_step_hash hash @5, topology_epoch u64 @6.

Projects to the eventd logged form (identical to statedep.dependency_registration(...), statedep.py lines 64–73, with the header keys added):

{"consumer":"<uuid>","consumer_step":4,"corr":"<uuid>","kind":"dependency_registration","kind_class":"control","producer":"<uuid>","producer_step":7,"producer_step_hash":"01:<64hex>","topology_epoch":2,"v":1}

(keys shown unsorted for readability; canonical_json sorts them by code point before hashing). The chain entry is then {"seq":N,"prev":"<64hex>","payload":<above>,"hash":sha256("<prev>"+canonical_json(payload))}, one LF-terminated line. python3 -m eventd verify over the resulting file MUST exit 0, and python3 -m eventd coldstart <consumer> MUST behave identically to a log written by the Python constructor.

6. Test plan

Success criterion: the Rust bridge's on-disk chain is byte-identical to, and verifiable by, the Python eventd oracle, fully on M1. The kickoff's "All tests pass on dev-cx53" line (kickoff line 43) is corrected below — dev-cx53 is off-limits and not required for the verifiable core.

6.1 M1-local (primary gate — everything below runs on macOS M1)

Verified available in this environment: cargo 1.95.0 (cargo test → 50 hcpbin tests pass), python3 -m eventd imports and runs. eventd uses fcntl.flock + os.fsync, both functional on macOS, so the entire eventd parity surface is M1-verifiable.

  1. Rust unit / golden (cargo test) — tests/eventd_golden.rs:
  2. chain primitive parity: GENESIS, canonical_json, chain_hash, make_entry, entry_line reproduce the FROZEN eventd/core.py bytes on shared vectors (the same vectors tests/spec/test_eventd.py group 1 uses against ralphcore).
  3. projection golden: each RFC 0004 §2 frame and each lifecycle event projects to the exact JSON logged form committed under protocol/hcp/litany/test/golden/.
  4. canonical-JSON edge cases: non-ASCII passthrough (ensure_ascii=False), code-point key sort, integer-only step fields, NaN/Inf rejection (allow_nan=False), no-Unicode-normalization ("é" vs "é" distinct — core.py line 32).
  5. Python spec test (python3 -m pytest tests) — tests/spec/test_litany_eventd.py: generate the golden chains from the eventd/* constructors (the oracle), assert the committed goldens equal them (drift guard), and assert structural fold equivalence (DependencyIndex/RuntimeState over Rust-written vs Python-written logs are equal).
  6. Cross-verify parity gate — protocol/hcp/litany/test/crossverify-eventd.sh (mirror of vakedz/test/crossverify.sh):
  7. Rust bridge writes a chain → python3 -m eventd verify exits 0; diff against the committed golden is empty.
  8. tamper test: flip one byte in a Rust-written log → python3 -m eventd verify exits 4 (TAMPERED) — the boot-time hard-error contract (eventd/__main__.py line 43, log.py lines 177–184).
  9. single-writer test: two Rust writers on one path → second refuses (exit 5 contract).
  10. stale-dependency test: Rust writes a registration then a rewind that voids it → python3 -m eventd coldstart <consumer> exits 3 (PAUSED) — RFC 0004 §3.3/§6.
  11. redaction regression (§4.3): a @redact body field's plaintext MUST NOT appear anywhere in the on-disk .jsonl (grep the file). This test must fail without the redaction code and pass with it (CLAUDE.md §5).
  12. determinism: write the same frame sequence twice → byte-identical files (entry_line determinism, core.py lines 83–89).
  13. repair --truncate-tail: append a torn tail to a Rust-written log → python3 -m eventd repair --truncate-tail drops the unverifiable suffix and logs log_repair (log.py lines 96–146).
  14. Task / CI wiring: task litany-eventd-verify runs (3); add it to a CI workflow on the M1 runner alongside task spec (Taskfile.yml line 111: python3 -m pytest tests -q) and task vakedz-verify.

7. Dependencies on other sprints

S5 is dependency-blocked today; this is why it is a planning artifact.

  1. WP3-S3 (litany crate exists). protocol/hcp/litany/ is created in S3 (kickoff line 14). S5 adds modules to it. Blocking.
  2. hcpbin type coverage (WP3-S1/S2/S3). The RFC 0004 §2 frames use uuid, timestamp, u64?, hash, string, records, optionals. The hcpbin crate today implements only varints / bool / bytes / hash — its own TODOs defer string/NFC (WP3-S2), f32/f64, records/frames/unions/maps/enums, and the frame header (src/lib.rs lines 18–23, 488–498); uuid and timestamp are not present at all. A full frame→logged-form projection is blocked on S1/S2/S3 landing those types and the RFC 0003 §4.4 header codec. Until then S5 can only project the subset already encodable. Blocking (partial).
  3. WP3-S2 (frame layer / round-trip suite) and WP3-S4 (control frames, RFC 0005) establish the frame model and control-frame admission S5 hooks into. Blocking for the full set.
  4. eventd reference (DONE). eventd/{core,log,statedep,runtime}.py + CLI + tests/spec/test_eventd.py already exist and are the oracle. Satisfied.
  5. RFC 0002 frozen (gate, by Jun 21). Precondition for all hcpbin work (kickoff line 48). Assumed satisfied.
  6. Provides → WP4-S4 (Zig eventd daemon). S5's committed goldens and parity gate become the oracle the Zig port is held to (the #15 pattern, kickoff line 47).

8. Open reconciliation items (must be filed as issues before implementation)

  • OR-1 — chain-byte contradiction. RFC 0003 §10 / §4.6 / §11 item 9 (chain over frame-bytes) vs eventd/core.py FROZEN + statedep.py note (chain over canonical_json(payload)). S5 implements the oracle (JSON logged form, §3). RFC 0003 needs a corrective erratum, or a deliberate format bump (breaking; re-digests all chained frames; out of S5 scope). Until resolved, the optional frame_bytes_sha256 payload field (§4.2) preserves the raw-frame digest inside the JSON chain.
  • OR-2 — non-scalar textual encodings. The deterministic JSON text form of bytes (base64 variant), hash (algo:digest hex), uuid (canonical string), timestamp (RFC 3339 precision), and map (key-sort reconciliation: hcpbin sorts by canonical key bytes per RFC 0002 §6.6, eventd's canonical_json sorts JSON object keys by code point — for string-keyed maps these can differ) MUST be pinned before coding, because they enter the hashed bytes. File and freeze.
  • OR-3 — kind_class vs kind collision. The projection adds a header-derived kind_class key alongside the payload's existing kind discriminator. Confirm no existing logged-form payload uses kind_class/corr/stream/end/seq as a body field name (none in statedep/runtime today); reserve these header-projection keys.
  • OR-4 — RFC 0004 epoch-sync provisional sections (§7.1, §4.2.1, §4.2.2, §3.3.x are marked "provisional"). S5 logs what the wire emits; the epoch-assignment authority (supervision plane, §7 / Security considerations) is a preceptord/supervisord concern, not S5's. Confirm the logged topology_epoch is supplied, never self-reported by S5.

9. Acceptance criteria

S5 is complete when, all verified on M1:

  1. cargo test in protocol/hcp/litany/ passes, including the eventd chain-parity, projection-golden, and canonical-JSON edge-case tests (§6.1 item 1).
  2. python3 -m pytest tests passes, including tests/spec/test_litany_eventd.py (§6.1 item 2).
  3. protocol/hcp/litany/test/crossverify-eventd.sh (and task litany-eventd-verify) passes: a Rust-written chain is accepted by python3 -m eventd verify (exit 0) and is byte-identical to the committed golden; tamper→exit 4; double-writer→exit 5; voided anchor→coldstart exit 3 (§6.1 item 3).
  4. A @redact field's plaintext never appears in the on-disk .jsonl (regression test fails without the redaction code, passes with it).
  5. The same frame sequence written twice yields byte-identical logs (determinism).
  6. The eventd chain function is unchanged from eventd/core.py (FROZEN): git diff shows no edit to eventd/core.py; the Rust eventd_chain.rs is proven byte-identical to it, not a re-spec.
  7. OR-1…OR-4 are filed as issues with the working decisions of §3/§8 recorded.

These are strong, looping criteria (CLAUDE.md §4): each is a runnable command with a defined pass/fail, so the implementer can iterate without re-clarification.
