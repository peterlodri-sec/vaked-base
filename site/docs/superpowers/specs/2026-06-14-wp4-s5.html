WP4-S5 — Capability enforcement hooks (all 3 daemons)

0. Scope note — which three daemons (read first)

A reader arriving from the capability lattices in the codebase will reasonably
assume "the 3 daemons" means the trio that already enforces capabilities today —
agent_guardd (network/ebpf), agent_sandboxd (fs/process), agent_memoryd
(mem). That is the wrong set for this sprint. The WP4 kickoff fixes the
daemon set exactly once and reuses it:

  • Scope (docs/superpowers/plans/2026-06-14-wp4-kickoff.md line 7): "Three
    daemons to MVP: sandboxd + agent-supervisord + eventd (Zig port)."
  • WP4-S6 (line 33): "integration test (3 daemons running on vakedos)."
  • Success criteria (lines 72-73): sandboxd, agent-supervisord, eventd Zig;
    "all 3 daemons start from systemd units generated by vakedz lower."

WP4-S5's "all 3 daemons" inherits that antecedent. The three subjects of this
spec are therefore the Zig daemons under daemons/:

Subject daemon Code home What "capability hook" means here
sandboxd daemons/sandboxd/ A real grant check on the fs/process lattice (the deny path, currently stubbed) at the exec/access boundary, emitting a common-shaped capability verdict.
agent-supervisord daemons/agent-supervisord/ The preceptord authority point: turn the stubbed-but-wired deny path into a real grant check on RFC 0005 control frames and worker spawn, with typed ControlRefusal, write-ahead logged.
eventd daemons/eventd/ The conformance-verification leg: a shared prover that replays the chain and proves every testified capability verdict conformed to the declared policy. eventd makes no per-request allow/deny decision (see §0.2).

0.1 agent_guardd and agent_memoryd are reference oracles, not subjects

Neither is part of WP4 (no code home under daemons/). Both already implement
the cap-hook + testimony pattern that the three subject daemons must conform to,
and serve as the Python reference / oracle in the #15 pattern (Python
fixes the decision + the bytes; the Zig daemon reproduces them — see
docs/runtime/agent-guardd.md lines 14-18):

  • agent_guardd: policy.decide (the pure deny-by-default verdict),
    evidence.egress_event (the on-chain testimony shape), verify.verify_run
    (the conformance prover). This is the canonical pattern the eventd prover
    generalizes.
  • agent_memoryd: capability.CapabilityToken + the mem lattice
    (none < recall < append < admin) + capability.check_capability (the
    POLA enforcement point raising PermissionError).

Illustrative gap (the bar the subjects must clear): agent_memoryd
decides at three sites (daemon.py lines 109/171/209 call
check_capability for APPEND/RECALL/ADMIN) but never testifies the capability
verdict to eventd — its only testimony event is memory_tombstone
(agent_memoryd/eventd.py line 127), and it has no conformance prover. By
contrast agent_guardd testifies every verdict and ships verify_run. WP4-S5
requires the three subject daemons to clear the guardd bar (decide → testify in
the common shape → be provable by the shared verifier), closing the gap memoryd
illustrates. memoryd itself is out of scope; it is shown here only as the
counter-example that motivates the common spine.

0.2 The discriminating constraint: who decides?

The hook looks different per daemon because not every daemon makes a per-request
decision:

  • sandboxd — yes. Per (path, mode) and per spawn: allow/deny at the
    exec/access boundary.
  • agent-supervisord — yes. Per RFC 0005 control frame and per worker spawn:
    allow/deny (the preceptord authority point).
  • eventd — no. eventd verifies after the fact. Its hook is the
    verification leg, not a decision leg. We deliberately do not invent a
    per-request decision point in eventd to make the trio symmetric.

The single concern that could be double-assigned is append authority (who
may write to the log). That is agent-supervisord's, via its single-writer
ownership (docs/superpowers/specs/2026-06-12-agent-supervisord-design.md lines
20-24: "owns the eventd writer — the RFC 0004 single-writer discipline: every
event flows through supervisord"; docs/superpowers/specs/2026-06-12-eventd-design.md
lines 55-57: "the OTP agent-supervisord owns the writer; fibers emit events to
it"). It is not eventd's hook. Assigning append-admission to eventd would
reimplement supervisord's concern inside eventd — the scope error the WP3-S3
spec warns against (docs/superpowers/specs/2026-06-14-wp3-s3.md §1.1).

1. Objective

Land the capability-enforcement hook in the three WP4 daemons so that, for
every privileged operation a daemon performs on behalf of a principal:

  1. the operation is gated by a grant check against the principal's declared
    capability (sandboxd: fs/process; supervisord: RFC 0005 control verbs +
    worker spawn);
  2. the verdict is testified to the eventd hash chain in a single
    cross-daemon common testimony shape (§4.1) — every allow and every
    deny, deterministically (no wall-clock in the chained payload), so a replay
    hashes identically;
  3. the run is provable: a shared eventd conformance prover (§4.4) replays
    the chain and proves (a) chain integrity and (b) that every testified verdict
    agrees with the declared policy — generalizing agent_guardd.verify.verify_run
    from one kind to the whole capability-verdict family.

The acceptance shape mirrors the guardd vertical slice
(docs/runtime/agent-guardd.md "The loop"): declare → lower → enforce →
testify → verify — closed for fs/process (sandboxd) and for control-plane
authority (supervisord), with eventd owning the verify leg for all of them.

Concern Owner Why not S5
network/ebpf membrane enforcement agent_guardd (oracle) / its eventual Zig port guardd is not a WP4 daemon; it is the reference pattern, already done.

The actual preceptord daemon future does not exist yet; S5 wires the authority point with an allow-all-default policy and a live deny path (§3.2).

2. Inputs / oracles

2.1 Normative inputs (RFCs)

  • RFC 0001 §"Authority and policy: preceptord" (protocol/rfcs/0001-hcp.md
    lines 330-360, 487-499): the authority model. preceptord decides whether a
    peer (SPIFFE ID) may access a named capability (network, filesystem, mcp, eBPF, process, memory) before dispatch to the handler; denied requests
    produce error frames with a typed ControlRefusal reason; authority is
    inherited through the OTP tree (agent ← supervisor ← session); revocation is
    effective on next frame evaluation. This is the contract supervisord's hook
    implements (with an allow-all default policy until preceptord ships).
  • RFC 0005 §1 frame catalog, §3 control_action, §4 ControlRefusal
    (protocol/rfcs/0005-control-frames.md): the control verbs
    (pause/resume/slow/step/rewind), the eventd payload kind recording an
    applied control frame (control_action), and the typed refusal reasons.
    "Authority is a preceptord policy decision, and every applied control
    action is itself an eventd entry    " (RFC 0005 Abstract) — the write-ahead
    testimony requirement for supervisord.
  • RFC 0004 §6 lifecycle + §7 topology epoch (protocol/rfcs/0004-multi-agent-state-dependency.md):
    the lifecycle states a worker may be in and the epoch a control action is
    evaluated under. The supervisor reads these; the hook fences a control verb on
    a stale epoch as a refusal (§3.2).

2.2 Oracle / reference implementations (Python, in-repo)

  • agent_guardd/policy.py — decide(membrane, host, port) returning
    (action, reason): the canonical pure, deterministic verdict function shape.
  • agent_guardd/evidence.py — egress_event(...): the canonical on-chain
    testimony payload (kind="ebpf_egress", principal, action, reason,
    mechanism, deterministic, no wall-clock). The §4.1 common shape generalizes
    this.
  • agent_guardd/verify.py — verify_run(policy, log_path) -> VerifyReport: the
    canonical conformance prover (boot-verify the chain, then for each verdict
    recompute decide and compare to the recorded action; strict
    principal→policy binding; a forged-but-chained event is caught as a
    conformance mismatch). The §4.4 eventd prover generalizes this across kinds.
  • agent_sandboxd/policy.py — decide_path(policy, path, mode) -> PolicyDecision
    and the FsGrant / ProcessGrant lattices. This is the sandboxd hook's
    decision oracle     — the Zig daemon reproduces its verdicts byte-for-byte on
    shared vectors.
  • agent_sandboxd/eventd.py — spawn_event / access_event (already carry
    action: "allow"|"deny", reason); these become the §4.1 common shape.
  • eventd/core.py — the frozen canonical-JSON + hash-chain functions
    (canonical_json, chain_hash, make_entry, verify_chain). The
    cross-language hash contract; the Zig eventd (WP4-S4) reproduces these bytes.
  • agent_memoryd/capability.py — CapabilityToken + check_capability: the
    illustrative POLA pattern (§0.1); used to cross-check the common token shape,
    not a subject.

3. Design

3.0 The common spine (all three daemons)

declare        capability fs/process (sandboxd) ; RFC 0005 verb + worker grant (supervisord)
   │             vaked → vakedc lower
   ▼ lower      gen/zig/<fiber>.json (sandboxd policy)  ;  gen/otp/<slug>_sup.erl + policy (supervisord)
   │
   ▼ DECIDE     sandboxd.decide_path / spawn-grant   ;   supervisord preceptord-point grant check
   │            (allow/deny, reason) — pure, deterministic
   ▼ TESTIFY    one CapabilityVerdict payload (§4.1) appended to the eventd chain
   │            — supervisord is the single writer; sandboxd emits THROUGH it (§3.3)
   ▼ VERIFY     eventd shared prover (§4.4): chain intact ∧ every verdict conforms to policy
                ⇒ "the capability held"

The three legs that are new in S5 are DECIDE (sandboxd + supervisord), the
common TESTIFY shape (§4.1, replacing each daemon's bespoke payload), and the
generalized VERIFY prover in eventd (§4.4). guardd already does all three for
network; S5 makes the other two daemons match and folds the verify into eventd.

3.1 sandboxd hook (decision: wire the deny path)

Today (Python ref, agent_sandboxd/daemon.py) the spawn path is allow-all +
log the principal with the deny path stubbed (sandboxd design lines 176-178).
S5 makes the grant check authoritative:

  1. Inputs. From the lowered config gen/zig/<fiber>.json (a 0012 lowering
    output): the principal id, the capability fs grant (none < repo_ro < repo_rw < host_ro < host_rw), the capability process grant
    (none < spawn_sandboxed < spawn < exec_host), and the concrete
    allowed_read / allowed_write path sets (sandboxd design §"Config
    contract"; mirrored by agent_sandboxd/policy.py FilesystemPolicy).
  2. fs decision. At every open/exec access boundary, call the verdict
    function (Zig port of decide_path): deny-by-default; a path is allowed iff
    it falls under an allowed_read prefix (read) or allowed_write prefix
    (write); grant=none denies everything; writes outside declared write bounds
    are denied even if readable. Output (action, reason).
  3. process decision. At spawn: the requested exec mode must be ≤ the
    process grant in the lattice. spawn_sandboxed requires the new namespace
    set (CLONE_NEWNS|NEWPID|NEWNET|NEWUSER, kickoff line 44); exec_host is
    refused unless the grant is exec_host. Output (action, reason).
  4. Enforce. A deny aborts the operation (no exec, no fd) — the membrane
    holds whether or not a cgroup/seccomp layer is also attached, exactly as the
    guardd reference datapath refuses a connect before a packet leaves
    (agent_guardd/enforce.py lines 49-59).
  5. Testify. Emit one CapabilityVerdict (§4.1) per decision through the
    supervisord writer (§3.3). sandboxd does no canonical hashing itself — the
    eventd oracle owns the bytes (sandboxd design lines 165-166, 190).

The reference oracle for legs 2-3 is agent_sandboxd/policy.py; the Zig daemon
must reproduce its verdicts byte-for-byte on shared vectors (§5.1).

3.2 agent-supervisord hook (decision: real preceptord point, deny path live)

Today the authority point is "allow-all + log the principal, with the deny path
stubbed but wired" (supervisord design lines 110-112). S5 turns the stub
into a live check at two sites:

  1. Control-frame authority (RFC 0005). For every inbound RFC 0005 request
    frame (pause/resume/slow/step/rewind), before applying it:
  2. resolve the target against the current topology epoch (RFC 0005 §2.4);
  3. epoch fence (RFC 0004 §7): a control frame carrying a non-current
    topology_epoch is refused stale_epoch;
  4. grant check (the preceptord point, RFC 0001 lines 487-499): does the
    peer (SPIFFE ID) hold the capability the verb requires under the policy in
    force? Default policy is allow-all (no preceptord daemon yet), but the
    deny path is live: a policy that denies does refuse, with a typed
    ControlRefusal reason (RFC 0005 §4). This is the one behavioral change
    from today — denials are no longer impossible.
  5. On allow: apply the verb and write a write-ahead control_action
    eventd entry (RFC 0005 §3; RFC 0005 Abstract "every applied control action
    is itself an eventd entry"). On deny: emit a ControlAck refusal and a
    CapabilityVerdict (§4.1) with action="deny".
  6. Worker-spawn authority. When the OTP tree starts a worker whose fiber
    declares a capability grant, the supervisor verifies the grant is
    authorized under policy before starting the child, emitting a
    CapabilityVerdict. (sandboxd then performs the fs/process enforcement at
    exec; the supervisor's check is the coarse-grained "may this principal hold
    this grant at all".)

Reference-phase intake (supervisord design lines 65-70): RFC 0005 frames
over Litany Wire are not available until WP3 lands the wire. Until then control
verbs arrive via the ralph control.json polled per tick; file-driven actions
synthesize their corr and refusals are log-visible only (no ControlAck
channel). The grant check and the CapabilityVerdict/control_action testimony
run identically on both intakes — the file is just a transport.

3.3 eventd hook (verification: the shared conformance prover) + the writer seam

eventd makes no per-request decision (§0.2). Its two S5 responsibilities:

  1. Be the single-writer funnel's sink (writer = supervisord). Append
    authority is supervisord's single-writer discipline (§0.2). eventd provides
    the append primitive (flock single-writer, fsync-on-append, boot-time
    chain verify — eventd/core.py + the EventLog daemon). sandboxd's
    CapabilityVerdict events flow through the supervisord writer
    (eventd design lines 55-57). S5 does not change the byte format or add an
    admission decision to eventd — it reuses the frozen contract verbatim.
  2. The shared conformance prover — the real new eventd code. Generalize
    agent_guardd.verify.verify_run from one kind (ebpf_egress) to the whole
    capability-verdict family:
   verify_capabilities(policies, log_path) -> CapabilityReport
     1. EventLog(log_path)  # boot-verify the hash chain; TamperError ⇒ chain_ok=False
     2. for each entry whose payload.kind ∈ {ebpf_egress, sandbox_fs_access,
        sandbox_spawn, control_action, capability_verdict}:
          principal = payload.principal
          policy    = policies.for(payload.daemon, principal)   # strict binding
          if policy is None: mismatch("no-policy-for-principal")  # forged principal caught
          expected  = recompute_decision(payload.daemon, policy, payload)  # re-run the daemon's
                                                                           # pure verdict fn
          if expected.action != payload.action: mismatch(recorded, expected, subject)
     3. held = chain_ok ∧ no mismatches

recompute_decision dispatches on payload.daemon:
- sandboxd → re-run decide_path (fs) or the process-lattice check over the
declared policy and payload.subject/payload.mode;
- agent-supervisord → re-run the verb-grant lattice check over the policy +
payload.required_level;
- agent_guardd → re-run policy.decide (the existing path, now one branch).

The prover proves both guarantees from the ledger alone (mirroring
agent_guardd/verify.py lines 1-13): integrity (a flipped byte breaks a
link and is refused) and conformance (a forged-but-chained verdict whose
action disagrees with the recomputed decision is caught — so a daemon that
says allow but policy says deny is "enforcement drift", not a capability
that held). Strict principal→policy binding (no silent fallback to an
unrelated policy) is carried over verbatim from verify.py lines 56-67.

During the reference phase the prover is the Python eventd oracle (a new
eventd.capverify module + python3 -m eventd capverify CLI, frozen exit
code alongside the existing 0/2/3/4/5); the Zig eventd port (WP4-S4)
reproduces it under the same parity gate.

4. Contracts

4.1 The common CapabilityVerdict testimony payload

A cross-daemon convention (eventd payloads are arbitrary JSON —
eventd/core.py lines 13-15 — so this is a shape the prover consumes, not
something the eventd format imposes). Every subject daemon emits exactly this
shape at each decision; deterministic, no wall-clock (so replay hashes
identically — agent_guardd/evidence.py lines 1-9):

{
  "kind": "capability_verdict",
  "v": 1,
  "daemon": "sandboxd" | "agent-supervisord" | "agent_guardd",
  "principal": "<agent/peer id>",
  "capability": "fs" | "process" | "network" | "control" | "spawn",
  "subject": "<path | host:port | target | verb>",
  "mode": "read" | "write" | null,
  "required_level": "<lattice value the op needs>",
  "granted_level": "<lattice value the principal holds>",
  "action": "allow" | "deny",
  "reason": "<human-readable decision basis>",
  "mechanism": "reference" | "ebpf-cgroup" | "seccomp" | "ns",
  "attempt": <int, optional ordinal>
}

Migration note: the existing guardd ebpf_egress and sandboxd
sandbox_fs_access/sandbox_spawn payloads already carry action/reason/
principal; S5 either (a) maps them into this shape at the verifier, or (b)
adds capability_verdict as the going-forward kind while the prover keeps
recognizing the legacy kinds (the §4.4 kind-set lists both). Decision deferred to
the impl PR; the verifier must accept both so no historical chain is
orphaned.

4.2 The lattices (frozen, from existing specs)

  • capability fs (sandboxd; agent_sandboxd/policy.py,
    docs/language/0011-type-system.md §5): none < repo_ro < repo_rw < host_rw
    and none < repo_ro < host_ro.
  • capability process (sandboxd): none < spawn_sandboxed < spawn < exec_host.
  • control-verb grants (supervisord): each RFC 0005 verb maps to a required
    capability under the RFC 0001 set (network … memory); the lattice is the
    preceptord policy's, allow-all by default.

4.3 Policy input contract

  • sandboxd: gen/zig/<fiber>.json (0012 lowering output) — principal, fs grant,
    process grant, allowed_read/allowed_write, cgroup limits.
  • supervisord: gen/otp/<slug>_sup.erl (lowered tree) + a policy doc (Nix
    module / runtime ACL — RFC 0001 line 483); allow-all default.
  • eventd prover: the union of the above, keyed by (daemon, principal).

4.4 Verifier kind-set

{ ebpf_egress (legacy), sandbox_fs_access (legacy), sandbox_spawn (legacy), control_action, capability_verdict }. Any other kind is passed over (not a
capability verdict); an unknown principal for a recognized kind is a mismatch.

5. Test plan

Two tiers. Tier L is everything verifiable locally on the M1 (the oracle +
decision + testimony + verify legs are pure / userspace — no namespace
syscalls). Tier K is what needs dev-cx53 / Linux (real namespaces, cgroup,
seccomp, BPF attach). dev-cx53 is off-limits for the 6h autoresearch window, so
all S5 spec-suite verification is Tier L; Tier K is the integration-test layer
folded into WP4-S6.

5.1 Tier L — local on M1 (the conformance backbone)

Run from repo root: python3 tests/spec/run_all.py (register a new
test_capability_hooks.py exposing run() -> (ok, lines) in
tests/spec/run_all.py, matching the existing module convention).

  1. Decision parity (oracle vs Zig). Shared golden vectors of
    (policy, op) → (action, reason) for sandboxd fs + process. The Python ref
    (agent_sandboxd/policy.decide_path + the process-lattice check) and the Zig
    daemon's decide must agree on every vector. Verify on M1 with
    cargo test-style host execution of the Python oracle and
    zig build -Dtarget=x86_64-linux test for the Zig decision unit (the decision
    function is pure — no syscalls — so it compiles and runs the unit tests via
    Zig's host test runner even cross-targeted; only the namespace code is
    compile-only on M1).
  2. Testimony shape conformance. Each emitted event validates against the
    §4.1 schema (deterministic: emit the same op twice → byte-identical
    canonical_json(payload) via eventd/core.canonical_json).
  3. "Capability held" end-to-end (the slice). For sandboxd: declare a
    capability fs repo_ro worker, lower it, run a scripted sequence of accesses
    (an in-bounds read, an out-of-bounds read, a write to a read-only path),
    testify each through a temp eventd log, then verify_capabilities(policy, log) ⇒ held=True and the deny verdicts are present. Mirrors
    tests/spec/test_agent_guardd.py's loop.
  4. Forged-verdict conformance catch. Append a chained-but-forged
    capability_verdict (e.g. action="allow" for a path the policy denies, or
    principal="intruder" with no policy) ⇒ the prover reports a conformance
    mismatch (not "held"). This is the regression test that proves the prover
    catches enforcement drift (write it before the prover — it must fail when
    the prover is a no-op).
  5. Byte-flip tamper catch. Flip one byte in the log ⇒ boot-verify
    TamperError ⇒ chain_ok=False. Reuses the eventd guarantee.
  6. supervisord deny-path unit (no BEAM in CI — supervisord design line 91).
    The grant-check + epoch-fence + typed-refusal logic is exercised as a pure
    function over a policy fixture + a synthesized RFC 0005 frame
    (control-file intake): allow-all policy → applied + control_action; a
    deny policy → ControlRefusal; a stale epoch → stale_epoch. Asserted via
    the testified events (python3 -m eventd state / the prover), exactly the
    task otp-smoke style (supervisord design lines 83-86).
  7. Zig compile gate. zig build -Dtarget=x86_64-linux compiles the sandboxd
    and eventd hook code (namespace/cgroup/BPF call sites compile; they are not
    run on M1).

5.2 Tier K — needs dev-cx53 / Linux (deferred to WP4-S6 integration)

  • Real unshare(CLONE_NEW*) + execvpe in the new ns; a denied process grant
    actually refusing exec (exit 1) and testifying (kickoff line 70: "policy
    violations → eventd + exit 1").
  • cgroup-v2 bound enforced (over-memory.max child OOM-killed; clone past
    pids.max refused) — sandboxd design lines 206-207.
  • seccomp-bpf allowlist refusal.
  • BPF cgroup attach alongside guardd (the attach probe that EINVALs in nested
    containers — docs/runtime/agent-guardd.md line 51).
  • All 3 daemons started from vakedz lower-generated systemd units on vakedos
    (kickoff success criteria line 73; this is WP4-S6/S7).

Why Tier K cannot run on M1: Linux namespaces, cgroup-v2, seccomp-bpf, and
bpf() are Linux-kernel facilities with no Darwin equivalent. The decision and
testimony/verify legs (the security-critical logic) are not kernel-bound and
are fully covered in Tier L — the same posture guardd documents
(docs/runtime/agent-guardd.md lines 52, 58: "the security-critical decision …
and the audit spine are real on every host"; only attach can be withheld).

8. Acceptance criteria

  1. sandboxd: an out-of-bounds (path, mode) and an over-grant process mode
    are denied (the deny path is live, not stubbed), the operation is aborted,
    and a CapabilityVerdict (action="deny") is testified. The Zig
    decide/capability.zig agrees with agent_sandboxd/policy.decide_path on
    every golden vector (§5.1.1). [Tier L; enforcement-abort proven Tier K/S6.]
  2. agent-supervisord: an RFC 0005 control verb under a deny policy yields a
    typed ControlRefusal; under allow-all it applies and writes a write-ahead
    control_action; a stale topology_epoch is refused stale_epoch. Each path
    is log-visible (§5.1.6). [Tier L.]
  3. eventd: verify_capabilities reports held=True for a conformant run and
    a conformance mismatch for (a) a forged-but-chained verdict and (b) an
    unknown-principal verdict; a single flipped byte yields chain_ok=False. The
    prover recognizes the full §4.4 kind-set including the legacy guardd/sandboxd
    payloads (no historical chain orphaned). [Tier L.]
  4. Common shape: every verdict from all three daemons (and guardd) validates
    against the §4.1 schema and hashes deterministically (same op twice →
    identical canonical_json). [Tier L.]
  5. Regression test (§5.1.4) discipline: the forged-verdict test fails when
    the prover is a no-op and passes once the prover lands (proves it catches
    enforcement drift, per CLAUDE.md §5).
  6. Local CI green on M1: python3 tests/spec/run_all.py passes with
    test_capability_hooks registered; zig build -Dtarget=x86_64-linux
    compiles the sandboxd + eventd hook code. [Tier L.]
  7. Tier K (S6 hand-off, not blocking S5 sign-off): on dev-cx53, a denied
    process grant refuses execvpe (exit 1) + testifies; a cgroup bound is
    kernel-enforced; the 3 daemons start from vakedz lower units.
